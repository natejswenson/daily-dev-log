## Shipped

devlog used to summarize each day's commits. v0.3.0 makes it summarize each release: when I tag a version, it writes one entry for that release and pushes it, and re-running does nothing until the next tag. Entries now come from a configurable voice profile instead of a generic standup tone, so the log sounds like me. In a monorepo each project sets its own tag prefix so several skills track releases independently, only real version tags become entries, and tag names are validated as untrusted input before they ever touch a shell.

This post is itself one of those entries, which makes it a decent test of whether the format holds up. The feature is "one entry per release," but the part worth writing about is the three decisions underneath it: anchor the work on release tags so re-runs are idempotent, treat the voice as external configuration instead of code, and treat the tag names git hands you as untrusted input. Here is the end-to-end build.

## Setup: per-project config and a voice path

Everything project-specific lives in config so the generator itself stays generic. Every project lives in one repo (a monorepo); each declares its `tagPrefix` and a `pathFilter`, so the generator can track several release streams independently and scope each one's commits to its own directory. The run declares a `voicePath`, an ordered list of places to look for the voice profile. The first file that exists wins, which is what lets the same skill run for anyone: the behavior is shared, the personality is supplied.

```js
// devlog.config.js
module.exports = {
  // Every project lives in this one repo (a monorepo), so every git command runs
  // against the same working tree; a per-project pathFilter scopes each one's
  // commits, reusing the pathFilter seam added in v0.2.0.
  repo: "claude-skills",
  projects: {
    devlog: { tagPrefix: "v",        entriesDir: "content/devlog", pathFilter: "skills/devlog" },
    resume: { tagPrefix: "resume-v", entriesDir: "content/resume", pathFilter: "skills/resume" },
  },
  // Voice profile search path, checked in order. First hit wins; if no listed
  // file exists, resolveVoice falls back to the bundled default, so the tool
  // runs even with no user config.
  voicePath: [
    "./.devlog/voice.json",
    "~/.config/ghostwriter/voice.json",
  ],
};
```

The `tagPrefix` is the seam that makes the rest of the tool release-anchored rather than day-anchored. Instead of asking "what did I touch today," every step from here keys off the set of version tags that match that prefix.

## Enumerate releases and validate every tag as untrusted input

A tag name is just text someone wrote, and this tool puts that text next to shell commands and filesystem paths. "It came from my repo" is not the same as "it's safe to interpolate into a command," and the gap between those two is where command injection lives. Git ref names are more permissive than you would guess; a name like `v1.0.0$(id)` is a perfectly legal tag that would be catastrophic if it reached a shell unquoted.

So before a tag is used for anything, it is checked against what a tag is *allowed* to be, and everything else is rejected. That is the allowlist principle: define what's valid and reject the rest, which OWASP recommends precisely because the alternative, blocking known-bad patterns, is easy to evade ([OWASP: Input Validation](https://cheatsheetseries.owasp.org/cheatsheets/Input_Validation_Cheat_Sheet.html); [OWASP: Injection Prevention](https://cheatsheetseries.owasp.org/cheatsheets/Injection_Prevention_Cheat_Sheet.html)).

```bash
#!/usr/bin/env bash
set -euo pipefail

# One argument: the project key. The tag prefix and output dir come from
# devlog.config.js, so a monorepo tracks each project's releases on its own.
PROJECT="${1:?usage: gen-entries <projectKey>}"
# Capture the config in a command substitution (not a process substitution): the
# explicit `|| exit 1` stops the run on a non-zero node exit, so an unknown
# project stops the run instead of silently leaving PREFIX/ENTRIES_DIR empty.
CONFIG="$(node -e '
  const { projects } = require("./devlog.config");
  const p = projects[process.argv[1]];
  if (!p) { console.error("unknown project: " + process.argv[1]); process.exit(1); }
  process.stdout.write(`${p.tagPrefix}\t${p.entriesDir}`);
' "$PROJECT")" || exit 1
IFS=$'\t' read -r PREFIX ENTRIES_DIR <<<"$CONFIG"
[[ -n "$PREFIX" && -n "$ENTRIES_DIR" ]] || { echo "bad config for $PROJECT" >&2; exit 1; }

# A tag is attacker-controllable input. First reject anything not shell/path-safe
# (the security boundary); then keep only tags shaped like a release version, so
# "vanity", "version-2", and "v1..0" never become entries.
is_safe_tag()    { [[ "$1" =~ ^[A-Za-z0-9][A-Za-z0-9._-]*$ ]]; }
is_release_tag() { [[ "${1#"$PREFIX"}" =~ ^[0-9]+\.[0-9]+\.[0-9]+(-[A-Za-z0-9.]+)?$ ]]; }

# Ensure the output dir exists so the atomic write's `> "$out.tmp"` redirection
# can't fail on a fresh checkout where content/devlog isn't there yet.
mkdir -p "$ENTRIES_DIR"

git tag --list "${PREFIX}*" | while IFS= read -r tag; do
  if ! is_safe_tag "$tag"; then
    printf 'skip: invalid tag name %q\n' "$tag" >&2
    continue
  fi
  if ! is_release_tag "$tag"; then
    printf 'skip: %s is not a release version\n' "$tag" >&2
    continue
  fi
  # tag is now shell-safe and a real release; safe to use below.
done
```

The shell-safety check runs on the way *in*, before the tag is interpolated anywhere: a name with a `$`, a backtick, a slash sequence, or a leading dash never makes it past the `continue`. The release-shape check is a separate filter; it answers "is this actually a version I should write up," and it is the reason a stray branch-like or marketing tag is ignored instead of generating a bogus entry.

## Idempotency: the tag is the key

Anchoring an entry to a version tag makes the tool safe to re-run. A tag either has an entry file or it doesn't, so running the generator twice produces the same result as running it once. That is idempotency, the property that applying an operation repeatedly has the same effect as applying it once (MDN frames this for HTTP methods, but the property is general: [MDN: Idempotent](https://developer.mozilla.org/en-US/docs/Glossary/Idempotent)). It matters because it removes a class of bookkeeping: the entry filename derived from the tag *is* the idempotency key, so the tool never has to track which tags it already processed.

Save the validation script above as `gen-entries` and make it executable (`chmod +x gen-entries`); the block below replaces that final placeholder comment inside the loop.

```bash
# ...inside the validated loop from the previous block:
  out="${ENTRIES_DIR}/${tag}.md"

  # Idempotent: a tag that already has an entry is a no-op. The tag is the key.
  if [[ -f "$out" ]]; then
    printf 'skip: %s already has an entry\n' "$tag"
    continue
  fi

  # Write atomically: generate to a temp file and publish only on success, so a
  # failed run never leaves a half-written entry that the check above would then
  # treat as "done" forever.
  if ! node generate.js "$tag" "$PROJECT" > "$out.tmp"; then
    rm -f "$out.tmp"
    printf 'fail: %s could not be generated\n' "$tag" >&2
    continue
  fi
  mv "$out.tmp" "$out"

  # Commit and push the new entry. The file write above is the idempotent part;
  # the push is best-effort. If it fails after the commit lands, the entry is
  # committed locally and the next run (seeing the file) won't retry the push, so
  # re-push by hand.
  git add "$out"
  git commit -q -m "devlog: ${tag}"
  git push -q || printf 'warn: %s committed locally but push failed; re-push manually\n' "$tag" >&2
  printf 'wrote: %s\n' "$out"
done
```

Because the existence of `"$out"` is the only thing that decides whether work happens, there is no separate "have I processed this tag" record to keep in sync with the filesystem.

## Resolve the voice profile and write the entry

The voice that makes the log sound like me does not live in the tool. It is a profile resolved from the configured `voicePath`, falling back through the search list to a bundled default. The resolver does one job: walk the candidates, return the first that exists, otherwise the default. Keeping voice as external config rather than hard-coding it is the same instinct as the rest of the config; the tool stays generic and the user-specific parts are values it reads, not assumptions it bakes in.

The profile is shared with the LinkedIn ghostwriter, so it carries both writing rules and feed-tuning knobs. A minimal `voice.default.json`:

```json
{
  "tone": ["plain", "concrete", "first-person"],
  "neverDo": ["em dashes in prose", "rhetorical questions", "hype words"],
  "reach": { "hashtags": 3, "hook": "question" },
  "engagement": { "cta": "ask for replies" }
}
```

```js
// voice.js
const fs = require("node:fs");
const os = require("node:os");
const path = require("node:path");
const BUNDLED_DEFAULT = require("./voice.default.json");

const expand = (p) =>
  p.startsWith("~") ? path.join(os.homedir(), p.slice(1)) : p;

// Fallback chain: first existing file in voicePath wins, else bundled default.
function resolveVoice(voicePath) {
  for (const candidate of voicePath) {
    const file = expand(candidate);
    if (fs.existsSync(file)) return JSON.parse(fs.readFileSync(file, "utf8"));
  }
  return BUNDLED_DEFAULT;
}

// A dev log is not a feed: keep tone and never-do rules, drop the reach tuning.
function devlogVoice(profile) {
  const { reach, engagement, ...rest } = profile;
  return rest;
}

module.exports = { resolveVoice, devlogVoice };
```

The generator takes the same project key the bash loop was invoked with, scopes the commit range to that project's path, wires the resolved voice to it, and renders the entry. The voice is passed in as data, so swapping profiles never touches this code.

```js
// generate.js
const { resolveVoice, devlogVoice } = require("./voice");
const { commitsForRelease, renderPost } = require("./render");
const { voicePath, projects, repo } = require("./devlog.config");

async function generateEntry(tag, projectKey = "devlog") {
  const project = projects[projectKey];
  if (!project) throw new Error(`unknown project: ${projectKey}`);
  const voice = devlogVoice(resolveVoice(voicePath));   // external config, not code
  const commits = await commitsForRelease(tag, project); // prevTag..tag, scoped to project.pathFilter
  return renderPost({ tag, repo, commits, voice });      // -> markdown
}

// CLI: `node generate.js <tag> <projectKey>` prints the entry to stdout (the bash
// loop redirects it). A thrown error exits non-zero, so a failed generation is a
// non-zero exit rather than empty stdout, and the caller never publishes it.
if (require.main === module) {
  generateEntry(process.argv[2], process.argv[3])
    .then((md) => process.stdout.write(md))
    .catch((err) => { console.error(err); process.exit(1); });
}

module.exports = { generateEntry };
```

The commit range itself comes from `render.js`, which scopes each entry to exactly one release and one project: the commits in `prevTag..tag` that touched the project's `pathFilter`, not everything since the tag. `prevTag` is the previous tag with the same prefix (so in the monorepo another project's release is never mistaken for this one's predecessor), and it is empty for the very first release, which makes the range the full history up to the tag.

```js
// render.js (excerpt)
const { execFile } = require("node:child_process");
const { promisify } = require("node:util");
const run = promisify(execFile);

// Commits belonging to THIS release: the range prevTag..tag, scoped to this
// project's path. Find the previous tag with the SAME prefix (so another
// project's tag in the monorepo isn't mistaken for this one's predecessor); if
// there isn't one (first release), fall back to the full history. Every git
// command runs in the one repo's working tree.
async function commitsForRelease(tag, { tagPrefix, pathFilter }) {
  const { stdout: prev } = await run("git", [
    "describe", "--abbrev=0", "--tags", "--match", `${tagPrefix}*`, `${tag}^`,
  ]).catch(() => ({ stdout: "" }));
  const range = prev.trim() ? `${prev.trim()}..${tag}` : tag;
  const { stdout } = await run("git", ["log", "--format=%s", range, "--", pathFilter]);
  return stdout.split("\n").filter(Boolean);
}

// renderPost (omitted) turns { tag, repo, commits, voice } into the markdown body.
module.exports = { commitsForRelease, renderPost };
```

`renderPost` is deliberately out of scope here: it is the one piece you supply. Its contract is small, it takes `{ tag, repo, commits, voice }` and returns the entry's markdown, so how you turn commits and a voice profile into prose is yours to decide.

`devlogVoice` strips the LinkedIn reach and engagement tuning before the profile reaches the renderer, because a dev log is not a feed; it keeps the tone and the never-do rules and ignores the parts that only make sense for a social timeline.

## Verify: re-runs are no-ops, bad tags are skipped

The two decisions worth defending are the ones you can demonstrate from the shell: running twice changes nothing the second time, and a malformed tag never reaches generation or the filesystem.

```bash
# 1. First run writes, commits, and pushes an entry for the new release tag.
$ ./gen-entries devlog
wrote: content/devlog/v0.3.0.md

# 2. Re-running is a no-op: the tag already has an entry (idempotent).
$ ./gen-entries devlog
skip: v0.3.0 already has an entry

# 3. A legal git tag that fails the allowlist is skipped before it can run.
$ git tag 'v9.9.9$(id)'                  # a valid git ref, but hostile as a shell string
$ ./gen-entries devlog
skip: invalid tag name v9.9.9\$\(id\)    # printf %q escaping; rejected on the way in
```

The second run proves the tag is the idempotency key: no duplicate file, no second push, no edited entry. The third proves the allowlist holds at the boundary. Every use site is already quoted (`"$tag"`, `"${ENTRIES_DIR}/${tag}.md"`, `-m "devlog: ${tag}"`) or passed through `execFile` with no shell, and parameter expansion inside double quotes doesn't re-trigger command substitution, so `$(id)` is inert even without the check. `is_safe_tag` is defense-in-depth: even though every use site is quoted, it rejects hostile names on the way in. If either property regresses, this exact transcript stops matching, which makes it a cheap thing to keep as a smoke test.
