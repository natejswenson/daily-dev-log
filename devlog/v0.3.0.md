## Shipped

devlog used to summarize whatever I'd committed that day. v0.3.0 rebuilds it around release tags instead: tag a version and it writes one entry for that release, scoped to the commits since the previous tag, in a voice profile instead of a generic standup tone. Run it again and nothing happens until the next tag. In a monorepo, each project sets its own tag prefix so several packages track releases independently.

None of that works unless three things are true: a tag name can't be trusted just because it came from your own repo, the "previous release" for a diff range has to be computed correctly even when prereleases sit between two final versions, and re-running the generator has to be a safe no-op. Here's how to build a small version of that pipeline yourself.

## Setup: how a tag sorts

You'll need Node and a git repo. Before writing any code, build a throwaway repo and tag it the way a real project would, mixing a prerelease and a build-metadata tag in with two final releases:

```bash
mkdir demo && cd demo && git init -q
git commit --allow-empty -q -m "init"
git tag v0.9.0
git commit --allow-empty -q -m "add feature A"
git tag v1.0.0-rc.1
git commit --allow-empty -q -m "fix regression found in rc"
git tag v1.0.0
git tag v1.0.0+build
git commit --allow-empty -q -m "post-release chore"
git tag v2.0.0-alpha
git tag --list 'v*' --sort=-v:refname
```

```text
v2.0.0-alpha
v1.0.0-rc.1
v1.0.0+build
v1.0.0
v0.9.0
```

Look at where `v1.0.0-rc.1` landed: above `v1.0.0`. Semantic Versioning says the opposite should be true, "a pre-release version has lower precedence than the associated normal version" ([semver.org](https://semver.org/)). Git's `--sort=version:refname` sorts tags as version-like strings, not as semver-ranked releases ([git-scm: git-tag](https://git-scm.com/docs/git-tag)), so a naive "take the next line down" approach to finding the previous release will occasionally hand you a prerelease or grab the wrong tag entirely. That's the bug the rest of this post designs around.

## Build it: allowlist the tag before it becomes a filename

A git tag is text someone with push access wrote, not something you wrote yourself. The version label you pull out of it is about to become a filename and, later, part of a URL, so it needs the same treatment as any other outside input: define what a release label is allowed to look like, and reject everything else. OWASP's guidance on this is blunt, allowlisting "involves defining exactly what IS authorized, and by definition, everything else is not authorized," and a denylist of "known bad" characters is always missing one ([OWASP: Input Validation Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Input_Validation_Cheat_Sheet.html)).

```js
// release.js
const { execFileSync } = require("node:child_process");

// A final release label: "v", then digits and dots, nothing else. No "-rc",
// no "+build", no anything a shell or a filesystem could get creative with.
const RELEASE = /^v\d+(\.\d+)*$/;

function releaseTags(repoPath, prefix) {
  // execFileSync passes argv straight to the git process; no shell parses
  // this string, so a tag containing `;`, a backtick, or `$(...)` can't break
  // out into a second command no matter what characters it holds.
  const out = execFileSync(
    "git",
    ["-C", repoPath, "tag", "--list", `${prefix}*`, "--sort=-v:refname"],
    { encoding: "utf8" },
  );
  return out.split("\n").filter(Boolean);
}

function versionLabel(tag) {
  const i = tag.search(/v\d/);
  return i === -1 ? null : tag.slice(i);
}

module.exports = { RELEASE, releaseTags, versionLabel };
```

`execFileSync` already closes off the classic injection path, there's no shell to break out of. The regex is a separate concern: it's the difference between "safe to pass to git" and "safe to use as a filename and a URL segment," and a tag can clear the first bar while failing the second.

## Build it: pick the previous release from the filtered list, not the raw sort

This is where the setup demo pays off. `prevRelease` has to walk the list of tags that already passed the `RELEASE` check, never the raw output of `git tag --list`, because the raw output can rank a prerelease above the final release it precedes, exactly like `v1.0.0-rc.1` did above `v1.0.0` a minute ago.

```js
// release.js (continued)
function filterReleases(tags) {
  return tags
    .map((tag) => ({ tag, label: versionLabel(tag) }))
    .filter(({ label }) => label && RELEASE.test(label));
}

function prevRelease(tags, currentTag) {
  const releases = filterReleases(tags);
  const i = releases.findIndex((r) => r.tag === currentTag);
  if (i === -1) throw new Error(`${currentTag} is not a final release`);
  return releases[i + 1]?.tag ?? null; // null: currentTag is the earliest release
}

module.exports = { RELEASE, releaseTags, versionLabel, filterReleases, prevRelease };
```

Filtering first means `v1.0.0-rc.1` and `v1.0.0+build` are simply invisible when computing `v1.0.0`'s range. The previous release is whatever final tag comes next in the filtered list, in this case `v0.9.0`, not whatever sits next in the raw sort.

## Make it idempotent: the version label is the only state you need

The entry's filename is derived directly from the tag, so a tag either has a file or it doesn't. There's no separate "have I already processed this" record to keep in sync with anything, which is the same idea behind an API idempotency key: you generate the same result for the same key every time and never repeat the underlying work ([Stripe: Idempotent Requests](https://docs.stripe.com/api/idempotent_requests)). Here, the version label is the key.

One more thing has to happen before any of this runs: fetch. A release is commonly cut by CI on the remote, so a local clone that hasn't fetched simply doesn't have the tag yet, and would report "nothing new" for a release that's already live.

```js
// generate-release-note.js
const fs = require("node:fs");
const path = require("node:path");
const { execFileSync } = require("node:child_process");
const { releaseTags, prevRelease } = require("./release");

function commitsInRange(repoPath, range, pathFilter) {
  const args = ["-C", repoPath, "log", range, "--format=%s"];
  if (pathFilter) args.push("--", pathFilter);
  const out = execFileSync("git", args, { encoding: "utf8" });
  return out.split("\n").filter(Boolean);
}

function renderEntry({ tag, prev, commits }) {
  // Your own template goes here; this stub is enough to prove the pipeline.
  return `# ${tag}\n\nSince ${prev ?? "the beginning"}:\n\n` +
    commits.map((c) => `- ${c}`).join("\n") + "\n";
}

function generate(repoPath, { prefix, entriesDir, pathFilter, tag }) {
  execFileSync("git", ["-C", repoPath, "fetch", "--tags", "--quiet"]);

  const tags = releaseTags(repoPath, prefix);
  const prev = prevRelease(tags, tag);
  const range = prev ? `${prev}..${tag}` : tag;
  const commits = commitsInRange(repoPath, range, pathFilter);

  const out = path.join(entriesDir, `${tag}.md`);
  if (fs.existsSync(out)) return { skipped: tag };

  const body = renderEntry({ tag, prev, commits });

  // Check again, right before writing. The first check is convenient; this
  // one is load-bearing (see Gotchas).
  if (fs.existsSync(out)) return { skipped: tag };
  fs.mkdirSync(entriesDir, { recursive: true });
  fs.writeFileSync(out, body);
  return { wrote: out };
}

module.exports = { generate };
```

## Use it, then verify it

Run it once against the demo repo from Setup:

```bash
node -e '
const { generate } = require("./generate-release-note");
console.log(generate(".", { prefix: "v", entriesDir: "./entries", pathFilter: null, tag: "v1.0.0" }));
'
```

```text
{ wrote: 'entries/v1.0.0.md' }
```

```bash
cat entries/v1.0.0.md
```

```text
# v1.0.0

Since v0.9.0:

- fix regression found in rc
- add feature A
```

The range correctly starts at `v0.9.0`, not at `v1.0.0-rc.1` or `v1.0.0+build`. Run it again and it does nothing:

```bash
node -e '
const { generate } = require("./generate-release-note");
console.log(generate(".", { prefix: "v", entriesDir: "./entries", pathFilter: null, tag: "v1.0.0" }));
'
```

```text
{ skipped: 'v1.0.0' }
```

Now try a tag built to break the pipeline:

```bash
node -e '
const { versionLabel, RELEASE, filterReleases } = require("./release");
const hostile = "v1.0.0$(touch pwned)";
console.log(RELEASE.test(versionLabel(hostile)));
console.log(filterReleases(["v1.0.0", hostile, "v0.9.0"]).map((r) => r.tag));
'
ls | grep -i pwned || echo "no pwned file"
```

```text
false
[ 'v1.0.0', 'v0.9.0' ]
no pwned file
```

The hostile tag's label fails `RELEASE`, so `filterReleases` drops it before it reaches anywhere near a shell, and no command it contains ever runs.

## Gotchas

**Git's version sort is not semver precedence.** `--sort=-v:refname` will rank `v1.0.0-rc.1` above `v1.0.0`, the reverse of what SemVer defines. If your range logic trusts the raw sort order to find "the previous release," it will occasionally compute a backwards or wrong range the first time a prerelease tag shows up. Filter to final-release tags first, then walk that list.

**A local clone doesn't see a tag until you fetch it.** In this project's own history, the very next patch release after this one (0.3.1) fixed exactly this: `/devlog` was listing only local tags, so a release cut by CI on the remote was invisible until the local clone fetched, and the run reported "nothing new" for a release that had already shipped. Fetch tags before you list them, and treat the fetch as best-effort, if it fails, fall back to whatever local tags exist rather than aborting the run.

**An existence check that swallows its own errors is a false negative waiting to happen.** If your "does this entry already exist" probe hides a network or filesystem error behind a generic `catch`, a transient failure reads as "doesn't exist yet" and the run proceeds to write a duplicate. Check again immediately before the write, using a path that can't silently eat an error the same way, so one bad check is never the only thing standing between a re-run and a duplicate entry.
