## Shipped

v0.4.2 changes one path. devlog can borrow its writing-voice profile from ghostwriter, a sibling tool that already maintains one. The fallback pointed at `~/.claude/skills/ghostwriter/voice`, which was never ghostwriter's real install location, so that branch of the lookup could never succeed. When ghostwriter moved its user data to a documented shared home at `~/.claude/ghostwriter/`, devlog pinned its fallback to the same place. The patch is one constant; the failure mode is worth understanding if any two of your tools share files by convention.

## Why a dead fallback stays invisible

devlog resolves its voice directory through an ordered chain: an explicit `voicePath` from config, then ghostwriter's directory if it exists, then a bundled default. A resolver like that is deliberately tolerant, in the spirit of Fowler's [Tolerant Reader](https://martinfowler.com/bliki/TolerantReader.html): "be liberal in what you accept," take what you need, and keep working when a source is missing. Here's the shape, complete enough to reuse:

```js
// voice.mjs
import { existsSync } from 'node:fs';
import { homedir } from 'node:os';
import { join } from 'node:path';

// One exported constant; every consumer imports it instead of retyping the path.
export const GHOSTWRITER_VOICE_DIR = join(homedir(), '.claude', 'ghostwriter', 'voice');

export const DEFAULT_CANDIDATES = (configPath) => [
  configPath,                                    // explicit config wins
  GHOSTWRITER_VOICE_DIR,                         // sibling tool, if installed
  join(homedir(), '.claude', 'devlog', 'voice'), // bundled default
];

// Candidates are injectable so a test can hand the resolver a fake layout.
export function resolveVoiceDir(configPath, candidates = DEFAULT_CANDIDATES(configPath)) {
  return candidates.find((p) => p && existsSync(p)) ?? null;
}
```

The tolerance is also exactly what hid the bug. With the old, wrong constant, `existsSync` was false forever, the chain fell through to the bundled default, and every run completed cleanly. Nobody configures an explicit `voicePath` and also checks which fallback fired. A fallback that can't fire doesn't error; it just quietly gives you the wrong behavior.

## Shared paths are a contract, so treat them like one

The deeper mistake was depending on where another tool happened to keep files without either tool writing that location down. That's Hyrum's Law territory: "with a sufficient number of users of an API, it does not matter what you promise in the contract: all observable behaviors of your system will be depended on by somebody" ([hyrumslaw.com](https://www.hyrumslaw.com/)). I was the somebody, and I depended on a location I'd guessed rather than observed. Desktop Linux solved this class of problem by standardizing the bases: the XDG Base Directory Specification exists to define "where these files should be looked for by defining one or more base directories," with `$XDG_CONFIG_HOME` defaulting to `~/.config` ([freedesktop.org](https://specifications.freedesktop.org/basedir/latest/)). You don't need the spec itself; you need its move. Pick one documented base, make both tools import it from a single constant, and change it in one place.

## Verify it

Check your own fallback chains actually fire. Build a temp layout where only the middle candidate exists, hand it to the resolver, and assert the middle wins over the last-resort default:

```js
// tests/voice.test.mjs
import { test } from 'node:test';
import assert from 'node:assert/strict';
import { mkdtempSync, mkdirSync } from 'node:fs';
import { tmpdir } from 'node:os';
import { join } from 'node:path';
import { resolveVoiceDir } from '../voice.mjs';

test('the middle of the fallback chain is reachable, not dead code', () => {
  const base = mkdtempSync(join(tmpdir(), 'voice-'));
  const sibling = join(base, 'ghostwriter', 'voice');
  const bundled = join(base, 'devlog', 'voice');
  mkdirSync(sibling, { recursive: true });
  mkdirSync(bundled, { recursive: true });

  // No explicit config, sibling present: the sibling must win.
  assert.equal(resolveVoiceDir(null, [null, sibling, bundled]), sibling);
  // Sibling path wrong (doesn't exist): the default silently wins. This
  // passing line IS the bug's signature; the assertion above is the guard.
  assert.equal(resolveVoiceDir(null, [null, join(base, 'nope'), bundled]), bundled);
});
```

Run `node --test tests/voice.test.mjs`. Then do the check no unit test can: on a machine with the sibling tool installed, run the resolver with no explicit config and confirm it returns the sibling's directory. If you get your bundled default while the sibling is right there, you've found this bug.

## Gotchas

- **Hardcoding a sibling tool's data path from memory.** Symptom: the fallback never fires, no error is raised, and the tool silently uses its last-resort default while you believe your real profile is in use. Escape: one exported constant for the shared location, pointed at the other tool's documented home, and a manual check on an installed machine that the middle of the chain actually wins.
