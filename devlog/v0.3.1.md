That split is a guard clause, and it's the same code that quietly broke again a few weeks later once the package started shipping through npm.

## What a CLI script needs before it's testable

A typical Node CLI entrypoint is one file that does two jobs at once: it defines the logic (parse args, validate input, do the work) and it runs that logic, both at module load time. That's fine until you want to `import` the file from a test to call one function in isolation. Node runs top-level code in an ES module the instant it's imported, same as when it's executed directly, so without a guard, `import { validateConfig } from './cli.js'` also runs the whole CLI as a side effect.

You need three things to fix that:
- A `package.json` with `"type": "module"` so `import.meta.url` is available.
- The functions you want to test declared with `export`, not left as unreachable local closures.
- A conditional around the dispatch code so it only runs when the file is the actual entrypoint, not when it's imported.

## Export the logic, guard the dispatch

Start by pulling every function you'd want to call from a test out from under the script's execution path and exporting it directly:

```javascript
// bin/cli.js
#!/usr/bin/env node
import { fileURLToPath } from 'node:url';

export function validateConfig(config) {
  if (!config || typeof config !== 'object') {
    throw new Error('Config must be an object');
  }
  // ...real validation rules...
  return config;
}

function runCli(argv) {
  const config = JSON.parse(argv[2] ?? '{}');
  validateConfig(config);
  console.log('ok');
}
```

`validateConfig` is now a plain, testable function. `runCli` is the part that touches `process.argv` and has side effects, and it's still just a function, not code that runs on its own yet.

Now add the guard that decides whether to actually call `runCli`:

```javascript
// bin/cli.js (continued)
// Only run the CLI when this file is executed directly, not when it's imported
// by a test. Importing the module must never have side effects.
const isMain = process.argv[1] === fileURLToPath(import.meta.url);
if (isMain) {
  runCli(process.argv);
}
```

`import.meta.url` is the module's own `file:` URL ([Node.js: `import.meta.url`](https://nodejs.org/api/esm.html#importmetaurl)), and `process.argv[1]` is the absolute path Node was invoked with ([Node.js: `process.argv`](https://nodejs.org/api/process.html#processargv)). When they match, this file is the one Node was told to run, not a file some other module pulled in. `fileURLToPath` converts the `file:` URL to a plain path so the comparison is apples to apples.

## Test the exported functions directly

With the dispatch behind a guard, a test file can import the module and call its functions without ever triggering `runCli`:

```javascript
// tests/cli.test.mjs
import { test } from 'node:test';
import assert from 'node:assert/strict';
import { validateConfig } from '../bin/cli.js';

test('validateConfig rejects a non-object config', () => {
  assert.throws(() => validateConfig(null));
  assert.throws(() => validateConfig('nope'));
});

test('validateConfig accepts a valid config', () => {
  const config = { name: 'ok' };
  assert.equal(validateConfig(config), config);
});
```

Run it with Node's built-in test runner:

```bash
node --test tests/
```

You should see both tests pass, and nothing printed by `runCli`. If you see `ok` in the output alongside the test results, the guard didn't hold; the import triggered the dispatch anyway, and that's the bug to chase before writing another test.

## Gotchas

**The trap:** `process.argv[1] === fileURLToPath(import.meta.url)` looks like a straightforward identity check, but it's comparing two things that aren't guaranteed to describe the same path the same way. npm's `bin` field doesn't copy your script anywhere; it creates a symlink to it in `node_modules/.bin` (or the global bin directory), and that symlink is what actually gets invoked ([npm docs: `bin`](https://docs.npmjs.com/cli/v10/configuring-npm/package-json#bin)). `process.argv[1]` reports the path Node was launched with, which is the symlink itself, not its target ([Node.js: `process.argv`](https://nodejs.org/api/process.html#processargv)). `import.meta.url`, by contrast, is resolved through Node's module algorithm, which explicitly sets the resolved URL to the real path of the file, symlinks followed ([Node.js ESM resolution algorithm](https://nodejs.org/api/esm.html#importmetaurl)). So a direct `node bin/cli.js` invocation has both sides pointing at the same real file and the guard matches; a package installed and run through `npx` or an npm script has `argv[1]` pointing at the symlink and `import.meta.url` pointing at the resolved target, and the guard never matches.

**The symptom:** the CLI exits `0` and prints nothing. There's no error, no stack trace, no exit code to grep for in CI. The script just quietly does nothing whenever it's run the way most users actually run an installed package. It's the kind of bug that survives `node bin/cli.js` testing indefinitely because a developer running the file straight from the repo never routes through the symlink that trips it.

**The escape:** resolve both sides to their real path before comparing, with `fs.realpathSync`, which follows symlinks and returns the canonical path on disk ([Node.js: `fs.realpathSync`](https://nodejs.org/api/fs.html#fsrealpathsyncpath-options)):

```javascript
import { realpathSync } from 'node:fs';

const isMain = (() => {
  if (!process.argv[1]) return false;
  try {
    return realpathSync(process.argv[1]) === realpathSync(fileURLToPath(import.meta.url));
  } catch {
    return false;
  }
})();
```

Wrap it in a `try/catch`: `realpathSync` throws if the path doesn't exist, and a guard that can throw during module load is worse than one that's occasionally wrong. This is exactly the shape of the fix devlog shipped once this hit the published package ([`e69b6ba`](https://github.com/natejswenson/claude-skills/commit/e69b6ba6df2943f5c5fcce170e71fa6620c2cd70)); the naive `===` version had been in the codebase since the test suite first exported these functions and passed every local test the whole time, because local invocation never goes through a symlink.

To catch this in your own test suite before a user does, add a regression test that invokes the CLI through a symlink deliberately, the same way `npm`/`npx` would:

```javascript
import { symlinkSync, mkdtempSync, rmSync } from 'node:fs';
import { spawnSync } from 'node:child_process';
import { tmpdir } from 'node:os';
import { join } from 'node:path';

test('the CLI dispatches when invoked through a bin symlink', (t) => {
  const dir = mkdtempSync(join(tmpdir(), 'cli-symlink-'));
  t.after(() => rmSync(dir, { recursive: true, force: true }));
  const link = join(dir, 'cli');
  symlinkSync(new URL('../bin/cli.js', import.meta.url), link);

  const result = spawnSync(process.execPath, [link], { encoding: 'utf8' });
  assert.equal(result.status, 0, result.stderr);
  assert.match(result.stdout, /ok/, 'silent output means the guard failed to match through the symlink');
});
```

That test fails against the naive `===` guard and passes once both sides are resolved through `realpathSync`, which is exactly the coverage gap a straight-line "run it locally, tests pass" workflow never surfaces.

## Sources

- [Node.js: `import.meta.url` and ESM resolution](https://nodejs.org/api/esm.html#importmetaurl) — defines `import.meta.url` and documents that module resolution sets it to the real (symlink-resolved) path.
- [npm docs: the `bin` field](https://docs.npmjs.com/cli/v10/configuring-npm/package-json#bin) — a package's `bin` entry is exposed as a symlink, not a copy of the file.
- [Node.js: `process.argv`](https://nodejs.org/api/process.html#processargv) — `argv[1]` is the path the process was invoked with, as given, not resolved.
- [Node.js: `fs.realpathSync`](https://nodejs.org/api/fs.html#fsrealpathsyncpath-options) — resolves a path through any symlinks to its canonical location on disk.
