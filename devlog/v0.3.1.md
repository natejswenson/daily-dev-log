## What Shipped

The headline is the bug it fixes, and it is a good lesson in trusting the wrong source of truth.

`/devlog` finds new releases by listing your git tags. It only ever listed local tags, with no fetch. But if your releases are cut by CI on the remote, which is the normal setup, the tag is born on the server. A local clone that has not fetched it sees nothing, so `/devlog` reports "no new release" for a release that is already published. I hit this for real: a project shipped a release, and `/devlog` quietly skipped it because my local checkout was a few tags behind.

The fix is small. `/devlog` now runs a best-effort `git fetch --tags` for each project before it reads tags. If the fetch fails because you are offline or there is no remote, it says so and continues with whatever local tags exist, rather than aborting the whole run.

This release also moves onto shared release CI: a reusable release workflow with per-skill callers, a shared scorer, and tests for the devlog package, so every skill releases on the same path and the devlog logic has coverage behind it.

## What's Next
