## Shipped

`/devlog` finds new releases by listing your git tags, then writes an entry for any tag that doesn't have one yet. v0.3.1 fixes a discovery bug: it only ever listed local tags, with no fetch. I hit it for real. A project shipped a release, CI cut the tag on the remote, and `/devlog` reported "no new release" because my local checkout was a few tags behind. The fix is a best-effort `git fetch --tags` before reading tags, and if the fetch fails because you're offline or there's no remote, it logs a note and continues with whatever local tags exist instead of aborting the run.

It's a small patch, but it's a clean example of a mistake that's easy to make far beyond git: trusting a local copy of state that the world actually owns. Here is the full discovery path, from the refresh through finding undocumented releases to a test that proves a remote-only tag gets seen.

## A local clone is a cache, not the source of truth

The bug was reading the wrong copy of the truth. In a distributed version control system, your clone is a snapshot of the remote from whenever you last synced ([Pro Git: Working with Remotes](https://git-scm.com/book/en/v2/Git-Basics-Working-with-Remotes)). The skill never ran a fetch at all, so it only ever saw the tags that were present at clone or last-pull time. Git does auto-follow annotated tags that point at objects a normal fetch already downloaded, but that still leaves a gap: a release tag sitting on a commit you didn't otherwise fetch won't arrive unless you ask for tags explicitly with `--tags`. So when releases are born on the server, the authoritative list of "what has shipped" lives on the server, and the local tag list is just a cache that's correct until it isn't.

The skill had quietly treated that cache as the single source of truth, and the two disagreed exactly when it mattered, the moment a new release existed remotely but not locally ([Single source of truth](https://en.wikipedia.org/wiki/Single_source_of_truth)). The real fix isn't "add a fetch," it's "refresh the cache before you trust it." Any time a program reads local state to decide something the world owns, whether that's tags, a config file, or a cached API response, the question to ask is whether that copy can be stale and what reading a stale copy would do. Here it silently skipped a release, which is the worst kind of wrong because nothing errored.

## Refresh before you read

The build is a discovery function that refreshes the cache, then reads it. The fetch comes first so the local tag list reflects what the server knows; the read happens afterward against that freshened list. The order is the whole point.

```bash
# devlog/discover.sh
# Refresh remote tags, then read the local list. Best-effort: if there's no
# remote or you're offline, warn and fall back to whatever local tags exist
# rather than aborting the run.
discover_tags() {
  local repo="$1"

  if git -C "$repo" remote get-url origin >/dev/null 2>&1; then
    if ! git -C "$repo" fetch --tags --quiet origin 2>/dev/null; then
      echo "warn: fetch failed for $repo; using local tags (may be stale)" >&2
    fi
  else
    echo "warn: no 'origin' remote for $repo; using local tags only" >&2
  fi

  # Now authoritative as of the fetch above (or honestly stale, and we said so).
  git -C "$repo" tag --list --sort=-creatordate
}
```

The fetch is the whole fix. The bug was that discovery never fetched at all, so `--tags` is what makes a remote-only release tag arrive before we read the list. Note the fetch is read-only: it only adds tags, it never deletes anything local, which is what a "find what shipped" step should do. If you also want to notice tags CI deleted on the remote, that's an opt-in, not the default: adding `--prune --prune-tags` will drop local tags no longer on the remote, but it will also delete purely local tags you created and never pushed, so reach for it only when you actually want that. Sorting by `creatordate` gives newest-first, which is the order the entry writer wants.

## Discover what hasn't been written yet

Refreshed tags are only half the job. The skill writes an entry per release, so discovery has to subtract the tags that already have a file from the tags that exist. That diff is what produces the actual work list, and it's the call site that consumes `discover_tags`.

```bash
# devlog/discover.sh (continued)
# Print every tag that does not yet have a devlog entry on disk.
find_undocumented_releases() {
  local repo="$1" content_dir="$2"

  # Versions we've already written, derived from entry filenames (v0.3.1.md -> v0.3.1).
  # mkdir -p so the first-ever run (no content dir yet) doesn't make find error out.
  mkdir -p "$content_dir"
  local documented
  documented="$(find "$content_dir" -name 'v*.md' -exec basename {} .md \; | sort -u)"

  discover_tags "$repo" | while read -r tag; do
    if ! grep -qxF "$tag" <<<"$documented"; then
      echo "$tag"   # a release with no entry yet
    fi
  done
}

# Call site: anything printed here is a release /devlog still needs to write up.
new=$(find_undocumented_releases "$HOME/code/myproject" "./content/devlog/myproject")
if [ -z "$new" ]; then
  echo "no new release"
else
  echo "writing entries for:"; echo "$new"
fi
```

Before the fix, `discover_tags` was just `git tag --list`, so the moment the remote was ahead the diff came back empty and the skill printed "no new release" with total confidence. With the refresh in front, the same run now sees the remote-only tag and queues it. On an offline run you get the warning on stderr and the best-available answer on stdout, for example:

```text
warn: fetch failed for /home/me/code/myproject; using local tags (may be stale)
writing entries for:
v0.3.1
```

## Fail soft when the refresh can't happen

The fetch is best-effort on purpose. A devlog tool that refused to run without a network would be worse than one that occasionally works from a slightly stale cache, so a failed fetch logs a note and falls back to local tags rather than aborting. That's the same principle behind graceful degradation: keep partial, useful function during a failure instead of shutting down, and make the reduced mode visible rather than silent. The cited guidance frames this around overload and load-shedding, but the general principle carries straight over to a dependency or network failure like a missing remote ([Google Cloud: Design for graceful degradation](https://docs.cloud.google.com/architecture/framework/reliability/graceful-degradation)).

The visibility is what keeps "fail soft" from becoming "fail silent." The warning goes to stderr while the tag list goes to stdout, so a human sees the degraded-mode note and a downstream pipe still gets clean data. The only failure this design refuses to absorb is the original one, treating a stale cache as authoritative without anyone knowing.

## Verify it

The bug was invisible because it depended on local and remote disagreeing, a state that's annoying to reach by hand. A test can manufacture it deterministically: stand up a bare repo as the remote, clone it, then create a tag directly on the remote so the clone is genuinely behind. Good discovery must surface that tag; broken discovery returns nothing.

```bash
# tests/test_discover.sh
set -euo pipefail
. devlog/discover.sh

# Set a git identity for this process so `commit` works on a bare CI runner.
export GIT_AUTHOR_NAME=t GIT_AUTHOR_EMAIL=t@t.t
export GIT_COMMITTER_NAME=t GIT_COMMITTER_EMAIL=t@t.t

tmp=$(mktemp -d)
git init --bare -q "$tmp/remote.git"
git clone -q "$tmp/remote.git" "$tmp/clone"
git -C "$tmp/clone" commit -q --allow-empty -m init
git -C "$tmp/clone" push -q origin HEAD:main

# Cut a release on the remote only; the clone never fetched it.
work=$(mktemp -d)
git clone -q "$tmp/remote.git" "$work"
git -C "$work" commit -q --allow-empty -m release
git -C "$work" tag v0.3.1
git -C "$work" push -q origin v0.3.1

# 1. refresh-before-read finds the remote-only tag.
# Note: under `set -e`, `cmd && echo ok` would NOT fail the run on a false left
# side, so a regression reads green. Each check has to hard-fail explicitly.
if ! discover_tags "$tmp/clone" | grep -qxF v0.3.1; then
  echo "FAIL: remote-only tag was not discovered" >&2; exit 1
fi
echo "ok: remote-only tag discovered"

# 2. fail-soft: a dead remote degrades to local tags instead of crashing
git -C "$tmp/clone" remote set-url origin /nonexistent.git
git -C "$tmp/clone" tag v0.0.1            # one local tag to fall back to
out=$(discover_tags "$tmp/clone" 2>/dev/null)
if ! grep -qxF v0.0.1 <<<"$out"; then
  echo "FAIL: degraded mode did not return local tags" >&2; exit 1
fi
echo "ok: degraded to local tags"
```

The first assertion fails against the old `git tag --list` discovery, which is exactly the regression that shipped. The second fails if a missing or broken remote ever starts aborting the run instead of warning and continuing.
