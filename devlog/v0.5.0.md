## Shipped

v0.5.0 is the biggest change this project has had. The skill that writes these posts used to be a markdown file instructing an LLM to run about twenty bash steps: list git tags, filter them with a regex, compute commit ranges, probe the GitHub API once per tag. Now a CLI does all of that in tested code, and the LLM's job starts where judgment starts. The same release added the other half: generated posts are now checked by a deterministic lint and scored by a budget-capped LLM judge. The pattern behind both halves is worth building yourself, whatever your agent generates.

## Draw the line between deterministic and judged

The rule I used to split the work: if two runs should produce the same answer, it's code and it gets tests; if the output is prose or judgment, it's the model and it gets evaluated. That's consistent with Anthropic's own guidance to prefer "finding the simplest solution possible" when building with LLMs, because "workflows offer predictability and consistency for well-defined tasks, whereas agents are the better option when flexibility and model-driven decision-making are needed" ([Anthropic: Building effective agents](https://www.anthropic.com/research/building-effective-agents)). Tag discovery is a well-defined task. Writing a post isn't. My mistake in earlier versions was letting the model do both.

There's a security bonus hiding in this split. Git tag names are attacker-influenceable input, and my old skill spent about eighty lines of markdown teaching the model to quote them safely for the shell. Once discovery moved into code that calls `spawnSync` with argv arrays, tag names stopped passing through any shell at all, and most of those instructions got deleted.

## Build the plan command

The deterministic side becomes one command that emits a JSON plan. Here's a minimal version of devlog's `scan`: it lists a repo's release tags, filters to final releases, and pairs each with the previous release as its range base. This file is complete and runnable as `node scan.mjs /path/to/repo`:

```js
// scan.mjs
import { spawnSync } from 'node:child_process';

const FINAL = /^v\d+(\.\d+)*$/; // final releases only; no rc/prerelease/build tags

function git(repo, ...args) {
  const r = spawnSync('git', ['-C', repo, ...args], { encoding: 'utf8' });
  return r.status === 0 ? r.stdout.trim() : null;
}

export function plan(repo) {
  const tags = (git(repo, 'tag', '--list', 'v*', '--sort=-v:refname') || '')
    .split('\n').filter((t) => FINAL.test(t));
  return tags.map((tag, i) => ({
    tag,
    prevTag: tags[i + 1] ?? null,
    commits: (git(repo, 'log', tags[i + 1] ? `${tags[i + 1]}..${tag}` : tag,
      '--format=%H|%s') || '').split('\n').filter(Boolean)
      .map((l) => ({ hash: l.slice(0, 40), subject: l.slice(41) })),
  }));
}

console.log(JSON.stringify(plan(process.argv[2] ?? '.'), null, 2));
```

Two design choices matter more than the git details. First, argv arrays instead of shell strings, so untrusted tag names can't inject anything. Second, the output is one dense JSON object rather than twenty intermediate results, which is the shape Anthropic recommends for agent-facing tools: "return only high signal information back to agents," and consolidate "multiple discrete operations (or API calls) under the hood" ([Anthropic: Writing tools for agents](https://www.anthropic.com/engineering/writing-tools-for-agents)). Your agent instructions collapse to "run scan, read the plan."

## Gate the generated output with a lint

The model writes the post, but the post has a contract: required sections, a minimum number of distinct cited sources, every code fence language-tagged. Contracts are deterministic, so they're code too. A useful lint fits in one function:

```js
// lint.mjs
import { readFileSync } from 'node:fs';

export function lint(text, { minSources = 3 } = {}) {
  const findings = [];
  for (const section of ['## Shipped', '## Gotchas', '## Sources']) {
    if (!text.includes(section)) findings.push(`missing "${section}"`);
  }
  const urls = new Set([...text.matchAll(/\]\((https?:\/\/[^)\s]+)\)/g)].map((m) => m[1]));
  if (urls.size < minSources) findings.push(`need ${minSources} distinct sources, found ${urls.size}`);
  let inFence = false;
  for (const line of text.split('\n')) {
    const m = /^```(.*)$/.exec(line);
    if (!m) continue;
    if (!inFence && m[1].trim() === '') findings.push('untagged code fence');
    inFence = !inFence;
  }
  return findings;
}

const findings = lint(readFileSync(process.argv[2], 'utf8'));
console.log(JSON.stringify({ ok: findings.length === 0, findings }, null, 2));
process.exit(findings.length === 0 ? 0 : 1);
```

Anything the lint catches costs zero tokens to catch. In devlog, the agent must run this on its own draft and fix every finding before it's allowed to publish.

## Add a judge for what the lint can't see

A post can pass every mechanical check and still be bad: fragments of code around an essay, verification that only proves the author's repo works. Grading that is a judgment call, and LLM judges are a workable proxy; the MT-Bench work found GPT-4's judgments agreed with human preferences over 80% of the time, calling LLM-as-a-judge "a scalable and explainable way to approximate human preferences" ([Zheng et al., 2023](https://arxiv.org/abs/2306.05685)).

One prerequisite for the live path: this judge shells out to the `claude` CLI (Claude Code), so it needs that installed and authenticated. Three rules keep it honest and affordable, and all three are in the code below. Lint failures fail first, before any model call. No API key means no spend, ever; the judge degrades to a labeled lint-only result instead of a surprise bill in CI. And live calls run behind a cap that refuses to start, not one that apologizes afterward:

```js
// judge.mjs (uses lint() from lint.mjs above; live scoring shells out to the
// `claude` CLI, so install and authenticate Claude Code for the live path)
import { spawnSync } from 'node:child_process';
import { lint } from './lint.mjs';

const PROMPT = (post) => `Score this post 0-10 against: a stranger can rebuild the
technique from the post alone; code blocks define what they reference; gotchas are
concrete trap/symptom/escape items. Return ONLY JSON: {"score": <0-10>}
=== POST ===
${post}`;

export function judge(post, { cap = 0.50, spent = 0, price = 0.005 } = {}) {
  if (lint(post).length > 0) return { score: 0, pass: false, mocked: false }; // $0 fail-fast
  if (!process.env.ANTHROPIC_API_KEY) {
    // No key (e.g. CI): never spend. Label the result so a lint-only pass
    // can't masquerade as a judged pass.
    return { score: null, pass: true, mocked: true };
  }
  const estimate = ((6000 + post.length / 4 + 800) / 1000) * price;
  if (spent + estimate > cap) throw new Error(`would exceed $${cap} cap`);
  const r = spawnSync('claude', ['-p', PROMPT(post), '--model', 'claude-haiku-4-5-20251001'],
    { encoding: 'utf8' });
  const { score } = JSON.parse(/\{[\s\S]*\}/.exec(r.stdout)[0]);
  return { score, pass: score >= 7, mocked: false };
}

// Runnable directly, same as the other two: node judge.mjs somepost.md
import { readFileSync } from 'node:fs';
console.log(JSON.stringify(judge(readFileSync(process.argv[2], 'utf8')), null, 2));
```

Then pin the judge itself with golden fixtures: one post that should pass, one that should fail the lint, and one that passes the lint but should fail the judge. That third fixture is the whole reason the judge exists.

## Verify it

Run the three pieces against your own files. `node scan.mjs .` on any repo with version tags should print a plan whose newest entry has `prevTag` set to the second-newest tag. `node lint.mjs somepost.md` on a file missing a Gotchas section should exit 1 and name it. For the judge, first run `node judge.mjs somepost.md` with no `ANTHROPIC_API_KEY` set and confirm it prints `"mocked": true` with a null score and no spend. Then, with the key set and the `claude` CLI authenticated, feed it a deliberately hollow post (all the right sections, code that references functions you never defined) and expect a score under 7; when I ran devlog's version of that fixture through a live judge, it scored 3.0 and named the undefined helpers as the worst problem.

## Gotchas

- **Your docs and your argument parser will drift.** My skill instructions said `scan --json`; the parser didn't declare a `--json` flag, so the CLI crashed on the exact invocation the docs spelled out. Symptom: every unit test green, the documented command broken. Escape: end-to-end tests that spawn the real binary with the literal command lines your instructions contain.
- **Mock mode passes vacuously.** With no API key, the judge above degrades to a lint-only result, and my lint-clean-but-hollow fixture "passed" in mock. Symptom: CI says your quality gate works when only the free half ran. Escape: label the degraded result (the `mocked: true` field above) and have your eval runner report live-only fixtures as skipped, never as passed, so the gap stays visible in every run.
