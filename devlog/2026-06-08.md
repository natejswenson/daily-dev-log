## What I Built

After folding three separate skills into a single monorepo, the dev-log generator had a problem: it gathered *every* commit in a repo and couldn't tell which skill a given commit belonged to. So I gave it a sense of place.

The new `pathFilter` config field scopes a project's commit gathering to a repo-relative subdirectory (e.g. `skills/devlog`). Several logical projects can now share the same `path` and `remote` while each collects only its own subtree's commits — `git log` simply grows a `-- <pathFilter>` pathspec. Commit links still resolve to the shared repo, so a filtered commit links exactly as before. A commit that legitimately spans two skills shows up in both feeds, which is the honest outcome.

Beyond the core mechanic, I documented the field in `SKILL.md` (schema, a security-validation row, and the Step 2 pathspec), and taught the `bin/devlog.js` CLI to validate `pathFilter` the same way it guards every other shell-interpolated value — no leading dash or slash, no `..`, single-quoted — and to surface it as a `scope:` line in `devlog config`. Shipped as 0.2.0 through the skill's own release workflow.

## What's Next

Watch the first real multi-skill run land entries in the right feeds, and consider letting `init`/`add-project` prompt for `pathFilter` interactively so monorepo setups don't need a hand-edit.
