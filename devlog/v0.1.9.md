## Shipped

The drop-in React component for rendering a dev log had an accessibility gap. Each entry's expand/collapse was a bare `onClick` on the `<article>` element, with no role, no `tabIndex`, no `aria-expanded`, and no keyboard handler. The README tells people to copy that component into their own site, so anyone who did inherited a feed that keyboard and screen-reader users could not operate. v0.1.9 reworks the toggle into a proper disclosure control on the entry header: `role="button"` with `tabIndex`, `aria-expanded` and `aria-controls` for screen-reader semantics, an Enter/Space handler that prevents Space's default so the page doesn't scroll, and a `:focus-visible` outline so keyboard focus is visible. The visuals are unchanged; padding, hover, and cursor just moved from the entry to the header.

Moving the control to the header instead of wrapping the whole card also fixed two quieter problems: the expanded body's links are no longer trapped inside an interactive ancestor, and normal text selection works again. The part worth writing about is that this was never one site's bug. A component meant to be copied carries its defects into every copy, so this walks the full fix end to end: the inaccessible starting point, the accessible disclosure that replaces it, the CSS that makes focus visible, the call site, and a test that proves the keyboard path works.

## The bug: a bare onClick is not a button

Here is the shape of what shipped before. It looks fine, because for a mouse user it is fine.

```tsx
// BEFORE: works for a mouse, invisible to everything else.
function Entry({ title, body, expanded, onToggle }) {
  return (
    <article onClick={onToggle}>
      <h3>{title}</h3>
      {expanded && <div className="entry-body">{body}</div>}
    </article>
  );
}
```

Two things are broken, and neither shows up in a quick click-test. First, `onClick` on a non-interactive element only fires for the pointer. The `<article>` is not in the tab order, so a keyboard user can never reach it, which fails the requirement that all functionality be operable from a keyboard ([W3C: Understanding SC 2.1.1 Keyboard](https://www.w3.org/WAI/WCAG21/Understanding/keyboard.html)). Second, the element carries no interactive role and no announced state, so a screen reader reads a block of text with no hint that it does anything or whether it is open. The `<article>` does expose an implicit `article` role, but that describes a document section, not a control, and it conveys nothing about expanded or collapsed. That is exactly the gap the Name, Role, Value criterion exists to close ([W3C: Understanding SC 4.1.2 Name, Role, Value](https://www.w3.org/WAI/WCAG22/Understanding/name-role-value.html)). A custom control built from a generic element has no button role and no announced state unless you add them.

## Build the disclosure header

A show/hide toggle is a named, specified widget: the disclosure pattern. The WAI-ARIA Authoring Practices Guide defines it as a button that controls the visibility of a section of content, carrying `aria-expanded` (true when the content is visible, false when hidden) and optionally `aria-controls` pointing at the controlled region, activated by both Enter and Space ([W3C APG: Disclosure Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/disclosure/)).

Because this is a custom control rather than a native `<button>`, every piece has to be added by hand. MDN's checklist for `role="button"` is the spec to follow: make the element focusable with `tabIndex`, handle `click` for the pointer, and add a `keydown` handler for Enter and Space, because the click event will not fire for the keyboard. The easy-to-miss detail is that Space scrolls the page by default, so the handler has to call `preventDefault()` for Space ([MDN: ARIA button role](https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Roles/button_role)).

```tsx
// EntryHeader.tsx
import type { KeyboardEvent } from "react";

type Props = {
  id: string;
  title: string;
  expanded: boolean;
  onToggle: () => void;
};

export function EntryHeader({ id, title, expanded, onToggle }: Props) {
  function handleKeyDown(e: KeyboardEvent) {
    if (e.key === "Enter" || e.key === " ") {
      e.preventDefault();   // Space scrolls the page by default; stop that first
      onToggle();
    }
  }

  return (
    <div
      role="button"                 // a generic element needs an explicit role
      tabIndex={0}                  // ...and must be put in the tab order
      aria-expanded={expanded}      // announces open/closed state to screen readers
      aria-controls={`body-${id}`}  // points at the region this toggles
      onClick={onToggle}            // pointer activation
      onKeyDown={handleKeyDown}     // keyboard activation; click won't fire for keys
      className="entry-header"      // :focus-visible outline lives here
    >
      {title}
    </div>
  );
}
```

One honest caveat: a native `<button>` gives you focusability, the role, and Enter/Space activation for free, and both the APG and MDN recommend reaching for it first. The custom-element version is what you use when the clickable region cannot be a `<button>`, and the cost of that choice is that you now own every behavior the browser would otherwise have handled.

## Make keyboard focus visible

Adding `tabIndex` puts the header in the tab order, but a keyboard user also has to see where focus is, and many resets strip the default outline. `:focus-visible` is the right selector here: it shows a ring for keyboard focus without painting one on every mouse click. The controlled region carries the matching `id` that `aria-controls` points at.

```css
/* entry.css */
.entry-header {
  cursor: pointer;
  padding: 0.75rem 1rem;
}

.entry-header:hover {
  background: #f4f4f5;
}

/* Visible ring for keyboard focus, quiet for the mouse. */
.entry-header:focus-visible {
  outline: 2px solid #2563eb;
  outline-offset: 2px;
}
```

## Use it in the feed

The header is one entry's control; the feed renders a list of them and owns which entries are open. Each `aria-controls` target gets the matching `id`, so the toggle and its body are wired together for assistive tech.

```tsx
// DevLogFeed.tsx
import { useState } from "react";
import { EntryHeader } from "./EntryHeader";

type Entry = { id: string; title: string; body: string };

export function DevLogFeed({ entries }: { entries: Entry[] }) {
  const [openId, setOpenId] = useState<string | null>(null);

  return (
    <section className="devlog-feed">
      {entries.map((entry) => {
        const expanded = entry.id === openId;
        return (
          <article key={entry.id}>
            <EntryHeader
              id={entry.id}
              title={entry.title}
              expanded={expanded}
              onToggle={() => setOpenId(expanded ? null : entry.id)}
            />
            <div id={`body-${entry.id}`} hidden={!expanded} className="entry-body">
              {entry.body}
            </div>
          </article>
        );
      })}
    </section>
  );
}
```

The body uses `hidden` rather than unmounting, so `aria-controls` always references a real element and a screen reader can find the region it points at.

## Verify with a keyboard test

The two behaviors worth defending are that the keyboard activates the toggle and that the state is announced. Testing Library is built for this; it queries by role and simulates real keyboard events, so the test exercises the same path a keyboard user takes ([Testing Library: user-event keyboard](https://testing-library.com/docs/user-event/keyboard/)). Both Enter and Space must toggle `aria-expanded` and show or hide the body.

```tsx
// DevLogFeed.test.tsx
import "@testing-library/jest-dom";   // registers toBeVisible / toHaveAttribute matchers
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { DevLogFeed } from "./DevLogFeed";

const entries = [{ id: "1", title: "First post", body: "Body text" }];

test("Enter and Space toggle the disclosure", async () => {
  const user = userEvent.setup();
  render(<DevLogFeed entries={entries} />);

  const toggle = screen.getByRole("button", { name: "First post" });
  expect(toggle).toHaveAttribute("aria-expanded", "false");
  expect(screen.queryByText("Body text")).not.toBeVisible();

  toggle.focus();                                  // reach it the way a keyboard user does
  await user.keyboard("{Enter}");
  expect(toggle).toHaveAttribute("aria-expanded", "true");
  expect(screen.getByText("Body text")).toBeVisible();

  await user.keyboard(" ");                         // Space collapses it again
  expect(toggle).toHaveAttribute("aria-expanded", "false");
  expect(screen.queryByText("Body text")).not.toBeVisible();
});
```

`getByRole("button", ...)` is itself part of the assertion: it only finds the element because the role is now exposed, so the test fails against the bare-`onClick` version before it ever presses a key. Run it against the old `<article>` and the query throws; run it against the disclosure and the keyboard path is proven.

## Why this earned its own release

The reason a keyboard fix was worth cutting a version for is the distribution model. This component isn't an internal page; the README points people to copy it into their own sites. A reusable artifact is a multiplier, and it multiplies whatever it contains, including its bugs. One inaccessible toggle in a template becomes an inaccessible toggle on every site that adopts it, and most adopters will never audit the keyboard path; they trust that a published component does the obvious thing right. The flip side is the encouraging part: fixing the template fixes every future copy by default, so anyone pulling 0.1.9 now gets the accessible version without doing anything. The standard for a shared component is higher than "works for me"; it's "the accessible behavior is the one you get when you copy it and change nothing," because the default is what actually propagates.

## Sources

- [W3C APG: Disclosure (Show/Hide) Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/disclosure/) — the specified disclosure widget: a button with `aria-expanded`/`aria-controls`, activated by both Enter and Space.
- [MDN: ARIA button role](https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Roles/button_role) — a custom button needs `tabIndex`, click and keydown handlers for Enter/Space, and `preventDefault()` on Space; prefer a native `<button>` when you can.
- [W3C: Understanding SC 2.1.1 Keyboard](https://www.w3.org/WAI/WCAG21/Understanding/keyboard.html) — all functionality must be operable through a keyboard interface.
- [W3C: Understanding SC 4.1.2 Name, Role, Value](https://www.w3.org/WAI/WCAG22/Understanding/name-role-value.html) — custom controls must expose their role, state, and value to assistive technology.
