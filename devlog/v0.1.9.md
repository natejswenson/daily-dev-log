## Shipped

The React component that ships with the devlog package for rendering a dev-log feed had an accessibility gap: each entry expanded and collapsed on a bare `onClick` attached to the whole card, with no role, no `tabIndex`, no `aria-expanded`, and no keyboard handler. Because the documented way to use the component is to copy it into your own site, that gap didn't stay contained to one place; every site that copied it inherited a feed that a keyboard or screen-reader user couldn't operate. 0.1.9 moves the toggle onto the entry's header and rebuilds it as a proper disclosure control: `role="button"` with `tabIndex`, `aria-expanded` and `aria-controls` for screen-reader semantics, an Enter/Space handler, and a `:focus-visible` outline, with the visual styling carried over unchanged.

The rest of this post builds that fix from the ground up: why a plain `onClick` on a `<div>` or `<article>` fails a keyboard user in the first place, the ARIA disclosure pattern that replaces it, the keydown handling a custom control has to do by hand, and a test that proves the keyboard path works before you ship it.

## Where a bare onClick breaks

Here's the shape of what a lot of expand/collapse components look like before anyone thinks about keyboard access. It works, because for a mouse it does work.

```jsx
// BEFORE: fine for a pointer, invisible to everything else.
function Entry({ title, body, expanded, onToggle }) {
  return (
    <article className="entry" onClick={onToggle}>
      <h3>{title}</h3>
      {expanded && <div className="entry-body">{body}</div>}
    </article>
  );
}
```

Two things are wrong, and a quick click-test won't surface either one. First, `onClick` on a non-interactive element only fires for a pointer. The `<article>` never enters the tab order, so a keyboard user has no way to reach it, which fails the requirement that all functionality be operable from a keyboard ([W3C: Understanding SC 2.1.1 Keyboard](https://www.w3.org/WAI/WCAG21/Understanding/keyboard.html)). Second, the element has no interactive role and no announced state. A screen reader reads it as a block of text with no indication that it does anything, let alone whether it's open or closed. `<article>` does carry an implicit role, but that role describes a document section, not a control, and it says nothing about expanded or collapsed. Custom controls have to expose their role, state, and value themselves; nothing does it for them ([W3C: Understanding SC 4.1.2 Name, Role, Value](https://www.w3.org/WAI/WCAG22/Understanding/name-role-value.html)).

There's a third problem that only shows up once the card renders real content: if the toggle is attached to the whole card and the expanded body contains a link or a button, clicking that nested element also fires the card's `onClick`, so a click meant to follow a link also closes the entry. The usual patch is `e.stopPropagation()` on every interactive child, which works but has to be repeated everywhere new interactive content gets added. The fix below sidesteps that category of bug rather than working around it.

## Build the disclosure header

A show/hide toggle already has a name: it's the disclosure pattern. The WAI-ARIA Authoring Practices Guide defines a disclosure as a button that controls the visibility of a section of content, carrying `aria-expanded` (true when the content is visible, false when hidden) and, optionally, `aria-controls` pointing at the region it controls, activated by both Enter and Space ([W3C APG: Disclosure Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/disclosure/)).

Because this is a custom element rather than a native `<button>`, every piece of that behavior has to be added by hand. MDN's guidance for `role="button"` lays out exactly what's missing: the element needs `tabindex` to be focusable, a `click` handler for the pointer, and a `keydown` handler for Enter and Space, since the browser only fires `click` from the keyboard on real `<button>` elements, not on elements that merely carry the role. The detail that's easy to miss is that Space scrolls the page by default, so the keydown handler has to call `preventDefault()` before it does anything else ([MDN: ARIA button role](https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Roles/button_role)).

```tsx
// EntryHeader.tsx
import type { KeyboardEvent } from "react";

type Props = {
  id: string;
  title: string;
  expanded: boolean;
  onToggle: () => void;
};

export function EntryHeader({ id, title, expanded, onToggle }: Props) {
  function handleKeyDown(e: KeyboardEvent) {
    if (e.key === "Enter" || e.key === " ") {
      e.preventDefault();   // Space scrolls the page by default; stop that first
      onToggle();
    }
  }

  return (
    <div
      className="entry-header"      // :focus-visible outline lives here
      role="button"                 // a <div> needs an explicit role to read as a control
      tabIndex={0}                  // ...and to enter the tab order at all
      aria-expanded={expanded}      // announces open/closed state to assistive tech
      aria-controls={`body-${id}`}  // points at the region this toggles
      onClick={onToggle}            // pointer activation
      onKeyDown={handleKeyDown}     // keyboard activation; click alone won't fire from a key
    >
      {title}
    </div>
  );
}
```

Putting the control on the header instead of the whole card also clears the nested-link problem from the previous section: only the header is clickable, so a link or button inside the expanded body behaves normally and never needs a `stopPropagation()` guard.

One honest caveat: a native `<button>` gets you focusability, the role, and Enter/Space activation for free, and both the APG and MDN say to reach for it first. Building a custom `role="button"` element is what you do when the interactive region genuinely can't be a `<button>` (here, because the header also needs to lay out a title and metadata that a `<button>`'s default styling fights), and the price of that choice is that you now own every behavior the browser would otherwise hand you.

## Make keyboard focus visible

Adding `tabIndex` puts the header in the tab order, but a keyboard user still needs to see where focus landed, and most CSS resets strip the browser's default outline. `:focus-visible` is the right selector for this: it draws a ring for keyboard focus without painting one on every mouse click. The region the header controls carries the matching `id` that `aria-controls` points at.

```css
/* entry.css */
.entry-header {
  cursor: pointer;
  padding: 0.75rem 1rem;
}

.entry-header:hover {
  background: #f4f4f5;
}

/* Visible ring for keyboard focus, quiet for the mouse. */
.entry-header:focus-visible {
  outline: 2px solid #2563eb;
  outline-offset: -2px;
  border-radius: 4px;
}
```

## Wire it into the feed, then verify with a keyboard test

The header controls one entry; a feed renders a list of them and owns which one is open. The controlled region has to keep the `id` that `aria-controls` names even while it's collapsed. If you conditionally unmount that region instead, `aria-controls` points at an id that doesn't exist in the DOM, and assistive tech has nothing to find. Rendering the region unconditionally and toggling it with the `hidden` attribute keeps the reference valid in both states.

```tsx
// EntryFeed.tsx
import { useState } from "react";
import { EntryHeader } from "./EntryHeader";

type EntryData = { id: string; title: string; body: string };

export function EntryFeed({ entries }: { entries: EntryData[] }) {
  const [openId, setOpenId] = useState<string | null>(null);

  return (
    <section className="entry-feed">
      {entries.map((entry) => {
        const expanded = entry.id === openId;
        return (
          <article key={entry.id}>
            <EntryHeader
              id={entry.id}
              title={entry.title}
              expanded={expanded}
              onToggle={() => setOpenId(expanded ? null : entry.id)}
            />
            <div id={`body-${entry.id}`} hidden={!expanded} className="entry-body">
              {entry.body}
            </div>
          </article>
        );
      })}
    </section>
  );
}
```

Two behaviors are worth defending with a test: the keyboard activates the toggle, and the state gets announced through `aria-expanded`. Testing Library is built for exactly this; it queries by role rather than by CSS selector, and its `user-event` package simulates real keyboard events instead of firing synthetic ones, so the test walks the same path a keyboard user takes ([Testing Library: user-event keyboard](https://testing-library.com/docs/user-event/keyboard/)). If you're wiring this into an existing Vitest or Jest suite with Testing Library already installed, this drops in as-is.

```tsx
// EntryFeed.test.tsx
import "@testing-library/jest-dom";
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { EntryFeed } from "./EntryFeed";

const entries = [{ id: "1", title: "First post", body: "Body text" }];

test("Enter and Space toggle the disclosure", async () => {
  const user = userEvent.setup();
  render(<EntryFeed entries={entries} />);

  const toggle = screen.getByRole("button", { name: "First post" });
  expect(toggle).toHaveAttribute("aria-expanded", "false");
  expect(screen.queryByText("Body text")).not.toBeVisible();

  toggle.focus();                    // reach it the way a keyboard user does
  await user.keyboard("{Enter}");
  expect(toggle).toHaveAttribute("aria-expanded", "true");
  expect(screen.getByText("Body text")).toBeVisible();

  await user.keyboard(" ");          // Space collapses it again
  expect(toggle).toHaveAttribute("aria-expanded", "false");
  expect(screen.queryByText("Body text")).not.toBeVisible();
});
```

Run it and you should see:

```bash
$ npx vitest run EntryFeed.test.tsx
✓ EntryFeed > Enter and Space toggle the disclosure

Test Files  1 passed (1)
     Tests  1 passed (1)
```

`getByRole("button", ...)` is itself part of the proof: it only finds the element because a role is now exposed. Run the same test against the original bare-`onClick` version and that query throws before a single key gets pressed, because there's no button role to find. That failure is the clearest evidence that the underlying behavior changed, not just the visuals.

## Gotchas

**Attaching the toggle to the whole card, not just the header.** The card-level `onClick` from the "before" version looks simpler, but it means every interactive element inside the expanded body (a link, a button, a copy-to-clipboard control) fires the card's toggle too when clicked. The fix people reach for first is `e.stopPropagation()` on each nested element, which works until the next person adds new content and forgets it. Scoping the click and keydown handlers to the header instead of the card removes the whole category of bug; content inside the body never has an interactive ancestor to escape.

**Space scrolling the page.** The browser's default behavior for the Space key is to scroll down, and that default fires before your keydown handler if you don't stop it. Skip `preventDefault()` and pressing Space to open an entry also scrolls the page out from under the user. Call `preventDefault()` for both Enter and Space, not just Space; it's harmless on Enter and easy to forget on the one case that needs it.

**Unmounting the controlled region instead of hiding it.** It's tempting to render the body only when `expanded` is true, since that's less DOM than always rendering it. But `aria-controls` has to reference an element that exists, in both the expanded and collapsed states, or the reference is dangling and assistive tech has nothing to associate with the toggle. Keep the region mounted and hide it with the `hidden` attribute (or an equivalent CSS approach) instead of conditionally rendering it out of the tree.

## Sources

- [W3C APG: Disclosure (Show/Hide) Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/disclosure/) — the specified disclosure widget: a button with `aria-expanded`/`aria-controls`, activated by both Enter and Space.
- [MDN: ARIA button role](https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Roles/button_role) — a custom `role="button"` element needs `tabindex`, click and keydown handlers for Enter/Space, and native `<button>` is preferred when it's an option.
