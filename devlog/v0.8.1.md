## Shipped

devlog v0.7.0 through v0.8.1 added a `private` project type that scans commits without ever linking them, and a full cover-image pipeline: every post now gets a rendered 1600x900 PNG that feeds the site's feed thumbnail, post hero, OG card, and RSS enclosure. The pipeline itself, turning a hand-composed HTML/SVG document into a PNG with a headless browser, is the reusable part: no image-generation API, no billed per-image cost, just a browser you probably already have installed for testing.

The first batch of 49 backfilled covers all shared one template though, and that's most of this post: what the render pipeline actually needs, and the two production bugs that only showed up once real covers hit a real site.

## Setup: two packages, two static assets

The pipeline needs `playwright` (the headless browser) and `sharp` (PNG post-processing) as dependencies, plus two assets you commit once: a font file and a short style guide describing the palette and composition rules. Playwright's own docs describe the approach plainly: you build an HTML string with CSS and hand it to a headless browser to screenshot, and "the browser handles fonts, gradients, flexbox, and every other CSS feature" for you, so the composition step is just markup, not a drawing API ([Playwright: Screenshots](https://playwright.dev/docs/screenshots)).

```bash
npm install playwright sharp
npx playwright install chromium
```

Keep the font as a plain file on disk (a `.ttf` works fine) rather than trying to reference a system font. A screenshot has to be reproducible on any machine that runs it, including CI, so the font has to travel with the code.

## Build it: turn an HTML document into a fixed-size PNG

The render function launches a browser, sets a fixed viewport so every cover comes out at the same pixel size, loads the HTML, and then injects the font as a base64 data URI so nothing outside the process needs to resolve it:

```js
// render-cover.mjs
import { chromium } from 'playwright';
import { readFileSync } from 'node:fs';
import sharp from 'sharp';

const FONT_PATH = new URL('./font.ttf', import.meta.url);
const FONT_FAMILY = 'CoverFont';

export async function renderCover(html, { width = 1600, height = 900 } = {}) {
  const fontBase64 = readFileSync(FONT_PATH).toString('base64');
  const browser = await chromium.launch();
  try {
    const page = await browser.newPage({ viewport: { width, height } });
    await page.setContent(html, { waitUntil: 'load' });
    await page.addStyleTag({
      content: `@font-face { font-family: '${FONT_FAMILY}'; src: url(data:font/ttf;base64,${fontBase64}) format('truetype'); }`,
    });
    await page.evaluate((family) => document.fonts.load(`1em '${family}'`), FONT_FAMILY);
    await page.evaluate(() => document.fonts.ready);
    const png = await page.screenshot({ type: 'png' });
    return await sharp(png).png({ palette: true }).toBuffer();
  } finally {
    await browser.close();
  }
}
```

The two `page.evaluate()` calls matter more than they look like they should. `addStyleTag()` injects the `@font-face` rule after the page has already parsed, so the browser hasn't necessarily started loading that font yet. `document.fonts.load()` kicks off the load explicitly, and `document.fonts.ready` is a promise that, per MDN, "will only resolve once the document has completed loading fonts, layout operations are completed, and no further font loads are needed" ([MDN: FontFaceSet.ready](https://developer.mozilla.org/en-US/docs/Web/API/FontFaceSet/ready)). Screenshot before that resolves and you get the fallback sans-serif on every cover, silently, because nothing throws.

## Build it: keep the file small without visible banding

Flat illustrations compress differently than photos. Running the same 1600x900 test render through `sharp`'s default PNG encoder versus its palette option on this machine:

```text
raw screenshot bytes: 13280
plain png() bytes: 27306
palette:true bytes: 4989
```

`palette: true` tells sharp to "quantise to a palette-based image with alpha transparency support" instead of encoding full 24-bit color ([sharp: output options](https://sharp.pixelplumbing.com/api-output)). For flat fills and line art that's a straight size win with no visible loss. It's the wrong tool for anything with a smooth gradient though: gradients band visibly once they're reduced to a small color palette, which is exactly why the style guide for these covers explicitly asks for flat, limited color over large gradients.

## Use it, then verify it

```js
// make-cover.mjs
import { writeFileSync } from 'node:fs';
import { renderCover } from './render-cover.mjs';

const html = `<!DOCTYPE html>
<html><head><style>
  html, body { margin: 0; width: 1600px; height: 900px; background: #0a0a0b; }
  h1 { font-family: 'CoverFont', sans-serif; color: #ededed; font-size: 64px; }
</style></head>
<body><h1>Hello, cover</h1></body></html>`;

const png = await renderCover(html);
writeFileSync('cover.png', png);
console.log(`wrote cover.png (${png.length} bytes)`);
```

```bash
$ node make-cover.mjs
wrote cover.png (4989 bytes)
$ node -e "require('sharp')('cover.png').metadata().then(m => console.log(m.width, m.height, m.format))"
1600 900 png
```

A static-site build is exactly the case this approach is built for and exactly the case a hosted image API is worst at. One recent writeup of the same tradeoff, picking Playwright over `@vercel/og` and `node-canvas` for a static Astro site, put it plainly: Playwright is "not suitable for on-demand generation" since it takes multiple seconds and a full Chromium binary per image, but for batch generation at build time "this latency doesn't matter since CI timing is controlled" ([dev.to: generating OG images with Playwright and zero API cost](https://dev.to/morinaga/what-i-learned-generating-og-images-for-articles-with-playwright-and-zero-api-cost-3c0e)). Every cover on this site renders once, at publish time, and then ships as a static file forever.

## Gotchas

**The pipeline rendered fine. The covers were still bland.** The first style guide for these covers described a palette and a font and left the actual composition to a large title, a small kicker, and an optional geometric shape. Every one of the first 49 backfilled covers took that literally: big headline text, one of three rotating stock shapes in a corner. The feedback on seeing them was direct: bland, repetitive, nothing unique per post. The fix wasn't a code change, it was rewriting the style guide to forbid the failure mode by name: one custom illustration depicting the specific mechanism the post is about, sized as the dominant element, with an explicit rule against falling back to a generic circle or checkmark when stuck for an idea ([`ef46024`](https://github.com/natejswenson/claude-skills/commit/ef460249b421ce9242ec121f982037e64e650ba0)). A style guide that only describes the palette will get you a consistent palette and nothing else; the part that actually needs to be explicit is the part you'd assume goes without saying.

**A local content cache silently overrode the field that made covers work at all.** The site that consumes these covers keeps a local copy of already-published posts for editorial preview, and that local copy wins over the remote source when both exist for the same post, which is nearly all of them. The local copy has no `cover` field in its frontmatter (there was never anywhere to put one), so the merge logic defaulted every cover-bearing post back to "no cover" the moment it also had a local copy, silently. Only 1 of the first 49 backfilled covers actually rendered on the live site until this shipped, discovered by rebuilding the site and checking the real output instead of trusting the pipeline had worked. The fix keeps the local copy for everything except the cover field, which always comes from the real source, since the cover image itself is fetched from that same real source no matter which copy of the text won.

**Once the cover carried the title, the plain-text heading above it became a second copy of the same headline.** A post with a cover was showing the exact same title twice: once inside the rendered image, once again in an `<h1>` right above it. The fix drops the separate heading for any post that has a cover and lets the card itself, image plus a small caption line for the date and read time, be the visible heading, while keeping a screen-reader-only `<h1>` so the page still has one real heading in its structure.
