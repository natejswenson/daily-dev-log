## Shipped

devlog 0.4.0 rewrote the instructions behind `/devlog`. The skill used to tell the model to summarize a release in a narrative paragraph. Now `SKILL.md` walks it through a specific sequence: derive the one engineering topic the release actually touched, research it against reputable outside sources, then write a setup-to-verify implementation guide with multiple code blocks and a `## Sources` section. A new `deepDive` config block (`topicDomains`, `minSources`) makes the citation floor and allowed topics configurable per project instead of hardcoded into prose.

Every word of that is still an instruction, not a mechanism. Nothing in the release adds code that checks whether the model cited real sources, hit the minimum, or included a runnable code block. That gap is worth closing, and it's a useful pattern beyond one skill: any time you ask a model to follow a structural contract, you eventually need a second, boring piece of code that checks whether it did.

## Setup: write the contract as data, not prose

The first move is separating "what good looks like" from the instructions that ask for it. devlog's `deepDive` block is a small example: a citation floor and a set of allowed topics, both readable by a script, not just by the model reading its own prompt.

```json
{
  "deepDive": {
    "topicDomains": ["AI", "DevOps/SRE", "software engineering"],
    "minSources": 3
  }
}
```

Generalize it slightly and you get a reusable content contract: required sections, a minimum citation count, and a requirement that at least one code block exists.

```json
{
  "requiredSections": ["Shipped", "Sources", "Changelog"],
  "minSources": 3,
  "requiresCodeBlock": true
}
```

That file is the thing you'll check against later. Keep it separate from the prompt text so both the model and your checker read the same numbers.

## Build it: compose the contract into the instructions

The prompt still has to turn that data into plain language, because the model doesn't read JSON schemas as commands. This is the same move `SKILL.md`'s Step 6a-6c makes: convert the config into an explicit, numbered sequence rather than a single "please cite your sources" line.

```python
def build_instructions(contract: dict) -> str:
    sections = ", ".join(f"`## {s}`" for s in contract["requiredSections"])
    lines = [
        "Write the post in this order:",
        f"1. Include these sections, in order: {sections}.",
        f"2. Cite at least {contract['minSources']} reputable, distinct sources "
        "under `## Sources`, each a real URL you verified supports the claim "
        "next to it.",
    ]
    if contract["requiresCodeBlock"]:
        lines.append("3. Include at least one language-tagged, runnable code block.")
    return "\n".join(lines)
```

This is the entire mechanism most people stop at: a well-specified prompt. It's necessary, and it's the improvement devlog 0.4.0 shipped. It is also not sufficient on its own, which is the part worth building next.

## Build it: a deterministic grader for what came back

The contract only means something if you check the output against it after generation, not before. The checker is deliberately dumb: parse the text, confirm the sections exist, count the links under `## Sources`, confirm a code block is present. A deterministic check can't be talked out of failing the way a model can be talked into believing it complied.

```python
import re
import pathlib

LINK = re.compile(r"\[[^\]]+\]\((https?://[^)]+)\)")
CODE_BLOCK = re.compile(r"^```[a-z]+\n.*?\n```", re.S | re.M)

def section(text: str, heading: str) -> str:
    """Body under a real `## heading` LINE, up to the next `## `. Anchoring on
    a line start matters: a post can mention "Sources" inline in its own prose,
    and a plain substring search would match that instead of the real heading."""
    start = re.search(rf"(?m)^## {re.escape(heading)}\s*$", text)
    if not start:
        return ""
    rest = text[start.end():]
    nxt = re.search(r"(?m)^## ", rest)
    return rest[: nxt.start()] if nxt else rest

def check(path: str, contract: dict) -> list[str]:
    text = pathlib.Path(path).read_text(encoding="utf-8")
    problems = [
        f"missing section: ## {h}"
        for h in contract["requiredSections"]
        if not section(text, h)
    ]
    if contract["requiresCodeBlock"] and not CODE_BLOCK.search(text):
        problems.append("no language-tagged code block")
    cited = LINK.findall(section(text, "Sources"))
    if len(cited) < contract["minSources"]:
        problems.append(f"{len(cited)} cited sources, need >= {contract['minSources']}")
    return problems
```

## Use it: run it against a real draft

Point it at whatever the model produced and read the problem list.

```bash
$ python check.py draft.md
missing section: ## Sources
1 cited sources, need >= 3
```

That output is the point: two contract violations the model's own confident prose gave no hint of. Fix the draft, or send the problem list back to the model as a revision instruction, and run the check again until the list is empty.

## Verify it: a resolvable link still isn't proof

Counting links isn't enough, because a model that fabricates a citation usually fabricates a plausible-looking one, formatting included. Research on this is blunt about how often it happens: a large-scale analysis of citations in model-assisted writing found fabrication rates from **14.23% to 94.93% across 13 tested models**, with roughly 1% of citations in a 2.2-million-reference sample turning out to be invalid ([GhostCite: A Large-Scale Analysis of Citation Validity in the Age of Large Language Models](https://arxiv.org/abs/2602.06718)). A syntactically valid Markdown link is not evidence the thing behind it is real.

The next check resolves the URL instead of just counting it:

```python
import urllib.request

def dead_links(urls: list[str], timeout: float = 10.0) -> list[str]:
    dead = []
    for url in urls:
        req = urllib.request.Request(
            url, method="HEAD", headers={"User-Agent": "content-linter"}
        )
        try:
            with urllib.request.urlopen(req, timeout=timeout) as resp:
                if resp.status >= 400:
                    dead.append(f"{url} -> HTTP {resp.status}")
        except Exception as exc:
            dead.append(f"{url} -> {type(exc).__name__}")
    return dead
```

A 200 response only proves the URL resolves, not that the page supports the claim next to it, so treat it as a screen that catches the embarrassing case, not a guarantee. When you can't hit the network at all, for example checking a claim during generation before it's published, there's a second, model-side signal: ask the model two or three indirect factual questions about the source it just cited (who wrote it, what year, what specific claim it makes) and check for contradictions. Research on reference hallucination found models are noticeably less consistent when questioned about fabricated references than real ones, which makes that kind of consistency probe a workable heuristic even without a live URL to check ([Do Language Models Know When They're Hallucinating References?](https://arxiv.org/abs/2305.18248)).

## Gotchas

**Trap:** the prompt reads well and a spot check or two comes back clean, so it's tempting to treat "the model usually complies" as good enough.
**Symptom:** the failure only shows up at volume, not on any single read, since a fabricated citation is formatted exactly like a real one.
**Escape:** run the deterministic check on every generation, not as a one-time sanity pass. Anthropic's own guidance on evaluating agents makes the same point about code-based graders generally: they're fast and objective specifically because they don't get talked out of failing, but they only work if they run continuously rather than once during development ([Anthropic: Demystifying evals for AI agents](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents)).

**Trap:** treating "the contract lives in the instructions" as done.
**Symptom:** an instruction file like `SKILL.md` is genuinely good at telling a model what to do and telling a human what the model is supposed to do ([Anthropic: Equipping agents for the real world with Agent Skills](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills)), but nothing about that format enforces compliance; a well-written instruction and an enforced invariant read identically until something violates the instruction.
**Escape:** the moment you write a structural requirement into a prompt, write the checker for it in the same sitting. Shipping the ask without the check just defers finding out it isn't being met.
