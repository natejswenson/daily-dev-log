## Shipped

v0.5.1 is a one-commit patch: the devlog CLI now actually runs when you invoke it through npx. Before the fix, `npx @natjswenson/devlog <anything>` exited 0 and printed nothing, on every version I'd ever published. The fix is two `realpathSync` calls. The useful part for you is why the bug exists at all, and why a green 118-test suite never saw it.

## Why an ESM CLI needs a main-module guard

A CLI's entry file usually leads a double life. Executed directly, it should parse arguments and run. Imported by a test suite, it should export its functions and do nothing else. In CommonJS the standard check was `require.main === module`. The recipe you'll find in most ESM CLIs today looks like this:

```js
// bin/cli.js
import { fileURLToPath } from 'node:url';

export function dispatch(argv) {
  // ...your command routing...
}

const isMain = process.argv[1] === fileURLToPath(import.meta.url);
if (isMain) dispatch(process.argv.slice(2));
```

`import.meta.url` is "the absolute `file:` URL of the module" ([Node.js ESM docs](https://nodejs.org/api/esm.html)), so converting it to a path and comparing against `process.argv[1]` looks airtight. Run `node bin/cli.js --version` and both sides are the same string. Tests import the file, `argv[1]` points at the test runner instead, and nothing dispatches. Exactly what you want.

## Where the symlink breaks it

The comparison dies the moment npm installs your package. npm doesn't copy your bin file into place; the executable "will be either linked inside the global bins directory" or, for local installs, "linked where it will be available to that package" ([npm docs: package.json bin](https://docs.npmjs.com/cli/v11/configuring-npm/package-json)). Under npx your CLI runs as `node_modules/.bin/devlog`, a symlink pointing at `../@natjswenson/devlog/bin/devlog.js`.

Now the two sides of that `===` disagree. `process.argv[1]` is the symlink path, exactly as typed. But Node's module loader resolves the entry point: by default "Node.js will dereference the link and use the actual on-disk 'real path' of the module" ([Node.js CLI docs: --preserve-symlinks](https://nodejs.org/docs/latest-v24.x/api/cli.html#--preserve-symlinks)). So `import.meta.url` is the real file while `argv[1]` is the link. They never match, `isMain` is false, and your CLI imports itself as a library and exits 0 in silence.

The fix is to resolve both sides before comparing:

```js
import { realpathSync } from 'node:fs';
import { fileURLToPath } from 'node:url';

const isMain = (() => {
  if (!process.argv[1]) return false;
  try {
    return realpathSync(process.argv[1]) === realpathSync(fileURLToPath(import.meta.url));
  } catch {
    return false;
  }
})();
```

If you can require Node 24.2+ (or 22.18+), skip all of this: `import.meta.main` is "true when the current module is the entry point of the current process," the ESM equivalent of `require.main === module` ([Node.js ESM docs](https://nodejs.org/api/esm.html)):

```js
if (import.meta.main) dispatch(process.argv.slice(2));
```

devlog still supports Node 18, so it ships the realpath version.

## Verify it

Don't take my word that your guard survives the symlink; make your suite prove it. This test reproduces npm's `.bin` layout with no npm involved:

```js
// tests/bin-symlink.test.mjs
import { test } from 'node:test';
import assert from 'node:assert/strict';
import { spawnSync } from 'node:child_process';
import { mkdtempSync, symlinkSync } from 'node:fs';
import { tmpdir } from 'node:os';
import { join, resolve } from 'node:path';

test('CLI dispatches when invoked through a bin symlink (the npx layout)', () => {
  const link = join(mkdtempSync(join(tmpdir(), 'bin-')), 'mycli');
  symlinkSync(resolve('bin/cli.js'), link);
  const r = spawnSync(process.execPath, [link, '--version'], { encoding: 'utf8' });
  assert.equal(r.status, 0);
  assert.match(r.stdout.trim(), /^\d+\.\d+\.\d+$/);
});
```

Run `node --test tests/bin-symlink.test.mjs`. With the naive guard it fails on empty stdout; with the realpath guard it passes. Exit code 0 with no output is the entire bug, so that's the exact thing to assert on.

## Gotchas

- **Comparing `argv[1]` to `import.meta.url` with `===`.** Symptom: `npx your-cli` exits 0 and prints nothing, while `node bin/cli.js` works perfectly, so every local check you do says "fine." Escape: realpath both sides, or `import.meta.main` on Node 22.18+/24.2+.
- **Testing the entry file only by direct path.** Symptom: my e2e suite spawned `node bin/devlog.js` for every documented command and stayed green while every published npx invocation was a silent no-op; I confirmed the previous release had the same bug once I finally ran the published package. Escape: at least one test invokes the CLI through a symlink, like the one above, because that's the only layout your users ever see.
