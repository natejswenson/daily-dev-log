## Shipped

v0.5.2 fixes how this dev log's `publish-entry` command orders the manifest that drives the feed. It used to sort by date only; releases cut on the same day kept insertion order, so a busy day rendered oldest-version-on-top and the newest post landed at the bottom of that day's group. Same-date ties now break by version, highest first, with each numeric component compared as a number. Dates still win overall, and legacy rows without a version keep their relative order.

It's a small diff, but it's a complete little lesson in comparator design: what your primary key is, what breaks ties, and which comparisons are safe to do as strings. That's worth building from scratch.

## The shape of the problem

Say your feed is an array of entries, each with an ISO date and a version label:

```js
// entries.js
export const entries = [
  { date: "2026-07-11", version: "v0.4.2", title: "The fallback that never fired" },
  { date: "2026-07-11", version: "v0.5.0", title: "Give the agent a plan" },
  { date: "2026-07-11", version: "v0.5.1", title: "npx ran my CLI and nothing happened" },
  { date: "2026-07-01", version: "v0.4.1", title: "Two directories deep" },
];
```

Sorting newest-first by date is one line, and it's the line I originally wrote:

```js
entries.slice().sort((a, b) => String(b.date).localeCompare(String(a.date)));
```

Comparing ISO dates as strings is legitimate, not a hack. RFC 3339 calls it out as a designed property of the format: when components run from least precise to most precise and the representations are uniform, "the date and time strings may be sorted as strings ... and a time-ordered sequence will result" ([RFC 3339, section 5.1](https://datatracker.ietf.org/doc/html/rfc3339)). `YYYY-MM-DD` qualifies, so `localeCompare` on the raw strings gives correct chronological order.

The trap is what happens when two dates are equal. A JavaScript comparator that returns zero doesn't mean "sort these however"; since ES2019 the spec guarantees `Array.prototype.sort` is stable, meaning equal elements keep their original order ([MDN: Array.prototype.sort](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Array/sort)). My manifest appends new entries at the end, so on a three-release day, stability faithfully preserved exactly the order I didn't want: oldest release first.

## Build the tie-break: compare versions numerically

The second key is the version, and the one rule that matters is: never compare version labels as strings. Semantic Versioning is explicit that "major, minor, and patch versions are always compared numerically" ([semver.org](https://semver.org/)), and string comparison gets this wrong the moment a component hits two digits: `"v0.10.0" < "v0.9.0"` lexicographically, because `"1" < "9"`.

So parse the label into numbers first, and refuse to guess when the label isn't version-shaped:

```js
// compare.js
function versionNums(entry) {
  const m = /^v(\d+(?:\.\d+)*)$/.exec(entry.version || "");
  return m ? m[1].split(".").map(Number) : null;
}

// Descending: higher version first. Entries without a parseable
// version return 0, so stable sort keeps their existing order.
export function compareVersionsDesc(a, b) {
  const va = versionNums(a);
  const vb = versionNums(b);
  if (!va || !vb) return 0;
  for (let i = 0; i < Math.max(va.length, vb.length); i++) {
    const d = (vb[i] ?? 0) - (va[i] ?? 0);
    if (d !== 0) return d;
  }
  return 0;
}
```

The `?? 0` padding makes `v1.2` and `v1.2.0` compare equal instead of crashing or mis-sorting on length. Returning `0` for unparseable versions is a deliberate lean on that ES2019 stability guarantee: rows that predate the version field don't get reshuffled, they just stay where they were.

## Compose the two keys

JavaScript's `||` makes multi-key comparators read almost like a sentence: try the first key, and if it says "equal", fall through to the next.

```js
// compare.js (continued)
export function sortEntries(entries) {
  return entries.slice().sort(
    (a, b) =>
      String(b.date).localeCompare(String(a.date)) || compareVersionsDesc(a, b)
  );
}
```

That works because a comparator's contract is just sign, not magnitude: negative means `a` first, positive means `b` first, zero means equal ([MDN: Array.prototype.sort](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Array/sort)). `localeCompare` returns 0 exactly when the dates tie, which is exactly when `||` moves on to the version.

Order of the keys is load-bearing. Date stays primary on purpose: a backported release (say `v1.9.1` tagged after `v2.0.0` shipped) should appear in the feed when it was actually released, not jump above newer work because its version number is "older". Version only ever arbitrates within a single day.

## Use it, then verify it

Drop both files in a directory and run the sort:

```js
// demo.js
import { entries } from "./entries.js";
import { sortEntries } from "./compare.js";

for (const e of sortEntries(entries)) {
  console.log(e.date, e.version.padEnd(7), e.title);
}
```

```text
$ node demo.js
2026-07-11 v0.5.1  npx ran my CLI and nothing happened
2026-07-11 v0.5.0  Give the agent a plan
2026-07-11 v0.4.2  The fallback that never fired
2026-07-01 v0.4.1  Two directories deep
```

That's real output from running these exact files. The 2026-07-11 group now leads with v0.5.1, and the date-only version of this demo prints the same group in the reverse order, which is the bug this release fixed. For a regression test, assert the one case that catches both failure modes at once:

```js
// compare.test.js  (node --test compare.test.js)
import { test } from "node:test";
import assert from "node:assert/strict";
import { sortEntries } from "./compare.js";

test("same-date ties break by numeric version, newest first", () => {
  const rows = [
    { date: "2026-07-11", version: "v0.9.0" },
    { date: "2026-07-11", version: "v0.10.0" },
  ];
  assert.equal(sortEntries(rows)[0].version, "v0.10.0");
});
```

If your comparator compares strings anywhere, this test fails with `v0.9.0` on top.

## Gotchas

**A stable sort preserves the order you forgot you had.** The original bug wasn't a wrong comparison, it was a missing one: equal dates returned 0, and ES2019's guaranteed stability ([MDN](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Array/sort)) dutifully kept append order. The symptom was subtle because it only shows on multi-release days; this log ran fine for weeks until v0.4.2, v0.5.0, and v0.5.1 all landed on 2026-07-11 and the feed buried the newest one. The escape is to make every ordering decision explicit: if two rows can tie on your primary key, either the tie-break is written down or your storage order is silently part of your sort.

**Version strings lie past single digits.** `v0.10.0` sorts below `v0.9.0` in any string comparison, and nothing warns you at nine versions and below. Parse to numbers and compare component-wise, per SemVer's numeric-comparison rule ([semver.org](https://semver.org/)). This repo hit double digits in another project's feed (a v0.10.0 through v0.21.0 run), so the fix was tested against reality the day it shipped.

**Don't promote the tie-break to a primary key.** Sorting by version first looks equivalent right up until a backported patch release exists; then it teleports out of chronological order. Keep date primary, version secondary, and your feed stays honest about when things actually happened.
