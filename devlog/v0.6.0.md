## Shipped

devlog 0.6.0 is the release where the skill's writing contract stopped being advisory. I spent a day rewriting all 38 back-catalog posts against the how-to contract, and every piece of friction the rewrites surfaced got folded back in: a topic-usability test, a fun-to-follow contract point, a self-check that assembles and runs a post's own code blocks, a verbatim-quote verification step for fetched sources. The headline change is the enforceable one: a new `sources-inline` lint rule that fails any post listing a source it never cites in the body.

That last one is a pattern you can reuse anywhere prose has rules: take the editorial sentence people keep skipping, and write the twenty-line checker that makes skipping it impossible.

## Why written rules drift

The contract already said sources must be "cited inline as markdown links AND in `## Sources`". Thirty-eight rewrites later, the writers' own notes flagged inline citation as the easiest step to skip silently: the bibliography looks complete either way, and no reader complains about a missing link they never knew should exist.

This is exactly the gap prose linters exist to close. Vale describes itself as "a command-line tool that brings code-like linting to prose" and is built around "a framework for creating and enforcing custom rules" that can fail CI builds ([Vale docs](https://vale.sh/docs)). The same instinct shows up wherever documentation has to stay true: rustdoc "supports executing your documentation examples as tests. This makes sure that examples within your documentation are up to date and working" ([rustdoc: documentation tests](https://doc.rust-lang.org/rustdoc/write-documentation/documentation-tests.html)), and Python's doctest "searches for pieces of text that look like interactive Python sessions, and then executes those sessions to verify that they work exactly as shown" ([Python: doctest](https://docs.python.org/3/library/doctest.html)). The shared idea: a rule about text becomes trustworthy the day a program checks it.

## Setup: split the post into sections

The check needs two ingredients: the URLs in the `## Sources` section, and the URLs everywhere else. So the first block is a section splitter that respects code fences (a `## heading` inside a fenced block is content, not structure):

```js
// sections.js
export function splitSections(body) {
  const sections = [];
  let current = null;
  let inFence = false;
  for (const line of body.split("\n")) {
    if (/^```/.test(line)) inFence = !inFence;
    const h = !inFence && /^##\s+(.+?)\s*$/.exec(line);
    if (h && !line.startsWith("###")) {
      current = { heading: h[1], lines: [] };
      sections.push(current);
    } else if (current) {
      current.lines.push(line);
    }
  }
  return sections.map((s) => ({ heading: s.heading, content: s.lines.join("\n") }));
}
```

## Build it: extract, normalize, compare

Markdown links look like `[title](url)`, so one regex pulls the URLs out. The subtle part is normalization: `https://example.com/docs/` and `https://example.com/docs#intro` are the same citation, and a checker that treats them as different will cry wolf until someone deletes it.

```js
// lint.js
import { splitSections } from "./sections.js";

export function extractUrls(text) {
  const urls = new Set();
  for (const m of text.matchAll(/\]\((https?:\/\/[^)\s]+)\)/g)) urls.add(m[1]);
  return urls;
}

// Ignore differences that don't change the destination.
export function normalizeUrl(url) {
  return url.replace(/#.*$/, "").replace(/\/+$/, "");
}

export function lintSourcesInline(body) {
  const sections = splitSections(body);
  const sources = sections.find((s) => s.heading === "Sources");
  if (!sources) return [{ rule: "section-sources", message: "Missing ## Sources." }];

  const inline = new Set();
  for (const s of sections) {
    if (s.heading === "Sources") continue;
    for (const u of extractUrls(s.content)) inline.add(normalizeUrl(u));
  }

  const findings = [];
  for (const u of extractUrls(sources.content)) {
    if (!inline.has(normalizeUrl(u))) {
      findings.push({
        rule: "sources-inline",
        message: `${u} is listed in Sources but never cited in the body — cite it where its claim is made, or drop it.`,
      });
    }
  }
  return findings;
}
```

Notice the finding message offers two fixes, not one. The first version of mine only said "cite it inline", and the rewrite crews immediately reported the failure mode: writers wedging a citation into a paragraph that didn't need it, just to satisfy the rule. A lint message is UX; if the rule has two legitimate resolutions, say both.

## Use it: wire an exit code

A checker only counts once something fails because of it. Ten lines make it a CLI whose exit code CI understands:

```js
// lint-cli.js
import { readFileSync } from "node:fs";
import { lintSourcesInline } from "./lint.js";

const findings = lintSourcesInline(readFileSync(process.argv[2], "utf8"));
for (const f of findings) console.error(`${f.rule}: ${f.message}`);
process.exit(findings.length ? 1 : 0);
```

Run it against a post that cites both its sources inline, then against one that lists a freeloader:

```text
$ node lint-cli.js good-post.md; echo "exit=$?"
exit=0

$ node lint-cli.js bad-post.md; echo "exit=$?"
sources-inline: https://example.com/uncited is listed in Sources but never cited in the body — cite it where its claim is made, or drop it.
exit=1
```

That's real output from running these exact files. In CI, the nonzero exit is the whole feature: the editorial rule is now a required check with the same authority as a failing unit test.

## Verify it with the rule's own edge cases

The regression tests worth writing are the ones that encode your normalization decisions, so nobody "fixes" them away later:

```js
// lint.test.js  (node --test lint.test.js)
import { test } from "node:test";
import assert from "node:assert/strict";
import { lintSourcesInline } from "./lint.js";

const post = (body) => `## Shipped\n\n${body}\n\n## Sources\n\n- [Doc](https://ex.com/docs/) — claim\n`;

test("flags a source never cited inline", () => {
  const r = lintSourcesInline(post("No links here."));
  assert.equal(r[0]?.rule, "sources-inline");
});

test("tolerates trailing-slash and fragment variants", () => {
  const r = lintSourcesInline(post("See [the doc](https://ex.com/docs#intro)."));
  assert.deepEqual(r, []);
});
```

Both pass against the code above; delete `normalizeUrl` and the second one fails, which is the point.

## Gotchas

**Your existing corpus fails the rule you just wrote.** The moment I added `sources-inline` to devlog's linter, seven of the project's own tests went red: the "known good" fixture posts listed sources they never cited, because they were written under the old, unenforced contract. That's not noise, it's the rule doing its job on day one. Budget for it: shipping a new lint rule means migrating everything the rule can see, in the same change, or the build stays red and the rule gets reverted by whoever it blocks first.

**Unnormalized URLs turn the rule into a nag.** A fragment (`#section`) or trailing slash difference between the inline citation and the bibliography row is invisible to a human and fatal to a naive string match. The symptom is false findings on posts that are actually fine; the escape is a `normalizeUrl` that strips exactly the parts that don't change the destination, and a test pinning that behavior so it survives refactors.

**A one-fix error message breeds malicious compliance.** Told only to "cite it inline", writers satisfied the rule by stuffing citations into sentences that didn't need them, which made posts worse while making the linter happy. The escape is in the message itself: name dropping the source as an equally valid fix. The rule's goal is honest bibliographies, not maximum link density.
