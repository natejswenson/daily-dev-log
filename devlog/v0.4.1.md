## Shipped

I turned this monorepo into a self-hosted Claude Code plugin marketplace a day earlier: a `marketplace.json` at the root, a `plugin.json` per skill, one `/plugin marketplace add` away from installable. It validated clean. Then I installed ghostwriter from it in Claude Desktop and got "This plugin doesn't have any skills or agents." devlog 0.4.1 is the fix for that, applied to devlog itself along with the other two skills in this repo. `SKILL.md` now lives at `skills/devlog/skills/devlog/SKILL.md`, one directory deeper than before.

That doubled path looks wrong until you know why it's there, so that's the actual post: how a plugin system decides what counts as a skill when nobody wrote the rule down as something you can check.

## The rule was a directory, not a document

`plugin.json` for a Claude Code plugin is small and mostly optional: a name, a description, maybe a version. It doesn't declare its skills anywhere in that file by default. Skills load from wherever the plugin's own `skills/` folder happens to be, and Anthropic's docs on building a marketplace confirm the shape directly: "By default, a plugin's skills load from the `skills/` directory under its source" ([Claude Code docs: create and distribute a plugin marketplace](https://code.claude.com/docs/en/plugin-marketplaces)). The walkthrough in that same doc builds a plugin at `plugins/quality-review-plugin/skills/quality-review/SKILL.md`, not `plugins/quality-review-plugin/SKILL.md`. The nesting isn't incidental; it's the mechanism.

I had `SKILL.md` sitting directly in each skill's own root, one level up from where that scan expects it. Nothing in `plugin.json` said this was wrong, and nothing in the file itself said so either, because the check isn't a field, it's a path shape. I confirmed it by brute force: I had the official Anthropic plugin marketplace already installed locally, so I grepped every `SKILL.md` across every plugin it ships. Twenty-three files, one convention, zero exceptions.

This is convention over configuration, the design principle David Heinemeier Hansson named while building Rails: "a developer only needs to specify unconventional aspects of the application" ([Wikipedia: Convention over configuration](https://en.wikipedia.org/wiki/Convention_over_configuration)). It's a good trade when you're building a plugin by hand, because you write less. It's a bad trade when you're restructuring one programmatically, because there's no schema to validate against, only a shape to imitate correctly.

## Two loaders, two answers

The part that actually cost time: the CLI didn't agree with the app. Running `claude plugin details ghostwriter@claude-skills` from the terminal reported "Skills (1) ghostwriter" against the exact layout Desktop rejected. The CLI has a fallback that treats a root-level `SKILL.md` as an implicit single skill when no `skills/` subdirectory exists at all. Desktop doesn't implement that fallback. Same plugin, same files, two different verdicts depending on which binary opened them.

VS Code's extension docs describe a comparable split for a much older plugin ecosystem: extensions declare their contributions in `package.json`, and "the `contributes` section defines extension capabilities and UI contributions" the host reads at load time ([VS Code docs: extension manifest](https://code.visualstudio.com/api/references/extension-manifest)). The manifest is the single source of truth there precisely because two different hosts reading two different heuristics is how you end up debugging a "works on my machine" bug in a distribution system, which is exactly what happened here. If discovery isn't declared, it's implemented, and every implementation is free to be a little more forgiving than the next one.

## Fixing the shape, not patching around it

The honest fix wasn't a workaround file that pointed back at the old location. That would have meant two files with the same frontmatter to keep in sync by hand, and a frontmatter drift bug is a worse bug than the one I started with. Every skill's real content, `SKILL.md` plus its scripts and tests, moved together as one unit:

```
skills/devlog/                          # plugin root
├── .claude-plugin/plugin.json
├── LICENSE, README.md, CHANGELOG.md    # plugin-level docs stay here
└── skills/
    └── devlog/                         # the auto-discovery path
        ├── SKILL.md
        ├── bin/, tests/, package.json
```

Nothing inside `SKILL.md` had to change, because everything it references (`scripts/`, `tests/`, `package.json`) moved with it; a relative path stays correct as long as the whole tree moves as one piece. The two things that couldn't move were `.claude-plugin/plugin.json`, which the manifest convention requires at the plugin root, and `CHANGELOG.md`, kept there deliberately so a human browsing the repo still finds it next to the skill's name without digging.

Two lint scripts already existed in this repo from the marketplace work the day before, one checking `plugin.json` against `SKILL.md`'s frontmatter, one checking the marketplace manifest against every plugin directory. Both needed one line changed: the path they read `SKILL.md` from, updated to the nested location:

```python
# tools/lint_plugin.py
nested_skill_dir = os.path.join(skill_dir, "skills", dir_base)
skill_md_path = os.path.join(nested_skill_dir, "SKILL.md")
package_json_path = os.path.join(nested_skill_dir, "package.json")
```

## Verify: run it against the real repo, not a mock

The check that actually mattered was empirical, not the lint script passing. The marketplace here is registered locally as a directory source, which meant I could refresh it and re-query it against my own working tree without pushing anything:

```bash
claude plugin marketplace update claude-skills
claude plugin update devlog@claude-skills
claude plugin details devlog@claude-skills
```

That returned `Skills (1) devlog` against the new path, with no `SKILL.md` left at the old one for the CLI's lenient fallback to fall back to. The absence of the old file is what made the result trustworthy: there was only one place left for the answer to come from.
