## Shipped

This release turned the skills in this monorepo into an installable Claude Code plugin marketplace: a `marketplace.json` at the repo root, a `plugin.json` per skill, and `/plugin marketplace add` replacing manual symlink installs. Installing devlog through it in Claude Desktop showed the plugin's own page with no skills listed. The fix moved `SKILL.md` one directory level deeper, and that same move broke the release workflow that reads `SKILL.md` to resolve a version number and cut a tag, because it was still reading the old path. Two bugs, one convention: this is a walkthrough of building that convention correctly, plus the two places you have to check it separately from the loader itself.

## Where a plugin's skills load from

Claude Code's plugin docs are specific about this, and it's easy to miss because it isn't a field you set, it's a location you have to match: "By default, a plugin's skills load from the `skills/` directory under its `source`" ([Claude Code docs: Create and distribute a plugin marketplace](https://code.claude.com/docs/en/plugin-marketplaces)). A plugin at `my-marketplace/plugins/quality-review-plugin` needs its skill at `my-marketplace/plugins/quality-review-plugin/skills/quality-review/SKILL.md`, not at the plugin's own root. Nothing about a root-level `SKILL.md` is invalid JSON or YAML; it just sits one directory above where the scan looks, so it never gets found.

This is [convention over configuration](https://en.wikipedia.org/wiki/Convention_over_configuration): the loader infers structure from where a file sits instead of reading a field that declares it, so "a developer only needs to specify unconventional aspects" and everything else follows the default path. It's less to write per plugin. It also means there's no schema to grep for when the location is wrong, unlike an ecosystem such as VS Code, where an extension declares its capabilities directly in a `contributes` field inside `package.json`, and the host reads that field at load time instead of inferring anything from a folder name ([VS Code docs: Extension Manifest](https://code.visualstudio.com/api/references/extension-manifest)). Neither approach is wrong, but a directory convention only works if you match the directory, and nothing before that stage tells you when you haven't.

Build the shape by hand once and it stops being abstract:

```bash
mkdir -p my-marketplace/.claude-plugin
mkdir -p my-marketplace/plugins/quality-review/.claude-plugin
mkdir -p my-marketplace/plugins/quality-review/skills/quality-review
```

```markdown
<!-- my-marketplace/plugins/quality-review/skills/quality-review/SKILL.md -->
---
name: quality-review
description: Review code for bugs, security, and performance
version: 1.0.0
---

Review the selected code or recent changes for bugs, security issues,
and performance problems. Be concise and actionable.
```

```json
// my-marketplace/plugins/quality-review/.claude-plugin/plugin.json
{
  "name": "quality-review",
  "description": "Adds a quality-review skill for quick code reviews",
  "version": "1.0.0"
}
```

```json
// my-marketplace/.claude-plugin/marketplace.json
{
  "name": "my-plugins",
  "owner": { "name": "Your Name" },
  "plugins": [
    { "name": "quality-review", "source": "./plugins/quality-review" }
  ]
}
```

Run `claude plugin validate .` from inside `my-marketplace`. It checks `marketplace.json` for schema errors and duplicate names, and validates each entry's `plugin.json` in turn. It will not catch a misplaced `SKILL.md` on its own, because a missing skill isn't a manifest error, it's just an empty scan result. That gap is exactly why the next two sections exist.

## Build it: tie the manifest to the directory with a lint check

`plugin.json`, `SKILL.md`, and (for npm-published skills) `package.json` all carry a `name` and a `version`, and nothing forces them to agree. A short script that reads all three and diffs them catches a stale field before a user does:

```python
#!/usr/bin/env python3
"""check_plugin.py <plugin-dir> -- verify plugin.json, SKILL.md, and
package.json agree on name and version, and that SKILL.md sits where
the plugin loader actually scans for it."""
import json, os, re, sys

def read_frontmatter(path):
    try:
        text = open(path, encoding="utf-8").read()
    except FileNotFoundError:
        return {}
    if not text.startswith("---"):
        return {}
    block = text.split("---", 2)[1]
    fm = {}
    for line in block.splitlines():
        m = re.match(r'^(name|version):\s*"?([^"\n]+)"?\s*$', line)
        if m:
            fm[m.group(1)] = m.group(2).strip()
    return fm

def main(plugin_dir):
    dir_name = os.path.basename(os.path.normpath(plugin_dir))
    plugin_json = json.load(open(os.path.join(plugin_dir, ".claude-plugin", "plugin.json")))
    nested = os.path.join(plugin_dir, "skills", dir_name)
    fm = read_frontmatter(os.path.join(nested, "SKILL.md"))
    if not fm:
        sys.exit(f"error: no SKILL.md at {nested}/SKILL.md (loader won't find it either)")

    pkg_path = os.path.join(nested, "package.json")
    pkg = json.load(open(pkg_path)) if os.path.isfile(pkg_path) else {}

    names = {"plugin.json": plugin_json.get("name"), "SKILL.md": fm.get("name")}
    if set(names.values()) - {dir_name}:
        sys.exit(f"error: names must all equal directory '{dir_name}': {names}")

    versions = {k: v for k, v in {
        "plugin.json": plugin_json.get("version"),
        "SKILL.md": fm.get("version"),
        "package.json": pkg.get("version"),
    }.items() if v}
    if len(set(versions.values())) > 1:
        sys.exit(f"error: version mismatch across present fields: {versions}")

    print(f"{dir_name}: clean ({versions})")

if __name__ == "__main__":
    main(sys.argv[1])
```

Run it against the tree from the previous section:

```bash
python3 check_plugin.py my-marketplace/plugins/quality-review
# -> quality-review: clean ({'plugin.json': '1.0.0', 'SKILL.md': '1.0.0'})
```

Bump `version` in only one of the two files and rerun it; it stops you before the mismatch ships. This is deliberately a name-and-version check, not a discovery check: it trusts that `skills/<dir-name>/SKILL.md` is the right path and verifies everything reading that path agrees, the same trust the plugin loader itself places in that location.

## Build it: make the release job read the same path

If a release workflow resolves a version from `SKILL.md` or `package.json` to cut a tag, it has to read those files from the exact path the plugin loader uses, not from the plugin's root. A `working-directory` set once at the top of the job is the trap: it feels like it should apply everywhere, and it does, right up until one directory in the tree moves and the rest don't.

```yaml
on:
  workflow_call:
    inputs:
      skill:
        required: true
        type: string

jobs:
  release:
    runs-on: ubuntu-latest
    defaults:
      run:
        # Applies to every step below unless a step overrides it.
        # CHANGELOG.md, LICENSE, README.md stay here at the plugin root.
        working-directory: plugins/${{ inputs.skill }}
    steps:
      - uses: actions/checkout@v7
        with:
          fetch-depth: 0

      - name: Resolve version
        id: ver
        # SKILL.md/package.json live one level deeper, where the plugin
        # loader actually scans -- override just this step.
        working-directory: plugins/${{ inputs.skill }}/skills/${{ inputs.skill }}
        run: |
          set -euo pipefail
          if [ -f package.json ]; then
            V="$(node -p "require('./package.json').version")"
          else
            V="$(awk '/^version:/{sub(/^version:[[:space:]]*/,""); gsub(/"/,""); print; exit}' SKILL.md)"
          fi
          echo "version=$V" >> "$GITHUB_OUTPUT"

      - name: Tag if new
        run: |
          TAG="${{ inputs.skill }}-v${{ steps.ver.outputs.version }}"
          if ! git rev-parse "$TAG" >/dev/null 2>&1; then
            git tag "$TAG" && git push origin "$TAG"
          fi
```

The GitHub Actions docs confirm this scoping: `working-directory` under `defaults.run` "define[s] the working directory for the `shell` for a step," and when more than one default is set for the same key, the most specific one wins ([GitHub Actions docs: Workflow syntax for GitHub Actions](https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions)). A step-level `working-directory` always beats the job-level default, which is exactly what lets `CHANGELOG.md` stay at the plugin root while `SKILL.md` reads from one level deeper in the same job.

## Use it, then verify it

Add the marketplace locally and install the plugin:

```bash
/plugin marketplace add ./my-marketplace
/plugin install quality-review@my-plugins
/quality-review:quality-review
```

If discovery is wired correctly, the slash command runs the skill. If `SKILL.md` is still sitting at the plugin root, the plugin installs with no error, and the failure only shows up as an empty skills list on the plugin's own page, which is easy to miss if you never open it.

Before trusting a CI release job with the same resolution logic, run it by hand against the real files first:

```bash
cd my-marketplace/plugins/quality-review/skills/quality-review
node -p "require('./package.json').version"
# -> 1.0.0
```

Then dispatch the actual workflow and confirm the tag landed, not just that the job reported green:

```bash
gh workflow run release.yml --ref main -f skill=quality-review
git tag -l 'quality-review-v*'
# -> quality-review-v1.0.0
```

A green checkmark on the dispatch only means the shell didn't error; the tag existing in `git tag -l` is the only proof the version resolved correctly.

## Gotchas

**A missing skill fails silent, not loud.** A `SKILL.md` at the wrong depth doesn't produce a validation error from `claude plugin validate`, a JSON parse failure, or anything else that stops a merge. It shows up as an empty skills list on the installed plugin's page, and if you never open that page after installing, you ship it without noticing. Open the actual plugin page after every install that touches directory structure, not just after the first one.

**Pull-request CI proving green doesn't prove the release job works.** A common pattern is one job that runs on every pull request and a separate release job that only runs on a push to the default branch or a manual dispatch. That split exists so a pull request isn't stuck waiting on work nobody needs yet, but it also means a bug living only in the release job's file-reading logic can sit in a merged PR for as long as nothing triggers a real release. The fix isn't a smarter default; it's running the exact resolution command by hand against the real files, as in the verify step above, before you ever dispatch the workflow for real.

**A partial move breaks whatever assumed co-location.** Moving `SKILL.md` one level deeper but leaving `CHANGELOG.md` at the plugin root is the right call (a human browsing the repo should find the changelog without digging), but any script or test that read both files relative to the same directory now has one path right and one wrong. Grep for every place that constructs a path from `SKILL.md`'s directory, not just the loader itself, before calling a directory move done.
