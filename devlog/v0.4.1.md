## Shipped

I turned this monorepo into a self-hosted Claude Code plugin marketplace a day earlier: a `marketplace.json` at the root, a `plugin.json` for each of its four skills, `claude plugin validate` green across the board. I installed ghostwriter through it in Claude Desktop to see it work, and its plugin page said: "This plugin doesn't have any skills or agents." This release, v0.4.1, is the fix, applied identically to devlog, ghostwriter, and resume (the fourth skill, github-stats, isn't tracked by this dev-log yet). All three shipped from the same root cause and the same two commits, which is exactly why this is one post and not three. Filing the same story under three project folders would have meant three summaries of the same bug and three copies of the same lesson, which teaches nothing a single honest account doesn't teach better.

## The rule was a directory, not a document

`plugin.json` for a Claude Code plugin is small, mostly just a name and a description. It doesn't declare where a skill's `SKILL.md` has to live, because there's no field for that; skills load from wherever the plugin's own `skills/` folder happens to be. Anthropic's docs on building a marketplace confirm the shape directly: "a plugin's skills load from the `skills/` directory under its source" ([Claude Code docs: create and distribute a plugin marketplace](https://code.claude.com/docs/en/plugin-marketplaces)), and the walkthrough in that same doc builds a plugin at `plugins/quality-review-plugin/skills/quality-review/SKILL.md`, not at the plugin's own root. I had `SKILL.md` sitting directly in each skill's root, one level up from where that scan expects it. Nothing in `plugin.json` said this was wrong, because the check isn't a field, it's a path shape, and my manifest validated clean regardless.

I confirmed the real shape by brute force. I already had the official Anthropic plugin marketplace installed locally, so instead of guessing I grepped every `SKILL.md` it ships: twenty-three files across every plugin in it, every single one nested one level under a `skills/` folder inside the plugin, never sitting at the plugin's own root. This is a working example of convention over configuration, the design principle David Heinemeier Hansson named while building Rails, a framework that "attempts to decrease the number of decisions that a developer using the framework is required to make," inferring structure from where files sit instead of a field declaring it ([Wikipedia: Convention over configuration](https://en.wikipedia.org/wiki/Convention_over_configuration)). It's less to write by hand. It's also nothing to grep for when you get it wrong, which is exactly why comparing my repo against real, working examples found the bug faster than reading prose would have.

## Two loaders, two answers

Before moving anything, I checked the CLI too, out of caution: `claude plugin details ghostwriter@claude-skills` reported "Skills (1) ghostwriter" against the exact layout Desktop had just rejected. Same files, two different verdicts. The CLI has an undocumented fallback that treats a root-level `SKILL.md` as an implicit single skill when there's no `skills/` folder at all; Desktop doesn't implement that fallback. If I'd only tested through the CLI, which is what I'd originally done, I'd have shipped a marketplace that worked for me and silently failed for the audience I actually built it for.

VS Code's extension docs describe a comparable split for a much older plugin ecosystem: an extension declares its capabilities in `package.json`'s `contributes` field, which the host reads at load time ([VS Code docs: extension manifest](https://code.visualstudio.com/api/references/extension-manifest)). The manifest is the single source of truth there for the same reason it should have been here: two hosts reading two different heuristics for the same undeclared structure is how you end up debugging a "works on my machine" bug in a distribution system.

## Build: move the tree, not just a file

The fix moves each skill's entire tree, `SKILL.md` plus its scripts and tests, one level deeper as a unit:

```
skills/ghostwriter/                     # plugin root
├── .claude-plugin/plugin.json
├── LICENSE, README.md, CHANGELOG.md    # plugin-level docs stay here
└── skills/
    └── ghostwriter/                    # the path Claude Code actually scans
        ├── SKILL.md
        ├── scripts/, tests/, voice/
```

I considered leaving `SKILL.md` where it was and adding a thin nested stub that just pointed back at it. I didn't, because that stub would carry its own copy of the frontmatter, and two files agreeing on a `name`/`version`/`description` by hand is a standing invitation for them to quietly disagree later. Moving the real files once, and updating the two lint scripts that check them, meant more file churn but zero ongoing duplication:

```python
# tools/lint_plugin.py
nested_skill_dir = os.path.join(skill_dir, "skills", dir_base)
skill_md_path = os.path.join(nested_skill_dir, "SKILL.md")
package_json_path = os.path.join(nested_skill_dir, "package.json")
```

That move surfaced a bug my first pass didn't catch. ghostwriter has a Tier-1 contract test, `test_version_matches_changelog`, that read `CHANGELOG.md` relative to `SKILL.md`'s own directory. `CHANGELOG.md` deliberately stayed at the outer plugin root, next to `README.md`, so a human browsing the repo finds it without digging; the test assumed it lived beside `SKILL.md`, which was true before the move and false after. CI caught this correctly on the pull request, before it merged, because a pull request is exactly the code path this repo's checks are designed to run on every time.

## The bug CI couldn't have caught, because it never runs it

Once that fix merged, I dispatched all four release workflows to cut the actual version tags. All four failed: `cannot open file 'SKILL.md'`.

This repo runs one CI job per skill on every pull request, and a separate release job per skill that only fires on a push to `main` or a manual dispatch, gated behind `needs: ci`. That split is deliberate: PR-time CI has to report on every pull request so a required-check gate never deadlocks, but running four release jobs on every commit would be wasted work. The problem is that "CI passed" and "the release will work" turned out to be two different claims. `_release.yml`, the reusable workflow shared by all four skills, resolves a version by reading `package.json` or `SKILL.md` directly off disk, from a `working-directory` that pointed at the plugin root:

```yaml
defaults:
  run:
    working-directory: skills/${{ inputs.skill }}
steps:
  - name: Resolve version
    run: |
      read_pkg() { node -p "require('./package.json').version"; }
      # ... reads ./SKILL.md if package.json isn't present
```

That directory was correct the day I wrote it and wrong the moment `SKILL.md`/`package.json` moved one level deeper. Nothing in the pull request's CI run ever executed this step, because the release job only fires on a real push to `main`, and by the time it fired for real, the file it expected wasn't there. This is the same failure as the plugin-discovery bug, one layer up: GitHub Actions' own model makes `working-directory` a default every subsequent `run:` step resolves relative paths from, and nothing checks the files you expect are actually there until the shell tries to open one and fails ([GitHub Actions docs: workflow syntax](https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions)). Convention over configuration doesn't stop being true just because the convention lives in a CI file instead of a plugin loader; nobody had written "package.json lives here" as an assertion either time, both workflows just assumed it, and both assumptions moved without anything noticing.

`CHANGELOG.md` had correctly stayed at the plugin root during the earlier move, so the release-notes and tagging steps that read it were already fine. Only the version-resolution and npm-publish steps needed the deeper path, so I overrode `working-directory` on just those two:

```yaml
- name: Resolve version
  id: ver
  working-directory: skills/${{ inputs.skill }}/skills/${{ inputs.skill }}
  run: |
    set -euo pipefail
    read_pkg() { node -p "require('./package.json').version"; }
```

## Verify: prove it against the real files, not the workflow's word

Two checks mattered more than a green checkmark. First, before touching CI again, I ran the exact resolution logic by hand against all four real skills:

```bash
cd skills/resume/skills/resume && node -p "require('./package.json').version"
# -> 1.0.1
cd skills/ghostwriter/skills/ghostwriter && awk '/^version:/{print $2}' SKILL.md
# -> 0.8.1
```

Second, the marketplace itself is registered locally as a directory source, not a fixed git snapshot, so I could re-query it against my working tree instead of trusting the fix on faith:

```bash
claude plugin marketplace update claude-skills
claude plugin update ghostwriter@claude-skills
claude plugin details ghostwriter@claude-skills
# -> Skills (1) ghostwriter
```

Then the actual release run, which is the only proof that matters for a release pipeline:

```bash
gh workflow run resume.yml --ref main
git tag -l 'resume-v*'
# -> resume-v1.0.1
```

The tag existing is the real proof, not the workflow's own exit code. A CI job can pass because it never touched the failing line, and a release job can look identical to a working one right up until the step that tries to open a file that isn't there. Both bugs here were the same shape: a file moved, and the thing reading it kept a stale, unwritten assumption about where to look. The fix for that isn't a smarter assumption. It's checking against a real, working example instead of trusting the one in your head.
