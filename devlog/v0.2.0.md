## Shipped

Devlog started out assuming one project per repository. Once three skills landed in the same monorepo, the log generator lost the ability to tell them apart: it gathered every commit in the repo for every project, with no way to know which commits belonged to which skill. v0.2.0 adds an optional `pathFilter` field to a project's config: a repo-relative subdirectory, like `skills/devlog`. Several project entries can now share the same local checkout and the same remote, each scoped to its own subtree, and the field is validated the same way every other config value that reaches a shell command is validated.

That's a small config field. Getting it right required two separate pieces of understanding: how git answers "which commits touched this directory," and why a config value headed for a command line isn't safe just because it came from your own machine.

## Setup: one repo, several projects

The starting shape is a config file with an array of projects. Each one already has a `path` (the local checkout) and a `remote` (where commit links point). The new field is `pathFilter`, and it's optional: leave it off and a project behaves exactly as before, scanning the whole repo.

```js
// devlog.config.js
module.exports = {
  projects: [
    {
      key: 'api',
      path: '/Users/me/src/platform',
      remote: 'me/platform',
      pathFilter: 'services/api', // scope this project's commits to this subtree
    },
    {
      key: 'web',
      path: '/Users/me/src/platform', // same checkout
      remote: 'me/platform', // same remote
      pathFilter: 'apps/web',
    },
    {
      key: 'infra',
      path: '/Users/me/src/infra-repo',
      remote: 'me/infra-repo',
      // no pathFilter: this one is the whole repo
    },
  ],
};
```

`api` and `web` point at the same checkout and the same remote. What makes them two distinct projects is the `pathFilter`, nothing else.

## Build it: scope git log with a pathspec

Git already has the primitive for this. `git log -- <path>` limits history to commits that touched something under that path; the docs describe it as showing "only commits that are enough to explain how the files that match the specified paths came to be" ([git-log documentation](https://git-scm.com/docs/git-log)). You don't need submodules, repo splitting, or rewritten history; a per-project commit feed is just a `git log` call with a pathspec appended.

```js
// buildLogArgs.js
function buildLogArgs({ author, since = 'midnight', pathFilter } = {}) {
  const args = ['log', `--author=${author}`, `--since=${since}`, '--format=%H%x1f%s'];
  if (pathFilter) {
    args.push('--', pathFilter); // everything after `--` is a path, never an option
  }
  return args;
}

module.exports = { buildLogArgs };
```

The `--` matters as much as the path does. The git-log docs say paths "may need to be prefixed with `--` to separate them from options or the revision range, when confusion arises" ([git-log documentation](https://git-scm.com/docs/git-log)). Without it, git parses arguments left to right and anything that looks like a flag gets treated as one, no matter what you meant it to be.

## Build it: validate the filter before git ever sees it

Here's the trap. `pathFilter` comes from a config file someone edits by hand, or from a CLI flag, and it ends up as an argument to `git`. "It's my own config, it's fine" is exactly the assumption that lets this go wrong. A value that happens to start with a dash stops being a path once it reaches git and becomes an option instead. This is a real, documented vulnerability class in version-control tooling: passing a crafted `--upload-pack=<command>`-style value where a repository name was expected has been used to get arbitrary commands executed, because the parser reads a leading dash as the start of an option rather than as the first character of data ([Snyk: argument injection when using git and mercurial](https://snyk.io/blog/argument-injection-when-using-git-and-mercurial/)). `pathFilter` isn't remote-user input in devlog's case, but the shape of the bug doesn't care where the string came from; a typo can trigger it just as easily as an attacker can.

The fix is to validate what the value is allowed to be, not to hunt for characters that might be dangerous. OWASP's input validation guidance is explicit about this: allowlisting "involves defining exactly what IS authorized," and a denylist approach is weaker because "it is trivial for an attacker to bypass such filters" ([OWASP: Input Validation Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Input_Validation_Cheat_Sheet.html)). For a pathspec, the allowlist is narrow: safe characters only, and it must start with a letter or digit.

```js
// validatePathFilter.js
// Anchoring the pattern to a leading letter/digit does double duty: it rules
// out a leading dash (misread as an option) AND a leading slash (an absolute
// path escaping the repo) in one rule, instead of two separate checks.
const RE_SAFE_SUBTREE = /^[a-z0-9][a-z0-9._/-]*$/i;
// '..' as its own path segment (not inside a real name like '..config'),
// which the anchor above doesn't catch on its own.
const HAS_DOTDOT_SEGMENT = /(^|\/)\.\.($|\/)/;

function validatePathFilter(value) {
  if (value == null) return null; // no filter: gather the whole repo
  if (typeof value !== 'string' || !RE_SAFE_SUBTREE.test(value)) {
    throw new Error(`unsafe pathFilter: ${JSON.stringify(value)}`);
  }
  if (HAS_DOTDOT_SEGMENT.test(value)) {
    throw new Error(`unsafe pathFilter (parent traversal): ${JSON.stringify(value)}`);
  }
  return value;
}

module.exports = { validatePathFilter };
```

## Use it, then verify it

Wire the two pieces together. Validate first, so a bad value never becomes a git argument at all, then invoke git with the argument list as an array rather than a single shell string. Node's `child_process` docs describe `execFile` as spawning the executable directly, "it does not spawn a shell by default," which is what keeps the array from being reinterpreted by anything in between ([Node.js child_process](https://nodejs.org/api/child_process.html)).

```js
// commitsFor.js
const { execFileSync } = require('node:child_process');
const { validatePathFilter } = require('./validatePathFilter');
const { buildLogArgs } = require('./buildLogArgs');

function commitsFor(project) {
  const pathFilter = validatePathFilter(project.pathFilter);
  const args = ['-C', project.path, ...buildLogArgs({ author: project.author, pathFilter })];
  const out = execFileSync('git', args, { encoding: 'utf8' }).trim();
  if (!out) return [];
  return out.split('\n').map((line) => {
    const [sha, subject] = line.split('\x1f');
    return { sha, subject, url: `https://github.com/${project.remote}/commit/${sha}` };
  });
}

module.exports = { commitsFor };
```

To see why the `--` separator earns its place, run this against any git repo you have locally, once with a dash-leading value passed as a bare argument and once with the same value passed after `--`:

```bash
# a value that happens to look like a flag, misread as one without the separator
git log --format=%H --all -1
# → one line of output: `-1` is read as "limit to 1 commit," not as a path

# the identical string, passed after `--`: now it can only ever be a path
git log --format=%H --all -- -1
# → no output: there's no file or directory literally named "-1"
```

The first command doesn't error. It runs, and it silently returns the wrong thing: history limited to one commit instead of history scoped to a path. That's the whole danger in one line: a misparsed argument doesn't usually crash, it quietly changes what the command does. The second command is what `commitsFor` actually sends, and it fails safe instead: an unmatched path returns nothing rather than reinterpreting itself as an option.

## Gotchas

**Denylisting "bad characters" instead of anchoring an allowlist.** It's tempting to check a value against a list of characters you've decided are dangerous (`;`, `|`, backticks) and call it validated. That list is never complete, and it does nothing for a value like `-1` or `--upload-pack=…`, which contain no dangerous punctuation at all, only a leading dash in a position git treats specially. Anchor the pattern to what a legitimate path looks like and reject everything else, per OWASP's allowlist guidance above.

**Validating the value but still building the command as one shell string.** Validation and the `--` terminator close two different doors, and args-as-array closes a third. If you pass a "validated" string into `exec()` or a template-literal shell command instead of `execFile()` with an argument array, a shell somewhere still gets a chance to reinterpret it, and your validation regex was never designed to be a full shell-escaping function. Keep all three: allowlist the value, terminate options with `--`, and hand it to `execFile`/`execFileSync` as a separate array element, never concatenated into a command string.

**Expecting a pathspec to partition history instead of filter it.** A commit that touches two projects' subtrees will legitimately show up in both projects' feeds. That's not a bug to deduplicate away; `git log -- <path>` answers "did this commit touch this path," and a commit that touched two paths correctly gets two "yes" answers. Trying to force each commit into exactly one project would mean lying about what the commit did.
