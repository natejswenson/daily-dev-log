## Shipped

I folded three separate skills into a single repo, and the dev-log generator immediately got confused: it gathered every commit in the repo and had no way to tell which skill a given commit belonged to. v0.2.0 adds an optional `pathFilter` config field that scopes a project's commit gathering to a repo-relative subdirectory, like `skills/devlog`. Several logical projects can now share the same `path` and `remote` while each collects only its own subtree's commits; under the hood, the `git log` call grows a `-- <pathFilter>` pathspec. Commit links still resolve to the shared repo, so nothing about navigation changes. The value is validated like every other path that reaches git (no leading dash or slash, no `..`), surfaced as a `scope:` line in `devlog config`, and documented in SKILL.md with a security-validation row.

The feature is small. The two ideas underneath it are not: scoping history to a subdirectory is a view, not a partition, and a path you hand to git is input you have to distrust. Here is the end-to-end build, from the config field to the test that keeps the guard honest.

## Setup: several projects, one repo

The config used to assume one project per repo. The change is one optional field. A project keeps its `path` (the local checkout) and `remote` (where commit links point), and may add a `pathFilter` naming the subtree it owns. Two projects pointing at the same `path` and `remote` with different `pathFilter` values are now two distinct feeds over one repository.

```js
// devlog.config.js
// Several logical projects can share one checkout. `pathFilter` scopes a
// project's commit feed to a subdirectory of that repo. Omit it for the
// original behavior: the whole repository.
module.exports = {
  projects: [
    {
      key: "devlog",
      path: "/Users/me/src/claude-skills",
      remote: "https://github.com/natejswenson/claude-skills",
      pathFilter: "skills/devlog",        // scope to this subtree
    },
    {
      key: "ghostwriter",
      path: "/Users/me/src/claude-skills",     // same checkout
      remote: "https://github.com/natejswenson/claude-skills",
      pathFilter: "skills/ghostwriter",
    },
    {
      key: "site",
      path: "/Users/me/src/natejswenson.io",
      remote: "https://github.com/natejswenson/natejswenson.io",
      // no pathFilter: whole repo, the original one-project-per-repo case
    },
  ],
};
```

## A pathspec turns one repo into several logical projects

Putting multiple projects in one repository is the monorepo pattern, and it's how some of the largest codebases in the world are organized; Google, Meta, and Microsoft all run very large monorepos, with Microsoft moving Windows to a Git monorepo ([Monorepo, Wikipedia](https://en.wikipedia.org/wiki/Monorepo)). The appeal is shared tooling and atomic cross-project changes, and it pairs naturally with trunk-based development, where everyone integrates into one trunk that stays releasable ([Trunk Based Development: Monorepos](https://trunkbaseddevelopment.com/monorepos/)). The cost is that "the project" is no longer "the repo." A tool that assumed one project per repo, like mine did, now over-collects.

Git already has the primitive for slicing history by location: the pathspec. `git log -- <path>` shows only the commits that touched files under that path, by filtering history down to the commits needed to explain how those files came to be ([git-log docs](https://git-scm.com/docs/git-log)). So a per-project commit feed is just a `git log` with a pathspec for that project's subtree. There is no restructuring, no submodules, and no rewriting history; one project sets `pathFilter: skills/devlog`, another sets `skills/ghostwriter`, and each feed collects its own subtree from the same shared repo.

## Build: validate the path before git ever sees it

Here's the trap. That `pathFilter` value comes from config, gets handed to a command, and reaches git. The intuition is "it's my own config, it's fine." That intuition is exactly how argument injection happens. Even setting aside classic shell metacharacters, an argument that starts with a dash is the dangerous case: git reads it as an option, not a path. A value passed where a path was expected stops being data and becomes a flag, and this exact class of bug has produced real CVEs in git-adjacent tooling, including argument injection through dash-leading values in VCS wrappers ([Snyk: argument injection in git and mercurial](https://snyk.io/blog/argument-injection-when-using-git-and-mercurial/)).

The first defense is to validate against an allowlist of what the value is allowed to be, rather than trying to block known-bad patterns, which is the approach OWASP recommends because denylists are easy to evade ([OWASP: Input Validation Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Input_Validation_Cheat_Sheet.html)). For a pathspec that means: allow only safe characters, reject a leading dash, reject a leading slash, and reject `..` traversal.

```js
// validatePathFilter.js
// Allowlist what a pathspec is allowed to be. Don't denylist known-bad
// patterns; allowlists are the OWASP-recommended default because denylists
// are easy to evade.
const SAFE_PATHSPEC = /^[A-Za-z0-9._/-]+$/;

function validatePathFilter(value) {
  if (value == null) return null;            // no filter: gather the whole repo
  if (typeof value !== "string" || !SAFE_PATHSPEC.test(value)) {
    throw new Error(`unsafe pathFilter (illegal characters): ${value}`);
  }
  if (value.startsWith("-")) {               // git would read this as an OPTION
    throw new Error(`unsafe pathFilter (leading dash): ${value}`);
  }
  if (value.startsWith("/")) {               // absolute path escapes the subtree
    throw new Error(`unsafe pathFilter (leading slash): ${value}`);
  }
  if (value.includes("..")) {                // parent traversal
    throw new Error(`unsafe pathFilter (traversal): ${value}`);
  }
  return value;
}

module.exports = { validatePathFilter };
```

The second defense is git's `--` separator, the POSIX convention that marks the end of options, so everything after it is treated as a path even if it looks like a flag ([git-log docs](https://git-scm.com/docs/git-log)). It's worth seeing why this matters at the command line. Without the terminator, a dash-leading value is parsed as an option and silently changes what git does:

```bash
# Suppose a project's pathFilter were the string "--all".
# Passed where a path was expected, git reads it as an OPTION and ignores
# your scoping intent, listing every ref's history instead of one subtree:
git log --format='%H %s' --all

# The `--` terminator ends option parsing. Everything after it is a path,
# even if it starts with a dash, so the value can only ever be data:
git log --format='%H %s' -- "skills/devlog"
```

Either defense alone is a meaningful improvement; together they close both the "looks like a flag" and the "contains something nasty" doors.

## Use it: gather one project's commits

Now wire the two defenses into the gather step. Validation runs first, so a bad value never reaches git at all. Then git is invoked with its arguments as an array, which means there is no shell to reinterpret them ([Node.js child_process](https://nodejs.org/api/child_process.html)), and the validated pathspec goes after `--`. The commit URL is built from `project.remote`, so links resolve to the shared repo no matter how the feed is scoped.

```js
// gather.js
const { execFileSync } = require("node:child_process");
const { validatePathFilter } = require("./validatePathFilter");

// Gather one project's commits. Args-as-array means no shell parses them,
// and `--` guarantees the pathspec is read as a path. Validation runs first
// so an unsafe value is refused before git is ever spawned.
function gatherCommits(project) {
  const pathFilter = validatePathFilter(project.pathFilter);

  const args = ["-C", project.path, "log", "--format=%H%x00%s"];
  if (pathFilter) args.push("--", pathFilter);   // terminator, then the path

  // maxBuffer defaults to 1 MB; a real monorepo's full git log can exceed
  // that and throw ENOBUFS, so raise it.
  const out = execFileSync("git", args, { encoding: "utf8", maxBuffer: 1024 * 1024 * 64 });

  return out
    .split("\n")
    .filter(Boolean)
    .map((line) => {
      const [sha, subject] = line.split("\x00");
      // Link points at the shared repo; scoping changes which commits we
      // collect, never where they live.
      return { sha, subject, url: `${project.remote}/commit/${sha}` };
    });
}

module.exports = { gatherCommits };
```

Run `gatherCommits` for the `devlog` project and you get only the commits that touched `skills/devlog`, each linking back to the shared `claude-skills` repo. Run it for `ghostwriter` against the same checkout and you get a disjoint-by-intent feed from the same history. The `site` project, with no `pathFilter`, behaves exactly as it did before.

## Verify it: prove the guard holds

The two failure modes worth defending are testable: a malicious `pathFilter` must be refused, and a normal one must pass through and scope. These became regression tests so the validation can't quietly rot.

```js
// validatePathFilter.test.js
const test = require("node:test");
const assert = require("node:assert/strict");
const { validatePathFilter } = require("./validatePathFilter");

test("refuses an option-injecting pathFilter", () => {
  assert.throws(() => validatePathFilter("--all"), /leading dash/);
});

test("refuses parent traversal", () => {
  assert.throws(() => validatePathFilter("../../etc"), /traversal/);
});

test("refuses an absolute path", () => {
  assert.throws(() => validatePathFilter("/etc/passwd"), /leading slash/);
});

test("accepts a normal subtree and returns it unchanged", () => {
  assert.equal(validatePathFilter("skills/devlog"), "skills/devlog");
});

test("treats a missing filter as whole-repo", () => {
  assert.equal(validatePathFilter(null), null);
});
```

The first three tests fail the moment their guard is removed; `--all` and `../../etc` are exactly the shapes that turn a path into a flag or a traversal. The last two prove the common cases still work: a real subtree passes through verbatim, and an absent filter falls back to the original whole-repo behavior.

## Scoping is a view, not a partition

The honest consequence of building this on `git log -- <path>` is that the filter is a lens over one shared history, not a wall between separate histories. A single commit that spans two skills will show up in both feeds. I could have deduplicated, or forced each commit to belong to exactly one project, but that would be lying about what the commit did; it really did touch both subtrees. The pathspec answers "which commits touched this path," and a commit that touched two paths is correctly two answers. Letting it appear twice keeps the feed faithful to the repository instead of faithful to a tidy mental model. It's the same reason `git log -- src/ docs/` happily lists a commit that changed both.
