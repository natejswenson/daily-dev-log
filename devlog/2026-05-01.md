## What I Built

The /devlog skill has been quietly running in production on natejswenson.io for weeks now — git commits go in, narrative dev log entries come out — and a few people on LinkedIn pinged me asking if they could use it. The skill itself was hardcoded to me: "Nate" as the author filter, my project list inline in SKILL.md, my dev-log repo as the push target. Sharing it would have meant explaining what to substitute and where, then watching adopters merge-conflict on every upstream update. So today I rebuilt the whole thing as a proper shareable package.

Started with /design to plan the shape: config-driven SKILL.md instead of inline registry, a single `config.json` at `~/.claude/skills/devlog/` so user data lives outside the skill, and an npx CLI to make first-time setup a one-command affair. Wrote the design doc, then implemented in one shot — the genericized skill reads config and substitutes user values; the CLI's `init` subcommand prompts for a few defaults (auto-filled from `gh` and `git config`), creates the user's daily-dev-log repo via `gh repo create`, installs the skill, and writes the config; `preview` reads that config and spins up a local Vite dev server pointed at the user's published log so they can see exactly what their site will render before integrating anything.

The lifted React example — the same `DevLogPage` and `useDevLogEntries` hook running on natejswenson.io — got self-contained CSS variables for theming, lost its router and site-chrome assumptions, and gained an optional config-override prop so the preview app can inject runtime values. The preview directory itself doubles as a deployable Vite app: anyone without a personal site yet can `vercel deploy` the preview folder and have a public dev log immediately. For the bare demo experience (running preview with no env vars), I added a tiny fetch override that serves canned manifest+entry data from a fake URL — so the demo renders through the *real* component with fake entries that loudly nag the user to wipe the placeholders and point at their actual repo.

Publishing was a two-stage gauntlet. First publish failed because npm auto-stripped the `bin` field — turns out modern npm rejects the `./` prefix on bin paths, silently corrects it, then refuses to publish without an explicit fix. Second publish failed with a 404 instead of the expected 403, which was the clue: my npm username is `natjswenson` (no 'e' — registration typo from years ago), so publishing under `@natejswenson` returned "scope not found." Renamed the package across every reference, retried, shipped. Final form: `npx @natjswenson/devlog init` is now a working command on any machine with Node 18+, `gh` authenticated, and Claude Code installed.

## What's Next

Linking it on the LinkedIn thread where the request originated and watching whether anyone files an issue. Likely first feedback areas are non-React framework examples (Astro and Next are the obvious follow-ups) and Windows path handling in the CLI's `init` flow.
