## Shipped

The `/devlog` skill has been running on natejswenson.io for weeks: it reads git commits and writes narrative dev-log entries. After a few people asked about it on LinkedIn, I genericized it, wrapped it in an npx CLI, and published it to npm so anyone can install it with one command. `npx @natjswenson/devlog init` now auto-fills a few defaults from `gh` and `git config`, creates the user's daily-dev-log repo with `gh repo create`, installs the skill, and writes a config file. A `preview` command spins up a local Vite app pointed at the user's published log, and the bundled demo renders the real component against canned data when there's nothing real to show yet.

The skill itself was the easy part. It already worked. The part worth writing about is what it takes to turn a tool that's hardcoded to one person into something a stranger can adopt without forking it: pulling the user's identity out into config, declaring the executable so `npx` can find it, scaffolding setup with an `init` command, and then verifying the whole thing resolves before you trust a publish. Here is the end-to-end build.

## Setup: pull the user out of the code

The original skill was hardcoded to me in three places: "Nate" as the commit-author filter, my project list written inline in `SKILL.md`, and my dev-log repo as the push target. Sharing it as-is would have meant writing a paragraph of "replace this, then this, then this," and then watching every adopter hit a merge conflict the next time I updated the upstream skill, because their edits and my edits would land in the same file.

So I moved all of that out of the code and into a single `config.json` at `~/.claude/skills/devlog/`, outside the skill directory entirely. This is the third of the twelve factors, store config separate from code; the methodology's own litmus test is whether you could open-source the codebase at any moment without leaking anything deploy-specific ([The Twelve-Factor App: Config](https://12factor.net/config)). Start with the shape of the config the skill reads.

```json
{
  "author": "Nate Swenson",
  "githubUser": "natejswenson",
  "logRepo": "natejswenson/daily-dev-log",
  "projects": ["natejswenson.io", "local-fitness", "ghostwriter"]
}
```

The skill becomes config-driven: instead of baking in assumptions, it reads these values, and falls back to a clear error when the file is missing rather than silently using my identity.

```js
// skill/config.js
import { readFile } from "node:fs/promises";
import { homedir } from "node:os";
import { join } from "node:path";

const CONFIG_PATH = join(homedir(), ".claude", "skills", "devlog", "config.json");

export async function loadConfig() {
  try {
    return JSON.parse(await readFile(CONFIG_PATH, "utf8"));
  } catch (err) {
    if (err.code === "ENOENT") {
      throw new Error(`No config at ${CONFIG_PATH}. Run \`npx @natjswenson/devlog init\` first.`);
    }
    throw err;
  }
}
```

Once the user's identity lives in a file the skill merely reads, the open-source test passes, and the practical payoff shows up too: adopters never touch the shipped files, so they can pull updates cleanly forever.

## Build: declare the executable with bin and a shebang

`npx` is what makes "install with one command" true. It runs a package's executable without a permanent global install, fetching it on demand if it isn't already present ([npm Docs: npx](https://docs.npmjs.com/cli/v11/commands/npx/)). For that to work, the package has to declare its executable in the `bin` field of `package.json`, which maps a command name to the file npm puts on the user's PATH ([npm Docs: package.json bin](https://docs.npmjs.com/cli/v11/configuring-npm/package-json/)).

My first publish surfaced a warning here. I'd written the bin path with a leading `./`, the way you'd reference a relative file most other places. npm warns and auto-cleans the leading `./` (suggesting `npm pkg fix`); the publish still succeeds and the command still works, but writing the path bare avoids the warning.

```json
{
  "name": "@natjswenson/devlog",
  "version": "1.0.0",
  "type": "module",
  "bin": {
    "devlog": "bin/cli.js"
  },
  "files": ["bin", "skill"]
}
```

The file `bin` points at must start with a `#!/usr/bin/env node` shebang, or npm will try to run it without the Node runtime. That line tells the OS to execute the script with whatever `node` is on PATH, which is the portable form across machines.

```js
#!/usr/bin/env node
// bin/cli.js: the shebang above is mandatory, or this runs without Node.
const command = process.argv[2];

if (command === "init") {
  const { runInit } = await import("../skill/init.js");
  await runInit();
} else if (command === "preview") {
  const { runPreview } = await import("../skill/preview.js");
  await runPreview();
} else {
  console.log("usage: devlog <init|preview>");
  process.exit(command ? 1 : 0);
}
```

Each branch loads its handler lazily, so `init` never imports the preview machinery and vice versa. The `preview` handler is small on purpose; it just boots the bundled Vite app against the user's published log. That `preview-app` is an ordinary Vite project shipped inside the package, so I'm leaving its scaffolding out of scope here.

```js
// skill/preview.js
import { spawn } from "node:child_process";
import { dirname, join } from "node:path";
import { fileURLToPath } from "node:url";

export async function runPreview() {
  // Boot the bundled Vite preview app; it reads the same config.json init wrote.
  const appDir = join(dirname(fileURLToPath(import.meta.url)), "preview-app");
  spawn("npx", ["vite", appDir], { stdio: "inherit" });
}
```

With the bin declared with a bare path, `npx @natjswenson/devlog init` resolves the `devlog` command and runs the CLI, no global install required.

## Build: an init command with sensible defaults

The point of `init` is that an adopter shouldn't have to know what goes in `config.json`. The CLI should fill in everything it can read from the tools they already have. `git config user.name` gives the author, and the GitHub CLI's `gh api user` gives the account login, so the only thing left to ask is which repo to push logs to, and even that gets a default.

```js
// skill/init.js
import { execFileSync } from "node:child_process";
import { writeFile, mkdir, cp } from "node:fs/promises";
import { homedir } from "node:os";
import { join, dirname } from "node:path";
import { fileURLToPath } from "node:url";

const SKILL_DIR = join(homedir(), ".claude", "skills", "devlog");
const CONFIG_PATH = join(SKILL_DIR, "config.json");

function tryCmd(file, args) {
  try {
    return execFileSync(file, args, { encoding: "utf8" }).trim();
  } catch {
    return ""; // tool missing or not logged in; fall back to a default
  }
}

export async function runInit() {
  const author = tryCmd("git", ["config", "user.name"]) || "Your Name";
  const githubUser = tryCmd("gh", ["api", "user", "--jq", ".login"]) || "your-handle";
  const logRepo = `${githubUser}/daily-dev-log`;

  const config = { author, githubUser, logRepo, projects: [] };

  await mkdir(SKILL_DIR, { recursive: true });
  await writeFile(CONFIG_PATH, JSON.stringify(config, null, 2) + "\n");
  console.log(`Wrote ${CONFIG_PATH}`);

  // Create the log repo (no-op if it already exists or gh is missing).
  tryCmd("gh", ["repo", "create", logRepo, "--public", "--clone=false"]);
  console.log(`Created repo ${logRepo}`);

  // Copy the bundled skill next to the config so the agent can load it.
  // Exclude config.json so a shipped one can't clobber what we just wrote.
  const skillSrc = dirname(fileURLToPath(import.meta.url));
  await cp(skillSrc, SKILL_DIR, {
    recursive: true,
    filter: (src) => !src.endsWith("config.json"),
  });
  console.log(`Installed skill to ${SKILL_DIR}`);
}
```

Defaults that come from the environment are the difference between a setup an adopter completes and one they abandon. The values are pre-filled and correct for the common case, and they only have to override the one that's wrong.

## Use it

For a new adopter, the whole setup is one command. `npx` fetches the package, finds the `devlog` bin, and runs the `init` branch.

```bash
$ npx @natjswenson/devlog init
Wrote /Users/you/.claude/skills/devlog/config.json
Created repo your-handle/daily-dev-log
Installed skill to /Users/you/.claude/skills/devlog

# later, see it rendered locally against your published log:
$ npx @natjswenson/devlog preview
```

No clone, no fork, no editing shipped files. The skill reads the config that `init` just wrote, and updates to the package land cleanly because the user's data was never in the package to begin with.

## Verify: confirm the bin resolves before you publish

Before publishing, you want proof that the executable is actually in the tarball and that it resolves to a runnable command. `npm pack` builds the exact tarball npm would publish and prints its contents, so you can confirm `bin/cli.js` is included; `npm link` then puts the bin on your PATH locally so you can run it by name.

```bash
# 1. See exactly what would publish; bin/cli.js must be listed.
$ npm pack --dry-run

# 2. Symlink the package globally and confirm the command resolves.
$ npm link
$ which devlog          # -> .../bin/devlog
$ devlog                # -> usage: devlog <init|preview>
```

The second publish failed even after the bin was correct, this time with a 404, and the instinct with a publish failure is to reach for a 403, a permissions problem. A 404 is different: it means the thing you addressed doesn't exist. The clue was the scope. I tried to publish under `@natejswenson`, but my actual npm username is `natjswenson`, missing the "e", a registration typo I made years ago and never noticed. npm scopes are tied to a username or org exactly, so `@natejswenson` returned "scope not found," which the registry reports as a 404 rather than a 403 ([npm Docs: scope](https://docs.npmjs.com/cli/v11/using-npm/scope)). HTTP status codes carry that distinction on purpose: 403 is "the request was understood but refused," 404 is "I have no representation for this resource" ([MDN: HTTP response status codes](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status)).

```bash
# Publishing under a scope you don't own returns 404, not 403.
$ npm publish --access public
npm error code E404
npm error 404 Not Found - PUT https://registry.npmjs.org/@natejswenson%2fdevlog
# the 404 body is the registry's way of saying the scope @natejswenson doesn't exist

# Fix: rename to the scope you actually own, then it ships.
$ npm whoami            # natjswenson  (no "e")
$ npm publish --access public
```

The lesson that generalizes: when a tool hands you a status code, trust what that specific code means instead of the failure you were braced for.
