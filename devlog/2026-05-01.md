## Shipped

The `/devlog` skill had been running on one machine for weeks, reading git commits and writing narrative dev-log entries to a hardcoded repo. This release pulled the hardcoded identity out into a config file, wrapped the skill in an npx CLI with `init` and `preview` subcommands, and published the result to npm so someone else could adopt it without forking anything. `init` runs an interactive setup that creates the user's dev-log repo, installs the skill, and writes their config; `preview` boots a local Vite app against whatever they've published so far, falling back to canned demo data before they've published anything real.

The interesting part isn't the skill logic, which already worked. It's what it takes to turn a script you run yourself into a tool a stranger installs with one command: a package that declares a real executable, a setup command that asks smart questions instead of demanding a hand-edited file, and a publish you've actually verified before you trust it. Here's the build, plus the two ways the first publish attempt didn't go as planned.

## Setup: declare a real executable

Everything starts with the `bin` field in `package.json`. It maps a command name to a file, and npm uses that map to put the command on the user's `PATH` when the package is installed, or, for our purposes, when it's run through `npx` ([npm Docs: package.json `bin`](https://docs.npmjs.com/cli/v11/configuring-npm/package-json)). `npx` is what makes "install with one command" true: it checks whether the command already exists locally, and if not, fetches the package into npm's cache just for that run instead of installing it permanently ([npm Docs: `npx`](https://docs.npmjs.com/cli/v11/commands/npx)).

```json
{
  "name": "@yourscope/your-cli",
  "version": "0.1.0",
  "type": "module",
  "bin": {
    "your-cli": "bin/cli.js"
  },
  "files": ["bin"]
}
```

The file the `bin` entry points at needs a shebang as its first line, or npm runs it without the Node runtime at all: "make sure that your file(s) referenced in bin starts with `#!/usr/bin/env node`; otherwise, the scripts are started without the node executable" ([npm Docs: package.json `bin`](https://docs.npmjs.com/cli/v11/configuring-npm/package-json)).

```js
#!/usr/bin/env node
// bin/cli.js
const arg = process.argv[2];

switch (arg) {
  case "init":
    await runInit();
    break;
  case "preview":
    // Same shape as init: read config.json, then boot your preview app
    // pointed at it. Left out here since it's app-specific, not CLI plumbing.
    await runPreview();
    break;
  default:
    console.log("usage: your-cli <init|preview>");
    process.exit(arg ? 1 : 0);
}
```

With the bin declared and the shebang in place, `npx @yourscope/your-cli init` resolves the command and hands off to whatever `runInit` does next.

## Build it: prompt instead of demanding a config edit

The whole point of an `init` command is that an adopter shouldn't have to know the shape of your config file. Reading the values you can from tools that are probably already installed, git and the GitHub CLI in this case, and using them as defaults the user can accept or override, is the difference between a setup someone finishes and one they close the terminal on halfway through. The `prompts` package handles the interactive part: each prompt takes an `initial` value and an optional `validate` function that runs before the answer is accepted ([`prompts` on GitHub](https://github.com/terkelg/prompts)).

```js
// runInit.js
import prompts from "prompts";
import { execSync } from "node:child_process";

function tryExec(cmd) {
  try {
    return execSync(cmd, { encoding: "utf8" }).trim();
  } catch {
    return null; // tool missing, not authenticated, or the lookup failed
  }
}

async function preflight() {
  const nodeMajor = parseInt(process.versions.node.split(".")[0], 10);
  if (nodeMajor < 18) {
    console.error(`Node 18+ required (you have ${process.versions.node}).`);
    process.exit(1);
  }
  if (!tryExec("gh --version")) {
    console.error("GitHub CLI (`gh`) is not installed. See https://cli.github.com/");
    process.exit(1);
  }
  if (!tryExec("gh auth status")) {
    console.error("GitHub CLI is not authenticated. Run: gh auth login");
    process.exit(1);
  }
}

export async function runInit() {
  await preflight();

  const answers = await prompts([
    {
      type: "text",
      name: "author",
      message: "Your name:",
      initial: tryExec("git config --global user.name") || "",
      validate: (v) => v.trim().length > 0 || "Required",
    },
    {
      type: "text",
      name: "githubUser",
      message: "Your GitHub username:",
      initial: tryExec("gh api user --jq .login") || "",
      validate: (v) => /^[a-z0-9-]+$/i.test(v.trim()) || "Invalid username",
    },
  ], { onCancel: () => process.exit(1) });

  // finishInit does the actual repo-creation and file-writing work; see next section.
  return finishInit(answers);
}
```

Preflight checks fail fast with a message that tells the user exactly what to fix, rather than letting a missing tool surface as a cryptic error three steps later. And every prompt's `initial` comes from a real value on the machine when one exists, so the common case is "press enter three times," not "go read the README to find out what goes in `githubUser`."

## Build it: let init do the annoying part, safely

The rest of `init` does the setup work an adopter would otherwise do by hand: create the target repo, install the tool's own config file next to itself, and write the answers to disk. The part worth being careful about is that `init` is meant to be re-run. If someone runs it twice, silently overwriting a config file they'd already customized would be a worse experience than the manual setup it's replacing.

```js
// runInit.js (continued)
import { existsSync, writeFileSync, mkdirSync } from "node:fs";
import { join } from "node:path";
import { homedir } from "node:os";

const CONFIG_DIR = join(homedir(), ".config", "your-cli");
const CONFIG_PATH = join(CONFIG_DIR, "config.json");

async function confirmOverwrite(label, path) {
  if (!existsSync(path)) return true;
  const { ok } = await prompts({
    type: "confirm",
    name: "ok",
    message: `${label} already exists at ${path}. Overwrite?`,
    initial: false,
  });
  return ok === true;
}

export async function finishInit(answers) {
  const targetRepo = `${answers.githubUser}/daily-log`;

  const repoExists = tryExec(`gh repo view ${targetRepo} --json name`) !== null;
  if (!repoExists) {
    execSync(`gh repo create ${targetRepo} --public --add-readme`, { stdio: "inherit" });
  }

  if (!existsSync(CONFIG_DIR)) mkdirSync(CONFIG_DIR, { recursive: true });

  if (await confirmOverwrite("config.json", CONFIG_PATH)) {
    writeFileSync(CONFIG_PATH, JSON.stringify({ ...answers, targetRepo }, null, 2) + "\n");
    console.log(`Wrote ${CONFIG_PATH}`);
  }
}
```

Checking whether the repo already exists before creating it, and asking before clobbering an existing config, both cost a few extra lines. What they buy is a setup command someone can run again without fear, which matters more than it sounds like the first time an adopter fat-fingers an answer and needs a redo.

## Use it, then verify it before you publish

For an adopter, the whole setup is one command:

```bash
$ npx @yourscope/your-cli init
✓ Node 18+, gh installed and authenticated
? Your name: Jane Doe
? Your GitHub username: janedoe
✓ Created github.com/janedoe/daily-log
✓ Wrote /Users/jane/.config/your-cli/config.json
```

Before you trust a publish, though, verify the package resolves the way you think it does. `npm pack --dry-run` builds the exact tarball `npm publish` would produce and prints its file list without writing anything, so you can confirm the bin file is actually in it ([npm Docs: `npm pack`](https://docs.npmjs.com/cli/v11/commands/npm-pack)). `npm link` then symlinks the package into your global `node_modules` and puts its bin on your `PATH`, so you can run the command by name exactly as an adopter would ([npm Docs: `npm link`](https://docs.npmjs.com/cli/v11/commands/npm-link)).

```bash
$ npm pack --dry-run    # bin/cli.js must show up in the file list
$ npm link
$ which your-cli        # -> .../bin/your-cli
$ your-cli               # -> usage: your-cli <init|preview>
```

## Gotchas

**A leading `./` on the bin path can get your executable dropped.** The trap: you write the bin path the way you'd reference any other relative file, `"./bin/cli.js"`. The symptom shows up only after publish: the package installs fine, but `npx @yourscope/your-cli` fails to find the command, because npm stripped the bin entry rather than accept the leading `./`. The escape: write bin paths bare, `"bin/cli.js"`, and confirm with `npm pack --dry-run` before you publish, since that's the one step that would have caught it. Current npm docs describe paths relative to the package root without a leading `./`, which is the form to use from the start.

**Your npm scope has to match your username exactly, and a mismatch fails as a 404, not a 403.** A scope is tied to one account; "each npm user/organization has their own scope, and only you can add packages in your scope" ([npm Docs: scope](https://docs.npmjs.com/cli/v11/using-npm/scope)). The trap is assuming your scope is whatever handle you go by everywhere else, when your actual registered npm username is spelled slightly differently. The symptom: `npm publish` fails with `E404`, not a permissions error, because as far as the registry is concerned the scope you addressed doesn't exist; 403 means "understood but refused," 404 means "no such resource," and an unrecognized scope is the second case ([MDN: HTTP response status codes](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status)). The escape: run `npm whoami` before you ever set the package name, and use that exact string as your scope.
