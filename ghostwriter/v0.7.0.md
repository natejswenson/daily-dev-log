## Shipped

Ghostwriter publishes to a real LinkedIn profile, which means a fabricated statistic ships to a real audience. Version 0.7.0 adds a gate: a post that makes any claim about the outside world cannot publish unless its claims trace to at least three real, live sources. Each draft carries a sidecar file pairing claims to source URLs, a verifier checks those URLs are actually reachable, and the publish script refuses to run unless the check passes. It fails closed, so the safe state is "do not publish." The honest framing matters too, so I will be clear about what this proves and what it does not. Here is the whole gate.

## Separate the claims from the prose

The post body never contains links, because a LinkedIn post reads badly with citations in it. So the evidence lives beside the draft, not inside it. Every draft gets a sidecar that maps each external claim to the URLs that back it:

```json
// drafts/my-post.sources.json
{
  "external_claims": true,
  "claims": [
    {
      "text": "HEAD returns headers without a body",
      "sources": ["https://developer.mozilla.org/en-US/docs/Web/HTTP/Methods/HEAD"]
    },
    {
      "text": "fail-closed means deny on error",
      "sources": ["https://owasp.org/www-community/Fail_securely"]
    }
  ]
}
```

A purely first-person post makes no outside-world claim, so it declares `{"external_claims": false}` and passes trivially. That keeps the gate honest: it only fires when there is something to verify.

## Check that a URL is actually live

The core is a liveness check. You want to confirm a URL resolves to something real without downloading whole pages. The [HTTP HEAD method](https://developer.mozilla.org/en-US/docs/Web/HTTP/Methods/HEAD) is built for exactly this. It asks the server for the headers it would send for a GET, with no response body, which is why it is the standard tool for testing links. The wrinkle is that many CDNs bot-wall a HEAD with a 401, 403, or 405 while serving the real page fine, so treat those as "live" and fall back to a ranged GET when HEAD is inconclusive:

```python
# verify_sources.py — is this URL reachable? (stdlib only)
import urllib.request

UA = "Mozilla/5.0 (compatible; source-verifier/1.0)"
LIVE_BUT_WALLED = {401, 403, 405}   # bot-walls; the resource still exists

def _status(url: str, method: str, headers: dict) -> int | None:
    req = urllib.request.Request(url, method=method, headers={"User-Agent": UA, **headers})
    try:
        with urllib.request.urlopen(req, timeout=10) as resp:
            return resp.status
    except urllib.error.HTTPError as e:
        return e.code                 # a 403 is a real answer, not a failure
    except Exception:
        return None                   # DNS failure, timeout, connection refused

def is_live(url: str) -> bool:
    code = _status(url, "HEAD", {})
    if code is not None and (200 <= code < 400 or code in LIVE_BUT_WALLED):
        return True
    # HEAD inconclusive: try one byte of the body
    code = _status(url, "GET", {"Range": "bytes=0-0"})
    return code is not None and (200 <= code < 400 or code in LIVE_BUT_WALLED)
```

## Require three distinct hosts

One live URL is weak evidence, and three URLs from the same domain is one source wearing three hats. So the gate counts distinct hosts, normalized by hostname, and requires at least three:

```python
from urllib.parse import urlparse

MIN_SOURCES = 3

def verify(sidecar: dict) -> tuple[bool, str]:
    if sidecar.get("external_claims") is False:
        return True, "first-person post, no external claims"

    urls = {u for c in sidecar.get("claims", []) for u in c.get("sources", [])}
    live_hosts = {urlparse(u).hostname for u in urls if is_live(u)}
    live_hosts.discard(None)

    if len(live_hosts) < MIN_SOURCES:
        return False, f"only {len(live_hosts)} live host(s); need {MIN_SOURCES}"
    return True, f"{len(live_hosts)} distinct live hosts"
```

## Wire it into publish, failing closed

The gate is only worth anything if it sits in the one path that matters: publishing. It runs after the dry-run and author checks and before any media upload, and the default on any failure is to refuse. This is the [fail securely](https://owasp.org/www-community/Fail_securely) principle: when something goes wrong, default to the safe state, which here is not publishing. A bare text publish with no sidecar is refused by design, and the only way past is a human-only flag:

```python
def publish(draft_path: str, allow_unverified: bool = False) -> None:
    sidecar = load_sidecar(draft_path)          # raises if missing -> refuse
    ok, reason = verify(sidecar)
    if not ok and not allow_unverified:
        raise SystemExit(f"BLOCKED: source gate failed ({reason})")
    upload_media_and_post(draft_path)           # only reached when the gate is green
```

Note the shape: the exception is the default, and publishing is the narrow path you only reach by passing. An empty or malformed sidecar throws on load and lands in the same refusal, which is the [deny-by-default](https://owasp.org/www-community/Fail_securely) posture you want. The `allow_unverified` escape hatch is deliberately human-only, never something the agent can set on its own.

## What it proves, and what it does not

Be honest about the boundary. This gate is liveness proof-of-work: evidence that real sources exist and were reachable when the post shipped. It is not a factuality checker. It cannot tell whether a live page actually supports the claim next to it, and it cannot judge whether a claim was personal or external in the first place. Those stay enforced by the research step that writes the sidecar, not by the code. Encoding the cheap, mechanical half in a fail-closed gate raises the floor, and naming the half it does not cover keeps the tool from pretending to verify more than it does.
