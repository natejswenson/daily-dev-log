## Shipped

The day before this release, I turned this repo into a self-hosted Claude Code plugin marketplace: `marketplace.json` at the root, a `plugin.json` for each of the four skills, `claude plugin validate` green across the board. I installed ghostwriter through it in Claude Desktop to see it work. Its plugin page said: "This plugin doesn't have any skills or agents." ghostwriter 0.8.1 is the fix, moving `SKILL.md` from the plugin's root to `skills/ghostwriter/skills/ghostwriter/SKILL.md`. The deeper lesson is what that screenshot didn't tell me, and what almost let the fix ship broken a second time.

## Diagnosing by example, not by guessing

Nothing in `plugin.json` says where a skill's `SKILL.md` has to live. There's no field for it, no validation error pointing at the right shape, just a scanner somewhere that either finds the file or doesn't. I had one other Claude Code plugin marketplace already installed on this machine, the official Anthropic one, so instead of guessing I checked every real example it shipped: twenty-three `SKILL.md` files across every plugin in it, every single one nested one level under a `skills/` folder inside the plugin, never sitting at the plugin's own root. Anthropic's own docs for building a marketplace confirm it directly: "a plugin's skills load from the `skills/` directory under its source" ([Claude Code docs: create and distribute a plugin marketplace](https://code.claude.com/docs/en/plugin-marketplaces)), and the walkthrough example in that doc builds exactly that shape, `plugins/quality-review-plugin/skills/quality-review/SKILL.md`. My plugin.json passed validation because validation checks that the manifest parses, not that the skill it describes is where the loader will actually look for it.

That's the practical version of "convention over configuration," the term David Heinemeier Hansson coined for Rails: a framework that "attempts to decrease the number of decisions that a developer using the framework is required to make," inferring structure from where files sit instead of a config field declaring it ([Wikipedia: Convention over configuration](https://en.wikipedia.org/wiki/Convention_over_configuration)). It's less to write. It's also nothing to grep for when you get it wrong, which is why comparing my repo against real, working examples found the bug faster than reading documentation prose would have.

## The two loaders disagreed, and only one of them told me

Before moving anything, I checked the CLI too, out of caution: `claude plugin details ghostwriter@claude-skills` reported "Skills (1) ghostwriter" against the exact layout Desktop had just rejected. Same files, two different verdicts. The CLI turned out to have an undocumented fallback that treats a root-level `SKILL.md` as an implicit single skill when there's no `skills/` folder at all; Desktop doesn't have that fallback. If I'd only tested through the CLI, as I originally had, I'd have shipped a marketplace that worked for me and silently failed for anyone using the desktop app, which was the whole audience I built it for.

## Build it: move the file, not just add one

The fix moves ghostwriter's entire tree, `SKILL.md`, `scripts/`, `tests/`, `voice/`, everything, one level deeper as a unit:

```
skills/ghostwriter/                     # plugin root
├── .claude-plugin/plugin.json
├── LICENSE, README.md, CHANGELOG.md
└── skills/
    └── ghostwriter/                    # the path Claude Code actually scans
        ├── SKILL.md
        ├── scripts/, tests/, voice/
```

I considered leaving `SKILL.md` where it was and adding a thin nested stub that just pointed back at it. I didn't, because that stub would need its own copy of the frontmatter, and two files carrying the same `name`/`version`/`description` is a standing invitation for them to quietly disagree with each other later. Moving the real files once, and updating the two lint scripts that check them, was more file churn but zero ongoing duplication.

That move surfaced a second bug my first fix didn't catch. This repo has a Tier-1 contract test for ghostwriter, `test_version_matches_changelog`, that reads `CHANGELOG.md` relative to `SKILL.md`'s own directory:

```python
ROOT = Path(__file__).resolve().parent.parent   # now the nested skill dir
# CHANGELOG.md deliberately stayed at the plugin root, one level up
PLUGIN_ROOT = ROOT.parent.parent
changelog_v = _changelog_top_version(
    (PLUGIN_ROOT / "CHANGELOG.md").read_text(encoding="utf-8")
)
```

`CHANGELOG.md` stays at the outer plugin root on purpose, next to `README.md`, so a human browsing the repo finds it without digging. The test assumed it lived beside `SKILL.md`, which was true before the move and false after. CI caught it on the pull request, correctly, before it merged: one path stayed correct because everything moved together, one broke because one file deliberately didn't move with the rest.

## Verify: the check that mattered ran against the real path

The Claude Code marketplace here is registered locally as a directory source, not a fixed git snapshot, which meant I could re-query it against my working tree directly instead of trusting the fix on faith:

```bash
claude plugin marketplace update claude-skills
claude plugin update ghostwriter@claude-skills
claude plugin details ghostwriter@claude-skills
```

`Skills (1) ghostwriter`, resolved from the new nested path, with nothing left at the old location for the CLI's lenient fallback to quietly cover for. A third bug turned up one step later, in the release automation that tags a version once it's merged, which is its own story. The pattern repeats, though: a code path only proves itself once something actually exercises it end to end, not when the parts around it individually look right.
