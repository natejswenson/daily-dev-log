## Shipped

`claude-skills` moved from manual symlink installs to a proper Claude Code plugin marketplace: a `.claude-plugin/plugin.json` per skill plus a root `marketplace.json`, installable with `/plugin marketplace add` and `/plugin install`. Right after that shipped, Claude Desktop's plugin UI showed "this plugin doesn't have any skills or agents" for all four skills. The CLI had been installing and running them fine the whole time. The fix moved each skill's `SKILL.md` one level deeper, from the plugin root to `skills/<name>/SKILL.md`, which is the only path Claude Code's plugin auto-discovery actually scans. Verified by checking that every one of the 23 plugins in [anthropics/claude-plugins-official](https://github.com/anthropics/claude-plugins-official) uses that nested layout, no exceptions.

The interesting part isn't the one-line fix. It's why a defect like this can ship, pass every test that exists, and only surface on a second client.

## Two clients, one spec, two outcomes

The CLI's plugin loader had an undocumented fallback: if it didn't find `SKILL.md` at the documented `skills/<name>/SKILL.md` path, it would also check the plugin root and load it from there anyway. Desktop's loader didn't have that fallback. It followed the documented path only, found nothing, and reported the plugin as empty.

Both clients were reading the same `plugin.json`, against the same documented convention. Only one of them enforced it. This is close to a textbook case of what's known as the robustness principle, or Postel's Law: "be conservative in what you send, be liberal in what you accept," originally coined for internet protocol design in 1979 ([Wikipedia](https://en.wikipedia.org/wiki/Robustness_principle)). The idea is that a tolerant receiver makes the overall system more resilient to small deviations. In practice, tolerance has a cost that shows up later, not immediately.

Marshall Rose documented the failure mode in 2001: a sender that emits slightly malformed output can run for years against a lenient receiver with nobody noticing, because the receiver quietly accepts the mistake instead of rejecting it. The bug doesn't go away, it goes dormant. It only resurfaces when a second, stricter implementation shows up and refuses to be as accommodating; by then the workaround has often calcified into the de facto behavior everyone has to keep supporting ([Wikipedia](https://en.wikipedia.org/wiki/Robustness_principle)). That's exactly what happened here. The CLI's fallback wasn't a feature, it was a silent tolerance for a layout that had never actually matched the documented spec, and it had been masking that mismatch since the first plugin conversion.

## The general fix: pick one discovery path and make every client enforce it

The actual repair had two parts, and only one of them was moving files.

**Part one, move the content to the path the spec says.** Every skill's functional content (`SKILL.md`, `scripts/`, `tests/`, `package.json`) moved from `skills/<name>/` (the plugin root) to `skills/<name>/skills/<name>/` (one level deeper). Only `.claude-plugin/`, `LICENSE`, `README.md`, and `CHANGELOG.md` stayed at the outer level:

```
skills/ghostwriter/
├── .claude-plugin/
│   └── plugin.json
├── CHANGELOG.md
└── skills/
    └── ghostwriter/
        ├── SKILL.md
        ├── scripts/
        └── tests/
```

This convention-over-configuration approach isn't unusual. VS Code extensions work the same way: every extension needs a `package.json` manifest at a fixed, predictable location relative to the extension root, and the editor's discovery logic depends on that location rather than on any config the extension author supplies ([VS Code Extension Manifest docs](https://code.visualstudio.com/api/references/extension-manifest)). A fixed path means the host application never has to guess or fall back; it either finds the manifest where the spec says it will be, or the plugin is broken.

**Part two, make the mismatch impossible to ship again.** A structural fallback in one client can't be relied on as documentation, so the CI lint that checks `plugin.json` consistency (`tools/lint_plugin.py`) was pointed at the nested path as the only source of truth:

```python
# Both SKILL.md and package.json live one level deeper than plugin.json,
# at <skill_dir>/skills/<dir_base>/ -- the ONLY path Claude Code's plugin
# auto-discovery scans. plugin.json itself stays at the outer level.
dir_base = os.path.basename(os.path.normpath(skill_dir))
nested_skill_dir = os.path.join(skill_dir, "skills", dir_base)

skill_md_path = os.path.join(nested_skill_dir, "SKILL.md")
package_json_path = os.path.join(nested_skill_dir, "package.json")
```

No fallback branch. If `SKILL.md` isn't at that exact path, the lint fails and the PR doesn't merge. The lenient CLI can keep whatever backward-compatible behavior it wants at runtime, but the thing that decides whether code merges never gets to be lenient about it.

## Use it

Any new skill added to the marketplace has to pass this before it's addable:

```bash
python tools/lint_plugin.py skills/ghostwriter
```

A skill with `SKILL.md` sitting at the plugin root instead of the nested path fails immediately with a clear "missing file" error pointing at the exact expected location, instead of silently working in one client and silently failing in another.

## Verify it

The regression test here isn't really "does the plugin load," because that already looked like it passed. It's "does the plugin load with *zero tolerance for path drift*, on every client, not just the one that happens to have a fallback." Running `/plugin install` against a fresh Desktop instance is the real verification; the CLI passing was never sufficient evidence on its own, since it was the CLI's leniency that hid the problem in the first place. The lesson that generalizes: when one implementation of a spec behaves more forgivingly than another, don't treat the forgiving one as ground truth. It's the thing most likely to be quietly wrong.
