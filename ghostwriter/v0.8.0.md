## Shipped

Ghostwriter generates the images that ride along with a LinkedIn post: diagrams, comparison cards, code snippets, carousels. Version 0.8.0 redesigned all of them, from a flat dark look to a lighter, more modern one. The colors are the least interesting part. What turned this into a few hours of work instead of a few days is that every card type re-themes with one CSS class, because the styling runs on design tokens and the final images are rendered from HTML by a headless browser. That pairing, a tokenized stylesheet plus a deterministic render step, is the part worth keeping, so let me walk the whole build.

## Start with tokens, not colors

A design token is a named variable that holds a design decision, like a background color or a border, so the rest of your styles reference the name instead of the raw value. The [W3C Design Tokens Community Group](https://www.w3.org/community/design-tokens/) treats them as the single source of truth for color, typography, and spacing, and the goal of a token layer is exactly this kind of theming: light and dark modes from one definition, no duplicated files.

In plain CSS you get tokens for free with custom properties. Define the semantic set once at the root and let it be the default theme:

```css
/* tokens.css — the single source of truth. Components never hardcode a color. */
:root {
  --bg:      #0d1117;  /* the canvas */
  --surface: #161b22;  /* a panel fill */
  --border:  #30363d;
  --text:    #e6edf3;  /* primary text */
  --muted:   #9da7b3;  /* secondary text */
  --accent:  #2d7df6;
}
```

The discipline that makes the rest possible: a component may read `var(--surface)` or `var(--text)`, but it may never write `#161b22`. If a raw hex appears in a component rule, theming will not reach it later.

## Flip the theme with one class

Because custom properties cascade, you can re-theme an entire subtree by redefining the tokens on a single ancestor. MDN notes that a custom property is [scoped to the element it is declared on and its descendants, and resolved where it is used](https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Cascading_variables/Using_custom_properties), not stored once at declaration time. So a `light` class that overrides the tokens cascades down to every nested component, and each one re-resolves its `var()` against the new values:

```css
/* One class redefines the tokens. Everything that reads them re-themes. */
.light {
  --bg:      #f7f8fa;
  --surface: #ffffff;
  --border:  #e6e9ee;
  --text:    #0c111b;
  --muted:   #727b8a;
  --accent:  #2d7df6;
}

/* A component reads tokens only. It has no idea which theme is active. */
.panel {
  background: var(--surface);
  border: 1px solid var(--border);
  color: var(--text);
  border-radius: 22px;
  padding: 34px 38px;
}
.panel .title { color: var(--text); }
.panel .sub   { color: var(--muted); }
```

Now `<div class="card">` renders dark and `<div class="card light">` renders light, and the `.panel` rule did not change a line. That is the whole trick: the component describes structure in terms of tokens, and the theme is just which token values are in scope. Adding a third theme later is one more class with one more set of values, not a rewrite of every component.

## Render the HTML to a PNG

A stylesheet is not an image yet. Ghostwriter renders each card by loading the HTML in a headless Chromium and screenshotting a single fixed-size element. Playwright makes this short, and two options matter. You can [screenshot a single element](https://playwright.dev/docs/screenshots) clipped to its box, and you can [set `device_scale_factor` to 2](https://playwright.dev/docs/emulation) so each CSS pixel becomes two image pixels and the output is crisp in the feed.

```python
# render.py — turn a styled #canvas into a high-DPI PNG, deterministically.
from playwright.sync_api import sync_playwright

def render(html: str, out: str, width: int, height: int) -> None:
    with sync_playwright() as p:
        browser = p.chromium.launch()
        page = browser.new_context(
            viewport={"width": width, "height": height},
            device_scale_factor=2,            # 2 image pixels per CSS pixel
        ).new_page()
        page.set_content(html, wait_until="load")
        page.wait_for_selector("#canvas")
        page.locator("#canvas").screenshot(path=out)   # clipped to the element
        browser.close()
```

The card HTML is one element at a fixed size with the theme class on it:

```python
html = '''
<div id="canvas" class="card light" style="width:1200px;height:1500px">
  <div class="panel"><div class="title">Hello</div><div class="sub">world</div></div>
</div>'''
# plus a <style> with tokens.css inlined

render(html, "card.png", 1200, 1500)   # portrait 4:5
```

You get back a 2400x3000 PNG that is exactly the `#canvas` box and nothing else, ready to attach to a post. The render is deterministic: same HTML and same tokens produce the same pixels every time, which is what makes a generated-image system safe to automate.

## The one token I forgot

Here is where the theming bit me, and it is the lesson worth the whole post. When I built the light class I flipped the obvious tokens, the background and the ink, and re-rendered. Most of the card looked right. But the stage titles on one card type had vanished. White text on a white panel.

The cause was a token I did not redefine. A few components read `var(--text)`, and my light class had only set a different token:

```css
/* The trap: I flipped the obvious tokens but missed one. */
.light {
  --bg:  #f7f8fa;
  --ink: #0c111b;
  /* --text was NOT redefined, so it stayed #e6edf3 from :root */
}

.stage-title { color: var(--text); }   /* near-white on a white panel = invisible */
```

The component was doing exactly what it should, reading a token. The token still held the dark value, because I only re-themed part of the set. The fix is not in the component. It is to redefine the full semantic token set on the modifier, every name a component might read, not just the ones you remembered:

```css
.light {
  --bg:#f7f8fa; --surface:#fff; --border:#e6e9ee;
  --text:#0c111b; --muted:#727b8a; --accent:#2d7df6;
}
```

For a visual system the honest check is to render and look at the PNG, since a passing build can still hide unreadable text. When you want that check in CI, Playwright can read a computed color and fail on the dark value leaking through:

```python
title = page.locator(".stage-title").first
color = title.evaluate("el => getComputedStyle(el).color")
assert color != "rgb(230, 237, 243)", "title kept the dark-theme color on the light card"
```

The token layer gives you cheap theming, and the cost is that a missed token fails silently somewhere downstream instead of loudly at the source. Theme the complete set, then render and read the image before you trust it.

## Sources

- [Design Tokens Community Group (W3C)](https://www.w3.org/community/design-tokens/) — design tokens as the single source of truth, and theming light/dark from one definition.
- [Using CSS custom properties (variables) — MDN](https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Cascading_variables/Using_custom_properties) — custom properties are scoped to an element and its descendants and resolved where used, which is why redefining them on one class re-themes the subtree.
