## What I Built

Today started with an App Store rejection email and ended with a resubmission. Apple's reviewer tested on an iPad Air 11-inch (M3), caught two broken flows, and kicked the build back under Guideline 2.1(a) — App Completeness. Fixing both and then making sure the whole app actually *feels* like an iPad app ate the day.

**Bug 1: The "Already set up? Skip" button was unresponsive.** The symptom was misleading — the button fired, but nothing happened. The root cause was in the launch routing: `RootView` required *both* `hasCompletedOnboarding` and a non-empty profile list to show the home screen, so when the user tapped Skip, the flag flipped but the profiles array was still empty, and the app silently re-rendered the onboarding flow. The fix was to have `completeOnboarding()` create a default kid profile if none exists, so skipping actually lets the user past the welcome screen. Not glamorous, but it explains why the button "didn't work" in review.

**Bug 2: The Tip Jar didn't load.** Harder to fully fix because this one has two failure modes — code *and* App Store Connect configuration. On the code side I added auto-retry on first-load failure, a clearer error state with a visible Close button so users aren't trapped, and made sure the retry path actually resets `loadError`. On the ASC side the original large tip localization was rejected for putting the price literally in the description ("$4.99 Tip") which violates IAP metadata rules, and the display name had marketing copy Apple doesn't allow. Rewrote all three tip products with plain descriptions, no prices in text, no exclamation points. The large tip got renamed in the process to `tip_large_999` at $9.99, which required updating the `productIds` array, the subtitle/icon switch statements, and the local `.storekit` testing config to match.

**Then the bigger job: redesign the entire iPad experience.** The prior build shipped as basically "iPhone UI stretched onto a giant canvas," which is why the reviewer's device uncovered problems in the first place — everything was iPhone-sized on an 820×1180 screen. Rather than sprinkle size-class branches at individual call sites, I took a systematic pass through every screen the reviewer would touch:

- **The jar itself needed two fixes.** First, `CleanJarView` had no explicit frame, so its layout box didn't match its visual extent — text placed below the jar overlapped the glass. Gave it `.frame(width: jarSize, height: jarSize)` so SwiftUI measures it correctly everywhere. Second, the 8 paper balls inside the jar were hardcoded at 48–56pt regardless of container size, so on iPad they looked like peas in a fishbowl. Added a floor-at-1.0 scale factor `max(1.0, jarSize / 346)` — iPhone unchanged, iPad gets proportionally larger balls. Then bumped the home jar formula cap from 480 to 720 and the height coefficient from 0.54 to 0.72 so the jar actually dominates the iPad screen. iPhone behavior is byte-identical since iPhone is always width-bound on the 0.88 coefficient.

- **The splash screen redesign was the most fun.** iPad splash jar grew to 560pt, title to 84pt Caveat-Bold, subtitle to 22pt. `SplashBall.generate()` now takes an `isPad` flag and produces 12 balls (vs 8) at 56–88pt (vs 28–44pt) with a 180–320pt burst distance (vs 80–140pt). The burst actually fills the screen on iPad instead of clustering around a single point.

- **Full scaling pass across every screen.** Home header avatar 54→96pt, settings gear 36→68pt, `SocialContextSegment` circles 32→62pt with larger icons and padding — kid-friendly tap targets well above HIG minimums. Onboarding form: age-group pills got 2× height, category tiles got proper weight (58→104pt), avatar picker circles 44→84pt, color swatches 24→36pt, section labels 13→18pt. Activity reveal: paper card now centers vertically on iPad instead of top-anchoring against a huge empty middle, activity name Caveat jumped 36→56pt, description 15→20pt, info chips and material tiles all scaled. Activity detail: `HoldToDoneButton` 92→136pt with a 24pt "Done!" label, step card titles 15→20pt, "Let's go!" header 32→48pt. And SimilarActivitiesView ("More Like This") got a prominent capsule Done button instead of a plain text link because it was getting lost in the corner.

- **Adaptive sheet presentation.** iPad's default `.sheet()` is a form-sheet — a ~540×620pt box floating over a dimmed background. That's why Settings looked like a tiny card in the middle of nothing. I wrote a `View+AdaptivePresentation.swift` extension that exposes `.adaptiveSheet(isPresented:)` which uses `fullScreenCover` on iPad and the normal `.sheet` on iPhone. Applied it to Settings, Edit Kid, Tip Jar, Privacy, Notification Settings, Favorites, Weekly Stats, Add Kid, and Similar Activities. Before shipping I audited every one of those views for explicit dismiss UI (toolbar Done, Cancel button, or capsule Done button) so nobody gets trapped on iPad — `fullScreenCover` has no swipe-to-dismiss gesture, so you need a real button to get out.

- **Background consistency.** Activity detail was using flat `Color.appBackground` while everything else uses the pastel `WavyBackgroundView`. Swapped it. Had to remove the gradient fade / solid background from the `doneArea` so the wavy pattern shows through behind the Hold-to-Done button.

Everything iPad-specific is gated behind `horizontalSizeClass == .regular` or an `isPad` helper, so iPhone rendering is completely untouched. Verified with `xcodebuild` against both iPad Air 11-inch (M3) and iPhone 17 Pro simulators after every change.

The resubmission went out at 22:06 as build 1.0 (6), tagged `v1.0_build_6`. Three tip IAPs (`tip_small_099`, `tip_medium_299`, `tip_large_999`) are attached to the version and in Ready to Submit.

## What's Next

Wait for Apple's verdict. If it comes back clean, start planning 1.1 — probably more activities and some of the UX polish I noticed while doing this audit. If it comes back rejected again, the most likely cause is IAP propagation (products not approved in parallel with the binary), which is fixable with another pass but not a code problem.
