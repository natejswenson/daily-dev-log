## What I Built

Until now the good part of local-fitness lived in the wrong place. The coaching synthesis, the curated brief, the coach voice, the saved notes, all of it ran server-side in the brief loop and never reached an MCP client. So if I opened Claude and wanted to ask about my training, the data was not really there. v0.3.0 closes that gap by making the MCP the primary way I talk to my own data.

The fitness database and the coaching surface are now exposed over MCP. There is a `coach` prompt that assembles the daily snapshot plus the coaching persona in a single round-trip, a `fitness://schema` resource and a `fitness://brief/latest` resource, and a one-call `daily_snapshot` tool so an agent can get the whole picture without stitching together five queries. Workouts come back in miles, formatted server-side. And for the first time there is a write surface: I can log observations and manual workouts that feed straight into the CTL/ATL/TSB training-load model.

The implementation reuses the SDK's already-wired tool server over a new transport rather than standing up a parallel one, so there is one source of truth and no duplicated schemas or handlers. It mounts behind the existing bearer-token gate, with a stdio mode for local auth-free use. The brief loop is fenced off from the new write tools at the structure level; it can only ever call the read-only set, so generating a brief can never mutate data. A design and quality-gate pass earned its keep again, catching execution-only gotchas like DNS-rebinding protection and a catch-all route shadowing the MCP mount, each of which became a passing regression test instead of an assumption.

## What's Next

This is the setup for the bigger inversion: if the agent can read and write everything over MCP, the synthesis no longer needs to run inside the server at all. That becomes the agent-first rearchitecture in v0.5.0.
