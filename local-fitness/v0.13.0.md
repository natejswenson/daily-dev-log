## Shipped

Local-fitness is the personal app that turns my Garmin data into a daily brief. Version 0.13.0 added a chart tool the agent can call, and alongside it did a round of coverage and YAGNI hardening: a large batch of new tests, and the removal of frontend and config code that nothing used anymore. Deleting code feels risky, which is why most unused code never gets deleted. The thing that made it safe here was writing tests around the code I was keeping before touching the code I was removing. That is a repeatable move, so here it is end to end.

## YAGNI applies to code you already wrote

YAGNI, "you aren't gonna need it", is usually aimed at the future: [don't add code today for a feature you only need tomorrow](https://martinfowler.com/bliki/Yagni.html), because it bloats the system and makes it harder to understand. The same logic runs backward. Code that was speculative when you wrote it, and still has no caller, is carrying the exact cost Fowler describes right now. A `StatCard` component nothing renders and a `utils` helper nothing imports are not free; they are surface area every reader has to scan and every refactor has to drag along. So the cleanup is not cosmetic. It is removing the cost of code you turned out not to need.

## Find what is actually unused

The first step is evidence, not vibes. Coverage is good at this. Fowler's point about [test coverage](https://martinfowler.com/bliki/TestCoverage.html) is that a coverage number is a poor target but a useful tool, and one of its honest uses is finding code that no test and no path exercises. Run it and look at the zeros:

```bash
# Python: show per-file coverage, surface the totally-uncovered modules
pytest --cov=src/local_fitness --cov-report=term-missing

# A module sitting at 0% that you also can't find a caller for is a deletion candidate.
```

For the frontend, the compiler is the equivalent tool. Type-checking the whole frontend in CI (which this release also wired into the validate job) turns an unused export into a signal you can grep for:

```bash
# Is anything importing this component anywhere?
grep -rn "StatCard" web/src --include="*.tsx" --include="*.ts"
# Only its own definition and test? Then nothing depends on it.
```

## Pin what you keep before you cut

Here is the part that makes deletion calm instead of scary. Before removing the dead module, make sure the behavior you are keeping is locked down, so that if a deletion has a side effect you did not predict, a test goes red instead of a user. This is a characterization test: it asserts current behavior so a refactor cannot quietly change it.

```python
# test_brief.py — pin the behavior that must survive the cleanup
def test_brief_has_takeaways_for_a_normal_day(client):
    resp = client.get("/api/brief?date=2026-06-20")
    assert resp.status_code == 200
    body = resp.json()
    assert len(body["takeaways"]) >= 1
    assert all("headline" in t for t in body["takeaways"])
```

With the kept paths under test, the deletion is a non-event:

```bash
git rm web/src/components/StatCard.tsx web/src/lib/utils.ts
pytest && npm --prefix web run type-check   # green = nothing depended on them
```

If either command goes red, you just learned about a real dependency cheaply, before it shipped. If they stay green, the code is genuinely gone and the suite proves it.

## The order is the whole technique

The reason this works is sequence. Tests first means the safety net exists before you remove anything. Coverage and the type-checker tell you what has no caller, the characterization tests freeze what does, and then deletion is just confirming the net holds. Adding a feature and removing dead weight in the same release is not a contradiction; the tests you write to make the new chart tool safe are the same tests that make the old `StatCard` safe to delete. Keep less code, and keep the tests that let you keep less code.
