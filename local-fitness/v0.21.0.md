## Shipped

This release retired the last piece of my fitness agent's web UI (a
React/Vite dashboard nobody had opened in months, since every real
interaction goes through Claude or opencode over MCP now) and closed the
gap that retirement exposed: three new MCP tools so the agent can commit,
discard, or abandon a training plan, actions that used to only have a
button in a UI that no longer exists.

The other half of the release is less visible and more interesting to
write about. A few of the MCP tools that answer "how's my training going"
were opening far more SQLite connections than one call should need, up
to nine for a single brief. I fixed it, and then had to answer a harder
question than "did I fix it": how do I know, in a way a reviewer (or
future me) doesn't have to just trust?

## The problem: nine connections for one answer

The [N+1 query problem](https://docs.sentry.io/product/issues/issue-details/performance-issues/n-one-queries/)
is usually described as an ORM issue: fetch a list, then loop over it
making one more query per row. My version wasn't an ORM loop, it was
worse in a duller way. `assemble_brief_context`, the function that
answers "what should today's coaching brief say", called into half a
dozen helper functions, and each helper opened its own `sqlite3.connect()`
and closed it when it returned. Nothing was wrong in isolation. Composed
together, one MCP tool call opened nine short-lived connections to the
same local file to answer one question.

[PlanetScale's writeup on the problem](https://planetscale.com/blog/what-is-n-1-query-problem-and-how-to-solve-it)
puts the general fix as batching, fetch what you need in one query
instead of many. Mine wasn't batchable in that sense, the queries were
already different questions (today's metrics, the training plan, recent
workouts). The fix was connection reuse: thread one optional
`conn: sqlite3.Connection | None = None` parameter through every function
in the call chain, and have the top-level caller open one connection and
pass it down. Callers that don't pass one still get the old
one-connection-per-call behavior, so nothing upstream breaks.

That part was a normal refactor. Proving it actually did what I claimed
is the part worth a full writeup.

## Setup: making "faster" a testable claim

I added [`pytest-benchmark`](https://pytest-benchmark.readthedocs.io/)
as a dev dependency and pointed it at a synthetic, fabricated multi-year
SQLite fixture (never real user data) with an active training plan, so
the benchmark exercises the same code paths a real brief would. By
default I don't want benchmark tests slowing down every `pytest` run, so
`pyproject.toml`'s `addopts` carries `--benchmark-skip`:

```toml
[tool.pytest.ini_options]
addopts = "--benchmark-skip"
```

`--benchmark-skip` composes with an explicit `--benchmark-only` later,
the flag that turns it back on wins, so a normal `pytest -x` run never
pays the benchmark cost, and a dedicated CI step can still opt back in.

## Build it: assert the connection count, not just the clock

A benchmark that only measures wall-clock time can pass for the wrong
reason, noisy CI runners, warm disk cache, a lucky run. What I actually
wanted to prove was structural: this call opens one connection, not nine.
So the test does both, times the call with pytest-benchmark's `benchmark`
fixture, and separately counts real `sqlite3.connect()` calls with a
thin counting wrapper:

```python
import sqlite3
import pytest

def _count_connects(monkeypatch):
    """Wrap sqlite3.connect to count real connection opens without
    changing its behavior."""
    calls = {"n": 0}
    real_connect = sqlite3.connect

    def counting_connect(*args, **kwargs):
        calls["n"] += 1
        return real_connect(*args, **kwargs)

    monkeypatch.setattr(sqlite3, "connect", counting_connect)
    return calls

def test_assemble_brief_context_opens_one_connection(benchmark, monkeypatch, fixture_db):
    calls = _count_connects(monkeypatch)

    result = benchmark(assemble_brief_context, db_path=fixture_db)

    assert result is not None
    assert calls["n"] == 1, f"expected 1 connection, opened {calls['n']}"
```

Before the fix, this same assertion read `calls["n"] == 9`. I verified
that number empirically with `git stash` (run the test against the old
code, note the count, restore the fix) rather than trusting my own memory
of how many helpers the function called.

## Wire it into CI: fail the PR, not just log a warning

A benchmark nobody looks at is decoration. The useful version fails the
build when a PR regresses. `pytest-benchmark` supports comparing the
current run against a saved baseline and failing on a threshold:

```yaml
- name: Perf-benchmark regression gate
  run: >
    uv run pytest --benchmark-only --no-cov
    --benchmark-storage=file://./.benchmarks
    --benchmark-compare=0001
    --benchmark-compare-fail=min:15%
```

`--benchmark-compare-fail=min:15%` means: if the minimum observed time
for any benchmark got at least 15% worse than the saved baseline named
`0001`, fail the step. `--no-cov` matters for a narrower reason, this
project also runs a global coverage gate, and a benchmark-only invocation
touches a small slice of the codebase, so it needs to opt out of coverage
enforcement rather than fail on low coverage for code the benchmark step
was never meant to exercise.

## Two ways this gate almost lied to me

**The baseline has to come from the same kind of machine, and "same kind"
is stricter than it sounds.** `pytest-benchmark` groups saved runs by a
machine id string built from
[`platform.system()`, `platform.python_implementation()`, the Python
version, and `platform.architecture()`](https://github.com/ionelmc/pytest-benchmark/blob/master/src/pytest_benchmark/utils.py),
something like `Linux-CPython-3.12-64bit`. I first captured my baseline
in a local `python:3.12-slim --platform=linux/amd64` Docker container on
my Apple Silicon Mac. The machine id string matched GitHub's
`ubuntu-latest` runners exactly, so the comparison ran without
complaint, and every single benchmark "regressed" 27 to 42% on the very
first real CI run. The string matched; the hardware didn't. Docker
Desktop's emulated CPU on Apple Silicon is dramatically slower than real
cloud metal, and `pytest-benchmark` has no way to know that from the id
string alone. I recaptured the baseline from an actual `ubuntu-latest`
run and the false regressions disappeared. The lesson: a benchmark
baseline is only valid from the exact class of machine that will run the
comparison, not from anything that merely reports the same platform tag.

**The baseline file has to actually make it into the CI artifact.**
`actions/upload-artifact@v4` ships with
[`include-hidden-files` defaulted to `false`](https://github.com/marketplace/actions/upload-a-build-artifact),
and `.benchmarks/` is a dotdir. My upload step ran clean, no error, no
warning in the log I was watching, it just silently excluded the one
directory I needed. The tell was in a different step entirely: pytest's
own log said "Saved benchmark data in: .benchmarks/...", and the very
next step said "No files were found with the provided path." Same job,
same workspace, no error connecting the two lines. Adding
`include-hidden-files: true` fixed it. I wouldn't have caught either of
these by reading the YAML twice, only by running the pipeline for real
and comparing the number I expected against the number I got.

## Verify it

The gate is only worth what it does on a real pull request. Here's what
a passing run looks like once both fixes above are in place:

```
Name (time in ms)                          Min      Max     Mean   Rounds
--------------------------------------------------------------------------
test_assemble_brief_context_...          4.1102   5.9834   4.3201      50
--------------------------------------------------------------------------
Comparing against 0001: min -2.11%, mean -1.84%
```

And the failure mode I'm actually guarding against, a future change that
reintroduces a per-lookup connection open, would show up as a `min` well
past +15% and a red CI check, not a "huh, feels a little slow today"
someone has to notice by hand.
