## Shipped

My fitness agent's PDF brief report got reworked this release. The takeaway
cards used to stack in one long column; now they reflow into a 2-column
grid that holds up no matter how many takeaways a given day produces. Below
that, there's a new Training Plan section: adherence percentage, days to
race, this week's planned versus actual mileage, a count of slipped
workouts, and a table grading the last 7 days against what I was actually
supposed to do.

The section also includes one paragraph written by Claude: a short line
prepping me for today's prescribed run, generated fresh every time the
report renders. That's the part worth digging into, because a single LLM
call sitting in the middle of a report generator is also a single point of
failure, and I wasn't willing to ship a PDF that can go blank because an
API call timed out.

## Why a report can't just trust the model to answer

A PDF render is a synchronous, user-facing action: I ask for the report, I
want the file. If the coaching-line call hangs, or the API is overloaded,
or the SDK returns nothing, the rest of the report (charts, adherence
numbers, the graded table) is all already computed and correct. Losing all
of it because one paragraph didn't come back would be a bad trade.

The fix is a pattern that shows up constantly once you start looking for
it: pair every non-essential model call with a deterministic path that
covers for it. [Martin Fowler's writeup on the Circuit Breaker
pattern](https://martinfowler.com/bliki/CircuitBreaker.html) frames the
general shape well: wrap the risky call, and when it fails, "all further
calls... return with an error, without the protected call being made at
all" rather than letting the caller hang waiting on a resource that isn't
coming back. My version doesn't need the stateful trip-and-reset machinery
a circuit breaker uses for a hot path under repeated load; it's a single
render, once a day, so a plain try/except with a hard timeout does the
same job at a much smaller scale.

## Build it: the primary call, on a deadline

The primary path is a single-turn Claude Agent SDK call. No tools, one
turn, and a hard timeout so a hung connection can't hang the whole report:

```python
import asyncio

async def generate_llm_text(prompt: str, *, model: str, timeout: float = 30.0) -> str:
    """Single-turn LLM call with a hard deadline. Raises on timeout,
    on a transport error, or on an empty response - it never returns
    partial or blank output as if it succeeded."""
    from claude_agent_sdk import AssistantMessage, ClaudeAgentOptions, TextBlock, query

    options = ClaudeAgentOptions(model=model, permission_mode="bypassPermissions", max_turns=1)

    async def _run() -> str:
        chunks: list[str] = []
        async for message in query(prompt=prompt, options=options):
            if isinstance(message, AssistantMessage):
                for block in message.content:
                    if isinstance(block, TextBlock):
                        chunks.append(block.text)
        return "".join(chunks).strip()

    text = await asyncio.wait_for(_run(), timeout=timeout)
    if not text:
        raise RuntimeError("LLM call returned an empty response")
    return text
```

That last check matters more than it looks. An empty string is a
successful API call at the transport layer, HTTP 200, no exception, and it
is also not a usable coaching line. Treating "returned nothing" as failure
rather than success is what makes the fallback trigger reliably instead of
only on network errors. [Anthropic's own error reference](https://platform.claude.com/docs/en/api/errors)
documents a fairly wide surface of distinct failure types a call like this
can hit, rate limits, overloaded-server errors, timeouts, connection
failures, which is the other reason this catches broadly rather than
hand-picking exception types: there's no single "the network is down"
exception to catch, there's a list, and the fallback needs to fire for all
of it, not just the ones I thought to enumerate.

## Build it: the fallback that can't fail

The fallback is a plain function. No I/O, no randomness, same inputs
always produce the same output:

```python
_VERDICT_PHRASE = {
    "done": "You hit yesterday's session clean.",
    "partial": "Yesterday came up short of the prescription.",
    "missed": "Yesterday was a skip.",
}

def deterministic_coaching_line(today: dict, history: list[dict], days_to_race: int | None) -> str:
    """Template-based coaching line, used only when generate_llm_text fails.
    Pure and side-effect free - this is the line of last resort, so it has
    to always produce something reasonable."""
    parts: list[str] = []
    prior = next((d for d in history if d.get("verdict") != "pending"), None)
    if prior is not None and prior["verdict"] in _VERDICT_PHRASE:
        parts.append(_VERDICT_PHRASE[prior["verdict"]])

    prescription = today["type"]
    if today.get("distance_mi"):
        prescription += f" {today['distance_mi']} mi"
    parts.append(f"Today: {prescription}.")

    if days_to_race is not None:
        parts.append(f"{days_to_race} days out.")

    return " ".join(parts)
```

## Wire them together at the call site

The caller is where the two paths actually meet, and it's deliberately
boring: try the real thing, catch everything, fall back, log so the
failure is visible even though the user never sees it:

```python
try:
    coaching_line = await generate_llm_text(prompt, model=DEFAULT_MODEL)
except Exception:
    log.warning("coaching-line generation failed, using fallback", exc_info=True)
    coaching_line = deterministic_coaching_line(today, history, days_to_race)

report["coaching_line"] = coaching_line
```

Everything upstream of this block, the chart rendering, the adherence math,
the graded table, has the same shape: compute a section, and if it throws,
log it and let the report render without that section rather than failing
the whole PDF. The coaching line just carries it one step further, because
unlike a missing chart, a missing paragraph would leave an obviously blank
gap in a "prescribed workout" box. It doesn't get to skip, it gets a
stand-in.

## Use it

Nothing about calling the report changes. The report-generation tool
still takes a date and returns a file path; the two-path logic above is
invisible from the outside:

```python
result = await generate_brief_report({"date": "2026-07-09"})
# {"path": "/tmp/brief-report-.../brief-2026-07-09.pdf"}
```

On a normal day, that coaching line is Claude's paragraph. On a day the
API is overloaded or my token's expired, it's the template, and the PDF
still opens looking complete.

## Verify it, and the failure mode that's easy to miss

The test that matters most isn't "does the primary call work", it's "does
the fallback actually get used when the primary call fails, and does it
produce something sane":

```python
def test_fallback_used_when_primary_raises(monkeypatch):
    async def boom(*args, **kwargs):
        raise RuntimeError("simulated network failure")

    monkeypatch.setattr("mymodule.generate_llm_text", boom)

    line = asyncio.run(build_coaching_line(today, history, days_to_race=6))

    assert line == deterministic_coaching_line(today, history, 6)
```

The failure mode worth calling out is the one this test guards against
directly: a fallback path that only gets exercised in production, on the
one day the primary call actually breaks, is a fallback you've never
actually tested. [Google's SRE book](https://sre.google/sre-book/handling-overload/)
makes this point about degraded-response code paths in general: "the code
path you never use is the code path that (often) doesn't work," and its
suggested fix, for their scale, is to deliberately exercise the degraded
path under real load on a regular basis. I don't have real load to
schedule against, so I did the smaller-scale equivalent: a fallback that
is pure, has no I/O, and gets its own unit test with fixed inputs and a
pinned expected output, so I know it works before the day it's the only
thing standing between me and a broken report.

## Sources

- [Martin Fowler, "CircuitBreaker"](https://martinfowler.com/bliki/CircuitBreaker.html): the general pattern of wrapping a risky remote call so failure returns immediately instead of hanging the caller.
- [Anthropic, "Errors" API reference](https://platform.claude.com/docs/en/api/errors): the actual set of distinct failure types (rate limits, overload, timeouts) a Claude API call can hit, which is why the catch is broad rather than hand-picked.
