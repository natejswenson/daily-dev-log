## Shipped

local-fitness v0.15.2 fixes a bug where the daily Garmin sync re-authenticated on every run. The client now passes an explicit token-store path when it logs in, so a pull resumes a cached session instead of doing a fresh login each time. Repeated logins were tripping Garmin's login rate limit, so the job that runs every morning kept logging a 429. The part worth writing about is where the rate limit came from: the logins, not the data calls.

## The login is often the expensive call

When you talk to an authenticated API, it is easy to picture the data requests as the load. For a background job, the bigger cost is frequently the authentication itself. A full login hits a dedicated auth endpoint, and those endpoints are usually rate-limited harder than read endpoints, because they are the ones credential-stuffing attacks hammer.

An HTTP 429 is the server asking you to slow down. [RFC 6585](https://datatracker.ietf.org/doc/html/rfc6585) defines `429 Too Many Requests` as the response for rate limiting, and it may carry a `Retry-After` header telling you how long to wait ([MDN](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status/429)). If a scheduled job authenticates from scratch on every run, and two runs land close together, it can trip that limit on the login alone, before it reads a single row.

That was the shape of this bug. The pull called `login()` with no arguments every time, so every run did a fresh credentialed login. Two runs inside a short window, the 6:30 job plus a manual sync, were enough to earn a 429.

## What the protocol already gives you

OAuth 2.0 is built to avoid exactly this. It issues a short-lived access token alongside a longer-lived refresh token, and the refresh token exists so a client can obtain new access tokens without the user authenticating again ([RFC 6749](https://datatracker.ietf.org/doc/html/rfc6749)). The access token stays short-lived on purpose, so a leaked one has a small window; the refresh token quietly renews it in the background.

The practical takeaway is simple: authenticate once, persist the session, and reuse it. You only pay for a full login when there is no valid session left to resume. Most well-behaved API clients support this if you give them a place to store the token. The common way to break it is to never give them that place.

## Build it: resume, refresh, then login

Here is the pattern as a small, general helper. Three steps in order: resume a cached session, refresh it if it is close to expiring, and only fall through to a full login when there is nothing to resume, persisting the result so the next run can resume too.

First, decide where the session lives. Let a deployment override the location, and default to a spot that works on a fresh clone with no setup:

```python
import os
from pathlib import Path

def token_store_path() -> str:
    override = os.environ.get("API_TOKEN_STORE")
    if override:
        return override
    return str(Path.home() / ".myapp" / "session.json")
```

Then the resume-refresh-login ladder:

```python
import json
from pathlib import Path

def get_session(store: str) -> Session:
    path = Path(store)

    # 1. Resume: reuse a cached session when one is on disk and still valid.
    if path.exists():
        session = Session.from_json(path.read_text())
        if not session.expires_soon():
            return session

        # 2. Refresh: trade the refresh token for a new access token.
        #    No full re-authentication, so no hit on the login endpoint.
        session = session.refresh()
        path.write_text(session.to_json())
        return session

    # 3. Only now pay for a full login, then persist it for next time.
    session = full_login()
    path.parent.mkdir(parents=True, exist_ok=True)
    path.write_text(session.to_json())
    return session
```

Many client libraries fold all three steps behind a single call: you pass a path and they resume, refresh, and persist for you. The bug in this release was that the code never passed the path, so the library quietly defaulted to step three on every run.

## Use it from the scheduled job

The daily job just asks for a session and gets to work. The first run does one real login and writes the file; every run after that resumes it:

```python
def daily_sync():
    session = get_session(token_store_path())
    for day in missing_days():
        ingest(session.get(day))
```

## Verify it, and the failure modes that bite

The test that matters is not "does a sync work", it is "does the second run avoid a second login". Pin that directly, so a regression back to per-run authentication fails the build:

```python
def test_second_run_reuses_session(tmp_path, monkeypatch):
    store = str(tmp_path / "session.json")
    logins = []
    monkeypatch.setattr(
        "myapp.full_login",
        lambda: logins.append(1) or fake_session(),
    )

    get_session(store)   # first run: one full login, session persisted
    get_session(store)   # second run: resumes from disk

    assert logins == [1]   # fails if we logged in twice
```

Two failure modes are worth knowing before they cost you an afternoon.

The first is a shared session file with mismatched owners. If a host process and a container both read the same bind-mounted file under different user ids, and one side tightens the file to owner-only permissions, the other can no longer read it and quietly falls back to a full login, which is the rate limit you were trying to avoid. I hit exactly this and dropped a permission-tightening step that would have caused it.

The second is a silent write failure. If persisting the session is wrapped in a broad `catch`, a failed write leaves you logging in every run with nothing in the logs to explain why. When a scheduled job mysteriously re-authenticates, confirm the token file is actually being written before you reach for anything fancier.

Backoff is the other half of the story, and it is worth having: for a 429 or 503, the standard advice is exponential backoff with jitter, so retries spread out instead of firing in lockstep ([Google Cloud](https://docs.cloud.google.com/storage/docs/retry-strategy)). But backoff only spaces out the requests you make; it does not reduce how often you authenticate. If a scheduled job keeps hitting a login rate limit, the first thing to check is whether it is reusing a saved session at all.
