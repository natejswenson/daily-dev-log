## Shipped

Local-fitness was a private repo, and version 0.15.0 did the work to make it shareable. Part of that was charts and an auth gate, but the part worth teaching is the security tooling I turned on first: CodeQL static analysis and dependency review, both free for public repos. A private repo can get away with a thin CI gate because the audience is one person. A public one is read, forked, and depended on by strangers, so the bar moves. These two workflows raise it without much effort, so here is each one.

## CodeQL: scan your own code for vulnerabilities

[CodeQL](https://docs.github.com/en/code-security/code-scanning/introduction-to-code-scanning/about-code-scanning-with-codeql) is GitHub's static analysis engine. It reads your code, looks for patterns that map to security problems like injection, broken auth, and path traversal, and reports them as code scanning alerts on the repo. Local-fitness is a Python backend plus a React/TypeScript frontend, so the workflow runs a matrix over both languages, uses the `security-and-quality` query suite, and adds a weekly scheduled run so new advisories get applied to code that has not changed:

```yaml
name: CodeQL
on:
  push: { branches: [main, dev] }
  pull_request: { branches: [main, dev] }
  schedule:
    - cron: "27 4 * * 1"   # weekly: catches new advisories on idle code
jobs:
  analyze:
    runs-on: ubuntu-latest
    permissions:
      security-events: write     # required to upload alerts
      contents: read
    strategy:
      fail-fast: false
      matrix:
        language: [python, javascript-typescript]
    steps:
      - uses: actions/checkout@v7
      - uses: github/codeql-action/init@v3
        with:
          language: ${{ matrix.language }}
          queries: security-and-quality
      - uses: github/codeql-action/autobuild@v3
      - uses: github/codeql-action/analyze@v3
```

The scheduled run is the part people skip. Most vulnerabilities you ship are not in code you are actively editing; they are in code that was fine when written and became a known issue later. A weekly scan re-checks the whole repo against the current rules, so idle code does not silently rot.

## Dependency review: gate the dependencies a PR adds

CodeQL covers code you wrote. The other exposure is code you pull in. [Dependency review](https://docs.github.com/code-security/supply-chain-security/understanding-your-software-supply-chain/about-dependency-review) runs on a pull request, diffs the dependency manifests, and flags any newly introduced package that carries a known vulnerability or a disallowed license, before the dependency lands. It pairs with Dependabot: Dependabot bumps the dependencies you already have, dependency review gates the new ones at the moment they are added.

```yaml
name: Dependency Review
on:
  pull_request: { branches: [main, dev] }
permissions:
  contents: read
jobs:
  dependency-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v7
      - uses: actions/dependency-review-action@v4
        with:
          fail-on-severity: high
```

By default the [action fails the check when it finds a vulnerable package](https://docs.github.com/en/code-security/how-tos/secure-your-supply-chain/manage-your-dependency-security/configure-dependency-review-action), and you tune the bar with `fail-on-severity`. Setting it to `high` means a PR that adds a high or critical CVE cannot merge, while low-severity noise does not block routine work. If you make the check required in branch protection, "do not add a known-vulnerable dependency" becomes a rule the repo enforces, not a habit you hope reviewers remember.

## What to check after you turn them on

Both workflows write to the repo's Security tab. After the first runs, open it and confirm CodeQL analyzed both languages and dependency review reported on a test PR. The honest expectation: the first CodeQL run on a real codebase usually surfaces a few findings, and that is the point. Triage them before you make the repo public, because once it is public, the unread alerts are visible too. Turn these on while the repo is still private, fix what they find, then flip the switch.
