## What Shipped

I asked my fitness agent a simple question: show me my training plan through today. It got there, but first it opened a shell, poked at the database schema, and ran a couple of SQL queries that errored before it finally printed a table. The answer was fine. Everything in front of the answer was noise I didn't ask for.

The problem wasn't the formatting. It was a missing tool. The agent had no clean way to ask for the graded plan day by day, so it reached for raw SQL. The data was already sitting there; only the web view knew how to read it.

v0.7.0 adds get_training_plan_progress, which returns the whole plan in one call: every prescribed workout with its verdict (done, missed, pending), plus days to race, adherence, and a projected finish. Now the agent makes one tool call and hands back a clean table. It doesn't open a shell or read the schema anymore.

I also added a short rule, in the prompt and in the project's own instructions, to reach for the structured tools first and never drop to sqlite by hand. The tool is the load-bearing part. The rule keeps it honest on the days the agent gets clever.

The change went through a design pass and a full review loop before any code landed, and the review paid for itself. It caught that the new tool needed real handling around the edge cases instead of a quick copy of the existing one, and it flagged a verification claim I couldn't actually back up. I would rather find that in review than read about it in a bug report later.

## What's Next

The brief's cross-model A/B check turned out to be flaky in a way that predates this release. Making that harness reliable is the next cleanup, so prompt changes stay cheap to verify.
