## Shipped

Local-fitness ships through a `feature → dev → main` flow, and promotions from `dev` to `main` are squash-merged. That combination has a sharp edge: after a squash promotion, the next `dev → main` pull request shows changes that were already merged, as phantom diffs. Version 0.14.0 added the fix as automation, a script plus a workflow that force-resets `dev` onto `main`'s commit after every promotion, so the two branches line up again. It is a small piece of release engineering with a couple of non-obvious parts, so let me walk it.

## Why a long-lived branch drifts

The root cause is in how squash merging works. GitHub's docs are explicit: a squash merge [creates a single new commit on the base branch, and because that commit is only on the base and not the head, the common ancestor of the two branches stays unchanged](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/incorporating-changes-from-a-pull-request/about-pull-request-merges). For a short-lived feature branch you delete after merge, that does not matter. For a long-lived branch like `dev` that you keep working on, it does: the next pull request between `dev` and `main` includes everything since that unchanged common ancestor, which is why GitHub recommends you do not squash-merge a branch you intend to keep using. The clean fix is to stop using the old `dev`: reset it to point at `main`'s new commit, so the histories share a tip again.

## Reset the branch to main's SHA

The reset itself is one API call. Read both branch tips, and if they differ, force-update the target to the base SHA. Make it idempotent so running it when nothing changed is a no-op:

```bash
#!/usr/bin/env bash
set -euo pipefail
REPO="${1:-$GITHUB_REPOSITORY}"
BASE="main"; TARGET="dev"

base_sha=$(gh api "repos/$REPO/git/ref/heads/$BASE" --jq '.object.sha')
target_sha=$(gh api "repos/$REPO/git/ref/heads/$TARGET" --jq '.object.sha')

if [ "$base_sha" = "$target_sha" ]; then
  echo "✓ $TARGET already at $BASE — nothing to do."; exit 0
fi
```

## Do the protected-branch dance, and always undo it

The catch: `dev` is protected, and by default [a branch protection rule disables force pushes](https://docs.github.com/en/rest/branches/branch-protection). A reset is a force update, so you have to briefly allow it. The safe pattern is to flip `allow_force_pushes` on, push, and flip it back, with a trap so protection is restored even if the push fails partway. Toggle exactly one field and preserve the rest of the protection config verbatim, so the branch is byte-identical before and after:

```bash
PROT="repos/$REPO/branches/$TARGET/protection"
snapshot="$(mktemp)"; gh api "$PROT" > "$snapshot"

# Re-marshal the GET payload into the PUT shape, flipping only allow_force_pushes.
build_body() {  # $1 = 1 to allow, 0 to forbid
  jq --argjson force "$1" '{
    required_status_checks: (if .required_status_checks then
      {strict: .required_status_checks.strict, contexts: (.required_status_checks.contexts // [])} else null end),
    enforce_admins: (.enforce_admins.enabled // false),
    required_pull_request_reviews: null,
    restrictions: null,
    required_linear_history: (.required_linear_history.enabled // false),
    allow_force_pushes: ($force == 1)
  }' "$snapshot"
}

restore() { build_body 0 | gh api -X PUT "$PROT" --input - > /dev/null; }
trap restore EXIT                                   # protection comes back no matter what

build_body 1 | gh api -X PUT "$PROT" --input - > /dev/null      # allow force pushes
gh api -X PATCH "repos/$REPO/git/refs/heads/$TARGET" -f sha="$base_sha" -F force=true > /dev/null
trap - EXIT; restore                                # restore on the happy path too
```

The `trap restore EXIT` is the load-bearing line. If the force-update throws, the script still re-locks the branch on the way out, so a failure never leaves `dev` unprotected.

## Run it after every promotion

Now make it automatic. A workflow triggered when a `dev → main` PR closes merged runs the script. The one requirement worth flagging: editing branch protection needs admin permissions, which the default `GITHUB_TOKEN` does not have, so the workflow passes a personal access token instead:

```yaml
name: reset-dev-after-promotion
on:
  pull_request:
    types: [closed]
    branches: [main]
jobs:
  reset:
    if: github.event.pull_request.merged == true && github.event.pull_request.head.ref == 'dev'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v7
      - run: ops/reset-dev-to-main.sh "$GITHUB_REPOSITORY"
        env:
          GH_TOKEN: ${{ secrets.ADMIN_PAT }}   # needs administration:write
```

After this, a promotion is followed seconds later by `dev` snapping back onto `main`, and the next promotion PR shows only genuinely new work. Verify it by reading the tips: `gh api repos/OWNER/REPO/git/ref/heads/dev` and the same for `main` should print the same SHA. The phantom diffs are gone because the branches share a tip again, which is the state squash merging quietly broke.
