## What I Built

### Training plans, end to end

The big one. You pick a goal — 5K, 10K, Half, Full, or Custom — a race date,
and a target time, and the coach drafts a periodized plan from your own Garmin
history. You riff with it in chat until it's right, then commit. Once committed,
the plan is tracked two places: a new **Training Plan tab** (a schedule with
per-day adherence verdicts, planned-vs-actual weekly mileage, a fitness
trajectory chart, and a goal header with a computed predicted finish) and the
**daily brief**, where it folds into the workout takeaway — yesterday's
adherence plus today's session, with recovery taking precedence over the
schedule on red-flag days.

I built this design-first. A full design doc went through an adversarial
security + correctness review before any code — that pass closed four Critical
and six High findings at the design stage, which is exactly when they're cheap
to fix. A few decisions that came straight out of it: the AI can only ever write
*draft* plans (activating or deleting is a human action, enforced in code, so a
prompt-injected note can't silently change what you're tracking); a single
active plan is guaranteed by a partial unique index at the database level; and
adherence is graded off the *data frontier* (the last day Garmin actually
synced) rather than the calendar, so the lag between a run and its sync never
produces a false "you missed it" roast. Then six test-first phases: schema,
pure logic, persistence + the agent tools, REST endpoints, brief integration,
and the React tab. Shipped as **v0.2.0**.

After the core landed I polished the runner-facing edges: the schedule shows
**Target vs Actual** distance and pace (red when you fell short), everything in
**miles** instead of kilometres, and the brief's "Today's Goal" card became a
**Today | Tomorrow** two-up so you can see what to do now and what to prep for.
Also fixed a sneaky timezone off-by-one that was rendering plan dates a day
early.

### A latency pass — and a negative result I'm glad I caught

The brief and chat run on the Agent SDK, so latency (round-trips), not dollars,
is the cost. I lined up five efficiency ideas, designed them, and put the design
through the same adversarial review. It split the work into two phases.

**Phase A shipped** — the safe, real wins: compact JSON for tool output (fewer
tokens every turn), a 3-way Haiku | Sonnet | Opus chat toggle (defaulting to
Sonnet, because quietly dropping the coaching path to the cheap model is a
silent quality hit you can't see), and a regression test that keeps the system
prompt cache-stable.

**Phase B got reverted.** The ambitious idea was to pre-compute the brief's data
in-process and inject it, collapsing ~8 sequential model round-trips into one.
It implemented cleanly — but the design's own gate demanded a *live before/after
latency measurement* before shipping, and that measurement said no: ~211s
tool-driven vs ~255s pre-fetched. The brief is **output-bound** — most of the
wall-clock is the model *writing* the takeaways, not gathering data — so killing
the round-trips barely moved the total. So I reverted it and kept Phase A. Six
test briefs well spent to avoid carrying complexity that doesn't earn its keep.
The whole point of insisting on a measurement gate is that it occasionally tells
you "no," and this time it did.

### A bonus bug, caught by the experiment

That latency A/B incidentally exposed that the brief only surfaced an active
training plan about a third of the time. The cause: the live plan starts
tomorrow, so the plan-status call returns "active, but no session today and
nothing graded yet" — a case the prompt hadn't defined, so the model improvised
and usually dropped the plan. Fixed it to always reference an active plan (goal
plus days-to-race) with explicit handling for the not-started / rest-day case.
Verified live: every test brief now leads with the plan — e.g. "Rest day on the
plan — sub-1:47 half is 89 days out."

I also ran a live security siege against the actual running container (auth
gating, injection, rate-limiting, the single-active invariant — all held) and
hardened two small things it flagged: stopped advertising the server stack and
added an HSTS header.
