## Shipped

The daily brief in local-fitness is generated by an LLM from my fitness data. Version 0.15.1 cut over to a V2 composer where the agent produces the brief as a structured object, an ordered list of takeaways, each with a headline, a tone, an optional metric to chart, and a markdown deep-dive. Handing a model the structured output is the easy part. Trusting it is the work. Three things made the cutover safe, and none of them is "the schema validated, ship it." Here they are.

## One: constrain the shape with a typed schema

Defining the output precisely is the first lever. Anthropic's guidance on [increasing output consistency](https://docs.anthropic.com/en/docs/test-and-evaluate/strengthen-guardrails/increase-consistency) is to specify the format exactly, and a [Pydantic](https://docs.pydantic.dev/latest/concepts/validators/) model does that on the receiving end: it is both the spec the prompt describes and the gate the response must pass. The detail that matters is the enums. A metric the brief charts has to be one the API can actually serve, so the type is a closed set, not a free string:

```python
from typing import Literal
from pydantic import BaseModel, Field, field_validator

MetricName = Literal["rhr", "sleep_score", "body_battery_max", "ctl", "atl", "tsb"]
Tone = Literal["positive", "caution", "critical", "neutral"]

class Takeaway(BaseModel):
    headline: str = Field(..., description="One-line insight, ~6-12 words")
    tone: Tone = Field("neutral")
    metric: MetricName | None = None
    details: str = Field(..., description="Markdown deep-dive shown when expanded")
```

The non-obvious bug: a model will occasionally emit a raw newline in the middle of an enum value, sending `"r\nhr"` instead of `"rhr"`, which fails validation for a reason that has nothing to do with intent. So a `before` validator scrubs whitespace on the enum fields, while leaving the prose fields alone where internal newlines are legitimate markdown:

```python
    @field_validator("tone", "metric", mode="before")
    @classmethod
    def _no_internal_whitespace(cls, v):
        return "".join(v.split()) if isinstance(v, str) else v
```

## Two: measure invention, but never gate on it

A schema proves the shape. It says nothing about whether the numbers in the prose are real. The composer is single-turn and toolless, so every number it may legitimately cite is already present in the context you gave it. That makes a cheap measurement possible: build the set of citable numbers from the context, then for each number in the generated prose, find the nearest known value. A number that sits right next to a known metric but is slightly off is a likely corrupted metric. A number far from any metric is an unrelated quantity, like a prescribed "45 min", and is ignored.

```python
def invention_rate(prose_numbers: list[float], citable: list[float]) -> float:
    if not prose_numbers:
        return 0.0
    flagged = 0
    for n in prose_numbers:
        nearest = min(citable, key=lambda c: abs(c - n)) if citable else None
        if nearest is None:
            continue
        rel = abs(nearest - n) / max(abs(nearest), 1e-9)
        if 0.02 < rel < 0.20:        # near a real value but not equal -> likely corrupted
            flagged += 1
    return flagged / len(prose_numbers)
```

The deliberate choice: this is a measurement, not a gate. It runs once on the finished brief, logs a number, and never rejects a brief or reprompts. A single-turn generator has no corrective round-trip, so blocking would just drop the brief. The signal exists to be watched over time, not to certify any single number, and an occasional false positive is acceptable noise in something advisory.

## Three: shadow-run V2 before you trust it

The schema and the signal tell you about one brief. Before cutting over, you want to know V2 behaves like V1 across many inputs. That is [shadow testing](https://microsoft.github.io/code-with-engineering-playbook/automated-testing/shadow-testing/): run the new version in parallel with the current one, compare outputs, and look for drift before switching. The comparison is not string equality, because the prose differs every run. You compare a fingerprint of structure: how many takeaways, whether it cites steps, which metrics it charts, the tones. Then you check parity against a saved baseline.

```python
def fingerprint(brief: dict) -> dict:
    return {
        "n_takeaways": len(brief["takeaways"]),
        "metrics": sorted(t["metric"] for t in brief["takeaways"] if t.get("metric")),
        "tones": sorted(t["tone"] for t in brief["takeaways"]),
    }

def parity_report(baseline: dict, shadow: dict) -> dict:
    out = {}
    for scenario, base in baseline["scenarios"].items():
        got = shadow.get(scenario, {})
        out[scenario] = (fingerprint_of(base) == fingerprint_of(got))
    return {"per_scenario": out, "overall_parity": all(out.values())}
```

Run V2 in the shadow across a fixed set of scenarios, diff the fingerprints against the baseline V1 produced, and only cut over when parity holds. When it does not, the report tells you exactly which scenario drifted, so you are debugging one case instead of a vague "the briefs feel different."

## The shape of trusting a model

Each check answers a different question. The schema asks "is the shape right", and that is the only one a model provider's structured output mode solves for you. The grounding signal asks "are the numbers invented", and the answer is a logged measurement you watch, not a switch. The shadow run asks "does the new version behave like the old one", answered against a baseline before any user sees it. Stack the three and you can let an agent own the output without pretending the schema alone made it trustworthy.
