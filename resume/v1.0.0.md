## Shipped

v1.0.0 deletes the résumé skill's entire subprocess pipeline. It used to shell out to its own `claude -p` call to tailor bullets, which meant a second API surface, its own timeout handling, and its own cost per run. Now the Claude Code agent you're already talking to does the reading, the job-description extraction, and the bullet rewriting directly, in-conversation, with no subprocess LLM call anywhere in the loop. What's left as actual code is only the part that has to be code: a zod schema plus a deterministic content-truth checker (`scripts/validate.mjs`), and PDF rendering across seven templates (`scripts/render.mjs`). This is a second attempt at exactly this architecture; a first version shipped and got reverted the same day because it was never run end to end before being judged done. This time it shipped with a real evaluation harness behind it, and that harness is the part worth walking through.

## Letting the agent do the work it's already doing

The instinct that this needed fixing came from watching the old pipeline in practice: it would read a résumé, then hand that text to a fresh subprocess call to a model that had to be told from scratch what the résumé said and what the job wanted. The agent doing the orchestrating already had all of that in its own context. Anthropic's own framing of Agent Skills makes the same point directly: skills work by giving an agent instructions and, where necessary, scripts and resources, and the agent applies its own reasoning rather than delegating that reasoning back out to another model call ([Anthropic: Equipping agents for the real world with Agent Skills](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills)). A `SKILL.md` in this world isn't a config file, it's a set of instructions the agent reads and follows in its own context.

```markdown
<!-- SKILL.md, the instruction the agent follows directly -->
## Step 3 — Read, extract, tailor, validate, render

This is the core step. No subprocess, no LLM call besides your own reasoning.

1. Read the résumé with your own Read tool.
2. Read `references/tailoring-rules.md` and rewrite bullets to lead with
   job-relevant framing. Never invent facts.
3. Write the tailored result as JSON matching the ResumeJSON shape.
4. Validate it (a script, deterministic — see below).
5. Render the PDF (a script, deterministic — see below).
```

That's the whole architectural bet: prose that an agent reads and reasons over for anything that requires judgment, and code only for the two things that genuinely aren't judgment calls, rendering a binary file format and checking a claim against a source text.

## Deciding what stays code: skills as the degrees-of-freedom dial

The harder design question wasn't "should tailoring be agent-native," it was "what should still be a script." Anthropic's own best-practices guidance frames this as a dial, not a binary: an instruction that hands an agent full discretion is right for fuzzy judgment calls, and a fixed script is right for the operations that need to run exactly the same way every time regardless of how the agent is reasoning that turn ([Anthropic: Skill authoring best practices](https://www.anthropic.com/engineering/claude-code-best-practices)). Rewriting a bullet to foreground the right experience is a judgment call; there's no single correct rewrite, and the agent's contextual understanding of the résumé and the job is exactly what should decide it. Checking whether that rewrite invented a duration the source résumé never stated is not a judgment call. It has one correct answer, and it's the kind of check that should give the same result every time, which is precisely the case for a deterministic script instead of a second opinion from a model.

```javascript
// scripts/validate.mjs — a deterministic check, not a judgment call
function findsUnsupportedDuration(bullet, sourceText) {
  const durationRe = /\b(\d+)\+?[\s-]*(?:years?|yrs?)\b/gi;
  const claimed = [...bullet.matchAll(durationRe)].map(m => m[1]);
  const sourceDurations = new Set(
    [...sourceText.matchAll(durationRe)].map(m => m[1])
  );
  // A bullet claiming "5 years" is only legitimate if the source résumé
  // actually contains that number next to "years" — not merely the digit 5.
  return claimed.filter(n => !sourceDurations.has(n));
}
```

That split, prose for judgment, code for verification, is what let the tailoring logic itself get simpler even as the guardrails around it got stricter.

## Making the intermediate output a checkpoint, not a black box

The other piece of that same best-practices guidance is to structure a multi-step skill so the agent produces a verifiable intermediate artifact between steps, rather than going straight from input to final output in one pass ([Anthropic: Skill authoring best practices](https://www.anthropic.com/engineering/claude-code-best-practices)). Here that artifact is the résumé JSON itself: the agent writes it to disk before anything gets rendered, `validate.mjs` runs against that file and the original résumé text, and only a clean validation run unlocks rendering.

```javascript
// scripts/validate.mjs — the checkpoint the agent's output has to clear
import { readFileSync } from "node:fs";

function validate(resumeJsonPath, sourceTextPath) {
  const resume = JSON.parse(readFileSync(resumeJsonPath, "utf8"));
  const sourceText = readFileSync(sourceTextPath, "utf8");

  const violations = [];
  for (const bullet of allBullets(resume)) {
    violations.push(...findsUnsupportedDuration(bullet, sourceText));
    violations.push(...findsUnsupportedScope(bullet, sourceText));
    violations.push(...findsBannedPhrases(bullet));
  }
  return violations;   // empty array = clean; the agent proceeds to render
}
```

If validation fails, the agent gets the specific violations back and fixes the JSON directly, since it wrote it and has full context on why each rule matters. That loop is capped at three attempts before it stops and shows the user the remaining violations rather than silently retrying forever, because at that point something structural is wrong that's worth a second pair of eyes, not a fourth automatic attempt.

## Proving it, not assuming it: two gates instead of one

None of the above answers the question that actually sank the first attempt: does this produce good output, reliably, across real inputs? Deterministic checks like the duration check above can only catch violations they were explicitly written to catch. Whether a tailored bullet is well-written, relevant, and well-targeted at the job is an open-ended quality judgment, which is exactly the case an LLM-as-a-judge is suited to: a second model scoring output against a rubric on dimensions a fixed assertion can't express ([Braintrust: What is an LLM-as-a-judge?](https://www.braintrust.dev/blog/llm-judge)). So the harness runs two distinct kinds of gate, not one, and they check for different failure modes.

```javascript
// scripts/evals/run.mjs — deterministic gate: same input, same answer, every time
async function runDeterministicGate(fixtures) {
  const results = [];
  for (const fixture of fixtures) {
    const tailored = await tailorViaSubprocess(fixture);  // fixed prompt for eval reproducibility
    const violations = validate(tailored, fixture.sourceText);
    results.push({ fixture: fixture.key, pass: violations.length === 0, violations });
  }
  return results;
}
```

```javascript
// scripts/evals/judge.mjs — LLM-as-a-judge gate: an open-ended quality call, budget-capped
async function judgeTailoringQuality(fixture, tailored, budget) {
  try {
    if (!budget.canAfford(ESTIMATED_JUDGE_COST)) {
      return { incomplete: true, reason: "budget cap reached" };
    }
    const verdict = await callJudgeModel(fixture, tailored);  // rubric-scored, not pass/fail-only
    budget.record(ESTIMATED_JUDGE_COST);
    return verdict;
  } catch (err) {
    // Any failure here — network, parsing, budget — degrades to "incomplete,"
    // never to a false pass or a crash of the must-pass deterministic gate.
    return { incomplete: true, reason: String(err) };
  }
}
```

The judge pass is capped by an enforced dollar budget, not a quoted estimate, and it's explicitly informational: the harness's single PASS/FAIL verdict is decided by the deterministic gate and a prompt-injection regression check, never by the judge's score. That separation matters because a judge call can fail for reasons that have nothing to do with output quality (a network blip, a parsing hiccup), and a gate that fails the whole build over infrastructure noise gets ignored fast, which defeats the point of having one.

## What actually proved it

`npm test` covers the deterministic layer end to end with no network calls. Proving the agent-native tailoring itself required something that unit tests can't provide: the harness run for real against real fixture pairs, its full results summary shown to a human rather than judged automatically, and one live interactive run producing an actual PDF that got opened and confirmed by hand before any of this was called done. That last step is the one the first attempt at this same architecture skipped, and skipping it is what got it reverted the same day it shipped. The architecture wasn't the problem the first time. Never running it end to end before calling it finished was.
