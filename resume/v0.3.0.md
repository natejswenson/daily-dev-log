## Shipped

v0.3.0 of the resume skill hardens the part of the pipeline most likely to break on any given run: fetching a job posting from a URL. LinkedIn job pages now route through the same Firecrawl stealth path Indeed, Glassdoor, and ZipRecruiter already used, but only behind an explicit `RESUME_ALLOW_LINKEDIN=1` opt-in, since LinkedIn actively pursues scrapers and I'm not going to make that call silently on someone's behalf. When extraction fails anyway, for any reason, the CLI stops handing back a raw internal error string and instead prints "Could not fetch this job posting automatically (`<reason>`). Paste the job description text instead and re-run." The style-picker step also became mandatory instead of conditional, and the output directory moved to a stable `~/resume-out` regardless of where the skill is installed.

The interesting part isn't any single line of that. It's the shape of the whole thing: a preferred path, a fallback path, and a hard floor that never fails silently. That shape has a name, and it's worth building out properly instead of reinventing it badly one `try/except` at a time.

## Building a fallback chain that degrades on purpose

The naive version of "try the good path, fall back if it breaks" is a single `try/except` wrapped around the whole operation. That works until you have more than one way to fail and more than one way to recover, at which point a single catch-all starts hiding exactly the information you need to decide what to do next. The fix is to model each step as a distinct link in a chain, where a failure at one link is a deliberate, typed handoff to the next, not a generic error swallowed into a log line.

```python
# fallback.py
from dataclasses import dataclass
from enum import Enum, auto

class FetchOutcome(Enum):
    OK = auto()
    BLOCKED = auto()       # site actively rejected the request (403, CAPTCHA wall)
    EMPTY = auto()         # request succeeded but returned no usable content
    TIMEOUT = auto()

@dataclass
class FetchResult:
    outcome: FetchOutcome
    text: str | None = None
    reason: str | None = None   # human-readable, safe to show the end user

def fetch_direct(url: str) -> FetchResult:
    """The cheap, no-dependency path: a plain HTTP GET and a naive text extraction.
    Most job postings on plain company career pages succeed here."""
    ...  # real implementation: request the URL, strip markup, return the text

def fetch_stealth(url: str, allow_linkedin: bool) -> FetchResult:
    """The heavier path: a stealth-mode scrape (residential proxy, browser rendering)
    for sites that block plain requests. Gated per-domain — LinkedIn only runs here
    if the caller opted in, because this domain actively pursues scrapers."""
    if "linkedin.com" in url and not allow_linkedin:
        return FetchResult(FetchOutcome.BLOCKED, reason="LinkedIn extraction is opt-in")
    ...  # real implementation: call the stealth-scraping provider
```

Each function returns a typed result instead of raising or returning a bare string, so the caller can branch on *why* it failed, not just *that* it failed. That distinction is what makes the next link in the chain possible.

## Chaining the links: circuit breaker, not infinite retry

The chain itself is a small loop: try the cheap path, and only pay for the expensive path if the cheap one gives you a reason to. This is the same shape Martin Fowler describes in the CircuitBreaker pattern, though the motivation here is cost and directness rather than protecting a failing remote service — the point in both cases is the same: don't keep hammering an operation that's already told you it isn't going to work, and make the failure visible instead of masked ([Martin Fowler: CircuitBreaker](https://martinfowler.com/bliki/CircuitBreaker.html)). The chain has exactly two real attempts, not an open-ended retry loop, because retrying the same blocked path a third time teaches you nothing new.

```python
# fallback.py (continued)

def fetch_job_posting(url: str, allow_linkedin: bool = False) -> FetchResult:
    """Two-link chain: direct fetch, then stealth fetch. No open-ended retry —
    each link either succeeds or hands a typed reason to the next."""
    direct = fetch_direct(url)
    if direct.outcome is FetchOutcome.OK:
        return direct

    stealth = fetch_stealth(url, allow_linkedin)
    if stealth.outcome is FetchOutcome.OK:
        return stealth

    # Both links failed. Prefer the more specific reason for the message.
    reason = stealth.reason or direct.reason or "unknown extraction failure"
    return FetchResult(FetchOutcome.BLOCKED, reason=reason)
```

This is graceful degradation in the sense the Google SRE book uses the term: when a system can't deliver the ideal outcome, it should fall back to a reduced but still useful outcome rather than failing the whole request outright ([Google SRE Book: Handling Overload](https://sre.google/sre-book/handling-overload/)). Here the "reduced but useful" outcome isn't a cached response, it's a clear, actionable instruction to the human on the other end: paste the text yourself.

## The floor: turning a typed failure into a message a person can act on

This is the step that actually shipped in v0.3.0, and it's also the one that's easiest to get wrong, because the instinct is to surface whatever the underlying library said. A raw exception string is written for a developer debugging the library, not for someone trying to get their résumé tailored. Nielsen Norman Group's guidance on error messages is blunt about this: an error message should be written in plain language, state the problem clearly, and, most importantly, tell the person what to do about it ([NN/g: Error Message Guidelines](https://www.nngroup.com/articles/error-message-guidelines/)). "Could not fetch this job posting automatically" is the plain-language problem statement; "Paste the job description text instead and re-run" is the instruction. Both parts have to be there, or the message is just a nicer-looking dead end.

```python
# messages.py
from fallback import FetchOutcome, FetchResult

def to_user_message(result: FetchResult) -> str:
    """Every failure path funnels through here. No caller ever shows a raw
    exception string or internal reason code directly to the end user."""
    if result.outcome is FetchOutcome.OK:
        raise ValueError("to_user_message called on a successful fetch")

    reason = result.reason or "unknown reason"
    return (
        f"Could not fetch this job posting automatically ({reason}). "
        "Paste the job description text instead and re-run."
    )
```

Funneling every failure outcome through one formatting function is what makes this a floor rather than a hope. Any new failure mode added to `fetch_job_posting` later, a new site that blocks scrapers, a new timeout case, still comes out the other end as an actionable sentence, because there's exactly one place that turns a `FetchResult` into text a human reads.

## Wiring it together

The call site reads as three steps because that's genuinely all it is: attempt the chain, check the outcome, and either proceed or hand the user something they can act on immediately.

```python
# cli.py
from fallback import fetch_job_posting, FetchOutcome
from messages import to_user_message

def get_job_description(url: str, allow_linkedin: bool) -> str:
    result = fetch_job_posting(url, allow_linkedin=allow_linkedin)
    if result.outcome is not FetchOutcome.OK:
        raise SystemExit(to_user_message(result))
    return result.text
```

Nothing here retries silently, and nothing here prints a stack trace at a person who just wants to paste in a job listing. The chain either gets the text or tells you exactly what to type next.
