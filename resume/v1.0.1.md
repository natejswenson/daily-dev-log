## Shipped

resume 1.0.1 moves `SKILL.md`, `package.json`, and every script this skill depends on from the plugin's root down to `skills/resume/skills/resume/`, the path Claude Code's plugin loader actually scans for skills. That part shipped clean and passed CI on the pull request. What didn't ship clean, on the very next step, was the automation that tags a release once the fix merged, because the fix I'd just tested was live in a code path the tests hadn't touched.

## Why the tests couldn't have caught this

This repo runs one CI job per skill on every pull request, and a separate release job per skill that only runs on a push to `main` or an explicit manual dispatch. That's a deliberate design, not an oversight: PR-time CI needs to report on every pull request so a required-check gate never deadlocks, but running four release jobs on every commit would be wasted work and wasted risk. The `ci` job is `needs: ci` for the `release` job, so release only ever runs after CI has already passed once.

The problem is that "CI passed" and "release will work" turned out to be two different claims. `_release.yml`, the reusable workflow shared by all four skills, resolves a version by reading `package.json` or `SKILL.md`'s frontmatter directly off disk:

```yaml
defaults:
  run:
    working-directory: skills/${{ inputs.skill }}
steps:
  - name: Resolve version
    run: |
      read_pkg() { node -p "require('./package.json').version"; }
      # ... reads ./SKILL.md if package.json isn't present
```

That `working-directory` pointed at the plugin root, which was correct the day I wrote it and wrong the moment `SKILL.md`/`package.json` moved one directory deeper. Nothing in the pull request's CI run ever executed this step, because `release` only fires on a push to `main`, and by the time it fired for real, the file it expected wasn't there anymore. I merged the packaging fix, dispatched all four release workflows to cut the actual version tags, and watched all four fail with `cannot open file 'SKILL.md'`.

## This is the same convention, one layer up

The interesting part isn't that I made a path mistake. It's that this was the exact same class of bug as the plugin-discovery fix that caused it, just in a different system. GitHub Actions' own model for this is explicit: a job or step's `working-directory` default sets where every subsequent `run:` command resolves relative paths from, and nothing checks that the files you expect are actually there until the shell tries to open one and fails ([GitHub Actions docs: Workflow syntax for GitHub Actions](https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions)). Convention over configuration, the same principle Claude Code's plugin loader relies on, is also what a CI job's directory defaults rely on: nobody wrote "package.json lives here" as an assertion, the workflow just assumed it, the same way the plugin loader assumed `SKILL.md` sat at a fixed depth. David Heinemeier Hansson's original framing of the trade is that you "only need to specify unconventional aspects" ([Wikipedia: Convention over configuration](https://en.wikipedia.org/wiki/Convention_over_configuration)), and both of my bugs were the same failure mode: a convention moved, and nothing was set up to notice.

## Build: fix the two steps that actually needed it, not the whole file

`CHANGELOG.md` correctly stayed at the plugin root during the packaging move, so the release notes and tagging steps that read it were already fine. Only the version-resolution step and the npm-publish step needed the deeper path, so I overrode `working-directory` on just those two, leaving the job-level default (and everything reading `CHANGELOG.md`) untouched:

```yaml
- name: Resolve version
  id: ver
  # SKILL.md/package.json live one level deeper than the plugin root.
  # CHANGELOG.md, read by later steps, correctly stays at the plugin root.
  working-directory: skills/${{ inputs.skill }}/skills/${{ inputs.skill }}
  run: |
    set -euo pipefail
    read_pkg() { node -p "require('./package.json').version"; }
    # ...
```

## Use it: prove the resolution logic before trusting a workflow run

GitHub Actions doesn't give you a fast local loop for this kind of bug, so I ran the exact shell logic by hand against the real files first, rather than pushing and waiting on a runner:

```bash
cd skills/resume/skills/resume
node -p "require('./package.json').version"
# -> 1.0.1
```

That's not a full simulation of the workflow, but it's the one line that actually mattered, and it's cheap enough to run for all four skills before touching CI again.

## Verify: watch the thing that failed, actually succeed

The only real verification for a release pipeline is a real release. I re-dispatched all four workflows after the fix merged and watched them resolve, tag, and publish:

```bash
gh workflow run resume.yml --ref main
gh run list --workflow=resume.yml --branch=main --limit=1 \
  --json conclusion --jq '.[0].conclusion'
# -> success
git tag -l 'resume-v*'
# -> resume-v1.0.1
```

The tag existing is the actual proof, not the green checkmark on the workflow run. A CI job can pass because it never touched the failing code, and a release job can look identical to a working one right up until the step that tries to open a file that isn't there.
