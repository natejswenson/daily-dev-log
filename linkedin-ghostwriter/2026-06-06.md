## What I Built

Today started with a LinkedIn post arguing that a Claude skill is code and should be validated like code: score the SKILL.md, unit-test the scripts, version the releases. Then I made the ghostwriter skill actually live up to that, instead of just preaching it.

First, the eval. A SKILL.md is a prompt, so you can't compile it — you score it. I wrote a scorer that runs grounded pass/fail checks against the skill's instructions: does it have frontmatter with a name, description, and version; does it declare its three operating modes; does it state the never-publish-without-approval guardrail; does it reference the no-automated-posting compliance rule and the voice inputs. It prints a score and exits non-zero on any failure so CI can gate on it. It currently passes 8 of 8.

Second, the tests. The skill had four runtime scripts and one bash script with zero coverage. I added a pytest suite covering all five Python scripts at 100% line coverage — 94 tests in total. The tricky parts were mocking the LinkedIn network calls, faking Playwright for the image renderer, and exercising the OAuth callback server without opening a real browser or binding a real socket on every path. The bash release-radar script can't be measured by Python coverage, so it gets a shellcheck lint instead, which I kept honest about rather than pretending it was "covered."

Third, I wired it all into a CI workflow so none of this can silently regress: every push and PR to main now runs shellcheck, the test suite with a hard 100% coverage gate, and the SKILL.md scorer. The first CI run actually caught a real bug — a test that assumed the gitignored personal brand-guide CSS existed, which it doesn't in a clean checkout. Good reminder that the value of the gate is catching the environment assumptions you didn't know you were making.

Finally, I cut v0.0.1: a version field in the SKILL.md frontmatter, a changelog entry, an annotated git tag, and a GitHub release.

## What's Next

With the eval, coverage gate, and release process in place, the natural next step is extending the SKILL.md scorer from structural checks toward behavioral ones — small fixtures that assert the generated drafts actually honor the voice rules — and adopting the same version-and-changelog discipline for future feature work.
