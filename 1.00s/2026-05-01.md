## What I Built

Apple Review came back with a Guideline 2.1 rejection on iPadOS 26.4.2: the ATT prompt couldn't be found. Root cause was overcautious wiring on my part — I'd read too much weight into the historical "don't cold-launch ATT" lore and tied the prompt to fire only after the user completed a full run (handleRunEnd + 2s defer). The reviewer never reached that path. Apple's current guidance is more relaxed: as long as `NSUserTrackingUsageDescription` reads cleanly (mine does — "This helps us show you more relevant ads and keep the app free."), the prompt firing on first launch is fine and in fact expected for apps that track.

The fix moved the call into `RootView.task` with a 1.5s defer past launch. That gives the BrandFlash (1.0s play + 0.3s fade) uncontested attention, then the ATT prompt fires before any ad request reaches Google. UMP/CMP for EEA consent runs immediately after, so the two system prompts don't visually compete. Auth restore continues to run in parallel from the start of the task. Removed the now-redundant post-run trigger — iOS gates the system prompt to once-per-device anyway, but the dead code wasn't worth keeping.

Verified on a clean iPad Air 11" (M3) install with privacy reset: ATT prompt appears with the app icon, the right copy, and both action buttons within ~3 seconds of launch. Same on iPhone 16 Pro Max — banner test ad visible at the bottom, prompt centered. Bumped CFBundleVersion to 5, archived, uploaded.

The release tag bookkeeping needed cleanup too: `v2.0` had been pointing at the merge commit (build 4, rejected). After resubmit I deleted both local and remote tags and re-cut `v2.0` at the actual ATT-fix commit (build 5, submitted). The tag now matches what App Review is actually evaluating.
